#### ⚠️ 问题 1.1：设计的 opts 开关数量超过实际执行需要

**位置**：[任务卡片统一方案.md:44-57](任务卡片统一方案.md#L44-L57)

方案第二节列出了 8 个功能开关参数：

```
opts.showClaim       → 显示[领取]按钮
opts.showSubmit      → 显示[提交]按钮
opts.showDrop        → 显示[放弃]按钮
opts.showExpand      → 显示展开/折叠
opts.showStatusEdit  → 显示状态下拉（管理员用）
opts.showAssignee    → 显示负责人名字
opts.showReview      → 显示审核按钮
```

但第五节 Step 1 明确说**只加 2 个**：

> 在已有 renderQuestCard 函数中加 opts 的开关参数：showStatusEdit、showAssignee

**分析**：前 4 个开关（`showClaim`/`showSubmit`/`showDrop`/`showExpand`）在当前代码中已经通过**状态推导**而非显式参数实现了——按钮的显示/隐藏是根据 `locked`/`isDone`/`myClaim`/`grey`/`myClaim.status` 等状态自动判断的，见 [dashboard.html:2790-2805](dashboard.html#L2790-L2805)。把它们声明为 opt 参数但不实现，会给阅读者"需要实现 8 个参数"的错觉，造成设计文档与实际代码不一致。

**建议**：删除当前不计划实现的 opt 声明，只保留 `showStatusEdit` 和 `showAssignee`。等未来确实需要时再扩充——这本身就是"少即是多"的实践。

**严重程度**：低（不影响执行，但影响文档可读性和维护者的理解）

#### ⚠️ 问题 1.2：砚仁工作台改用完整卡片可能降低信息密度

**位置**：[任务卡片统一方案.md:145-148](任务卡片统一方案.md#L145-L148) 与 [dashboard.html:2667-2686](dashboard.html#L2667-L2686)

当前工作台的任务渲染是**紧凑行式布局**，每个任务一行：

```
🏕️ 任务名 · NT 100 · 07-15 · 🔄进行中 · [📤 提交]
```

信息密度高，适合"我的任务仪表盘"这种个人效率场景。

改用 `renderQuestCard` 后，每个任务将渲染为：
- 类型徽章 + 任务名 + scope 标签 + 共创警告 + 展开按钮（header 行）
- 发布者 + 截止日期 + 名额 + NT（meta 行）
- 提交/放弃按钮（actions 行）
- 展开后还有详情区

预估每个任务占用的垂直空间是当前的 **2-3 倍**。对于需要一览全部任务的个人工作台来说，这是用更重的 UI 替换了更高效的 UI。

**建议**：在 Step 2 中增加一个设计决策——是接受信息密度降低，还是为工作台增加 `opts.compact: true` 模式来保留紧凑布局。紧凑模式下可以只渲染一行 header + 操作按钮，不渲染 meta 行和展开区。

**严重程度**：中（影响用户体验，但非功能 bug）

#### ❌ 问题 4.1：展开/折叠的回调函数硬编码了三个 context，扩展到五个会断裂

**位置**：[dashboard.html:2755](dashboard.html#L2755)

当前 `renderQuestCard` 中展开/折叠的 onclick：

```javascript
onclick="...'+(ctx==='village'?'renderVillageQuestHall()':ctx==='questboard'?'renderQuestBoard()':'renderWorkspace()')+'"
```

这是一个三元表达式链，只覆盖了三种情况。当 context 为 `'member'` 或 `'taskhall'` 时：
- 不等于 `'village'` → 进入第二个分支
- 不等于 `'questboard'` → 落入 else，调用 `renderWorkspace()`

**后果**：在成员档案或工会大厅中点击卡片展开/折叠，会触发 `renderWorkspace()` 重渲染——这是完全错误的渲染路径。不仅 UI 不更新，还可能把工作台的内容渲染到错误的 DOM 容器中。

**方案缺失**：Step 1 说"扩展 renderQuestCard"，但没有提到需要修改展开回调逻辑以支持新 context。这是一个**执行层面的遗漏**——如果按方案的 Step 1 实现，Step 3 和 Step 4 执行后展开功能直接断裂。

**修复方案**（二选一）：
- **方案 A**：在 `opts` 中增加 `opts.onToggle` 回调函数，由调用方传入重渲染函数
- **方案 B**：在 `renderQuestCard` 外部建立 context→渲染函数的映射表，展开时查表调用

**建议**：推荐方案 B（映射表），因为它不需要每个调用方都写重复的回调。在 Step 1 中增加这个修改项。

**严重程度**：高（会导致运行时 bug——两个新场景的卡片展开/折叠完全失效）

#### ❌ 问题 5.1：成员档案的任务对象可能与 data.tasks 不是同一引用，indexOf 会静默失败

**位置**：[dashboard.html:2733](dashboard.html#L2733)

`renderQuestCard` 内部通过 `data.tasks.indexOf(t)` 获取任务索引：

```javascript
var tasks = data.tasks || []; var idx = tasks.indexOf(t);
```

这个 `idx` 被用于所有操作按钮：`showSubmitModal(idx)`、`dropTask(idx)`、`claimTask(idx)`。

**风险分析**：成员档案传入的任务来自 `m.tasks`（成员任务聚合数组）。如果 `m.tasks` 中的任务对象不是 `data.tasks` 中的同一个引用（例如经过 `Object.assign`、展开运算符 `{...t}` 或任何形式的浅拷贝/深拷贝），`indexOf` 将返回 `-1`。此时：
- `showSubmitModal(-1)` → 静默失败或访问不存在的任务
- `dropTask(-1)` → 同上
- `claimTask(-1)` → 同上

查看了代码中 `m.tasks` 的构建方式（在 `renderMemberDetailFor` 调用方 `renderMemberDetail` 中），需要确认任务对象是否保持引用。但即使当前实现恰好保持了引用，这也是一个**脆弱的依赖**——任何未来的重构（比如对任务数组做 map/filter 变换）都可能破坏它。

**建议**：在方案 Step 1 中增加一项保护措施——在 `opts` 中增加可选的 `opts.taskIndex` 参数。如果传入，优先使用；如果不传入，fallback 到 `indexOf`。

```javascript
var idx = (opts.taskIndex !== undefined) ? opts.taskIndex : tasks.indexOf(t);
```

同时建议 Step 3（成员档案改造）中显式传入 `taskIndex`，从源头切断隐患。

**严重程度**：高（当前可能不触发，但属于定时炸弹式的技术债）

#### ⚠️ 问题 5.2：opts.context 字符串枚举模式随场景增多会退化为"上帝函数"

**位置**：[任务卡片统一方案.md:44](任务卡片统一方案.md#L44) 整体设计

当前 `renderQuestCard` 内部已经有多处基于 `ctx` 的条件分支：
- 展开回调的选择（[2755](dashboard.html#L2755)）
- `locked` 的判断（`ctx==='village'`，[2741](dashboard.html#L2741)）
- "你已领取"标签的显示（`ctx!=='workspace'`，[2771](dashboard.html#L2771)）

方案计划将 context 从 3 个扩展到 5 个。如果继续在 `renderQuestCard` 内部用 `if (ctx === 'xxx')` 的方式处理差异化逻辑，这个函数会随场景数量**线性膨胀**。每次新增一个场景，就要在函数内部多处插入新的条件分支。

**更可持续的模式**：让 `opts` 支持**行为注入**而非**状态标记**：

```javascript
// 当前模式（容易膨胀）
if (ctx === 'village') { /* special */ }
if (opts.showStatusEdit) { /* render dropdown */ }

// 更可持续的模式（开放扩展点）
var actionsHTML = opts.renderActions ? opts.renderActions(t, idx, myClaim) : defaultActions(t, idx, myClaim);
```

**判断**：当前 5 个 context 尚在可控范围，问题 5.2 是**预警级别**而非阻塞级别。但如果未来增加到 7-8 个场景，问题会变得严重。建议在方案末尾加一句"未来扩展方向"备注。

**严重程度**：低（当前不影响，但值得记录为技术债意识）
