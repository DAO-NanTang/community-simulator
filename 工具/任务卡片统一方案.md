# 任务卡片统一方案

> 2026-07-08 | 目标：五个看任务的地方，同一套卡片渲染，不同数据来源

---

## 一、当前问题

系统里有五个地方能看到任务卡片，但各自渲染逻辑独立：

| 位置 | 当前渲染 | 问题 |
|------|---------|------|
| 村口任务大厅 | `renderVillageQuestHall` 自己画 | 已改用共用函数 ✅ |
| 主界面任务大厅 | `renderQuestBoard` | 已改用共用函数 ✅ |
| 砚仁工作台 | `renderWorkspace` 自己画 | ❌ 未统一 |
| 成员档案 | `renderMemberDetailFor` 表格行 | ❌ 无详情展开 |
| 工会大厅 | `renderDailyView` 自己画 | ❌ 按日期，但卡片不统一 |

同一个"工笔画临摹"任务，在五个地方看起来不一样。改一处样式要改五个函数。

---

## 二、解决方案

**抽一个共用函数 `renderQuestCard(t, opts)`，五个地方都调用它。**

```
                data.tasks（唯一数据源）
                      │
        ┌─────────────┼─────────────┬──────────────┬──────────────┐
        │             │             │              │              │
    村口过滤      主界面过滤     砚仁过滤       成员过滤       工会过滤
   active任务    active任务   claimant=我    assignee=选中   全部+日期
        │             │             │              │              │
        └─────────────┴─────────────┴──────────────┴──────────────┘
                      │
              renderQuestCard(t, opts)
                      │
              同一套卡片：展开/折叠、类型徽章、scope标签、操作按钮
```

`opts` 参数控制不同场景的差异：

```
opts.context     → 'village' | 'questboard' | 'workspace' | 'member' | 'taskhall'
opts.userName    → 当前查看者名字
opts.isMember    → 是否是营队成员

当前实际使用的开关（随需求逐步增加，不预声明未实现参数）：
opts.showStatusEdit → 显示状态下拉（管理员用）— Step 3 需要
opts.showAssignee  → 显示负责人名字 — Step 3-4 需要
opts.taskIndex     → 任务在 data.tasks 中的索引（见问题 5.1，避免 indexOf 引用依赖）
opts.memberName    → 选中的成员名（用于 member 场景的回调函数名）
```

**展开/折叠回调**（修复问题 4.1）：当前代码用三元表达式硬编码了 3 个 context，扩展到 5 个会断裂。改为映射表：

```js
var _cardRefreshFn = {
  village: 'renderVillageQuestHall()',
  questboard: 'renderQuestBoard()',
  workspace: 'renderWorkspace()',
  member: 'renderMemberDetail(' + escAttr(opts.memberName) + ')',
  taskhall: 'renderDailyView("staff")'
};
// onclick 中使用 _cardRefreshFn[ctx] 替代硬编码三元表达式
```

**索引保护**（修复问题 5.1）：`indexOf(t)` 依赖对象引用一致性。成员档案传入的任务可能是拷贝对象。改为：

```js
var idx = (opts.taskIndex !== undefined) ? opts.taskIndex : tasks.indexOf(t);
```

## 三、五个场景的数据来源和功能差异

### ① 村口任务大厅（village）

- **数据**：`data.tasks` 中 `status==='active'` 的任务，按 scope/类型/快捷筛选过滤
- **分组**：可领取 / 我已领取 / 已完成（折叠）
- **特殊**：非营队成员看营地任务 → 🔒灰掉，不可领取
- **按钮**：领取、提交、放弃（仅自己的任务）

### ② 主界面任务大厅（questboard）

- **数据**：同上，但不灰掉营地任务
- **分组**：可领取 / 已领取 / 已归档（折叠）
- **按钮**：领取、提交、放弃

### ③ 砚仁工作台（workspace）

- **数据**：`data.tasks` 中 `claimant===当前用户` 或 `assignee===当前用户` 的任务
- **特殊**：共建者需要切换标签来区分两种来源——

```
👤 砚仁
─────────────────────────
[🏛️ 创世工作] [📋 大厅委托] [✏️ 私人委托]
```

| 标签 | 数据过滤 | 用途 |
|------|---------|------|
| 🏛️ 创世工作 | `assignee===我`（创世终端分配的） | "这是我的工作" |
| 📋 大厅委托 | `claimant===我`（从大厅主动接的） | "这是我接的私活" |
| ✏️ 私人委托 | `scope==='personal'`（个人发布的） | "私人间的委托" |

冒险者没有"创世工作"标签——他们没有创世终端分配的任务。

- **附加**：左边 NT 余额卡 + 流水 + 设置面板

### ④ 成员档案（member）

- **数据**：`assignee===选中的人` 或 `claimant===选中的人`
- **不同角色的数据来源**：

| 选中的人 | 数据来源 | 显示 |
|---------|---------|------|
| 共建者 | 创世终端分配的任务（assignee） + 大厅接的（claimant） | 全部显示 |
| 冒险者 | 大厅领取的任务（claimant） | 全部显示 |
| 云村民 | 私人委托（claimant） | 如有则显示 |

- **特殊**：管理员可编辑状态——卡片上有状态下拉框（待开始/进行中/已完成）
- **附加**：材料表视图

### ⑤ 工会大厅（taskhall）

- **数据**：**全部任务**（不只是 claimant 过滤），按日期组织
- **特殊**：和数据来源①-④完全不同——工会大厅看到的是"创世终端打包后分配给每个人的全部工作"，不管是否有人领取
- **显示**：选中某个人 → 显示他当天要做的创世工作 + 从大厅接的任务
- **按钮**：无领取/提交（这是管理视图，不是操作视图）
- **日期筛选**：按日期分栏，里程碑分类

## 四、工会大厅 vs 任务大厅的本质区别

```
任务大厅（给参与者看）              工会大厅（给管理者看）
─────────────────────              ─────────────────────
"有什么可以接？"                    "大家做得怎么样了？"
"我接了什么？做完了吗？"             "谁掉队了？什么遗漏了？"
数据：有 claimants 的任务            数据：全部任务（创世分配 + 大厅发布）
分组：按领取状态                    分组：按日期 + 按人
操作：领取、提交、放弃              操作：查看进度
```

## 五、执行步骤

### Step 1：扩展 renderQuestCard

在已有 `renderQuestCard` 函数中加 `opts` 的开关参数：

- `showStatusEdit`：为 true 时，卡片下方显示状态下拉框（待开始/进行中/已完成）
- `showAssignee`：为 true 时，meta 行显示负责人名字

### Step 2：改造砚仁工作台

- 任务卡片区改用 `renderQuestCard(t, {context:'workspace', userName:currentUser})`
- **过滤体系设计**（修复问题 2.1）：三层标签 + 状态筛选是上下层关系，不是互相替代：

```
第一层（任务来源）：[🏛️ 创世工作] [📋 大厅委托] [✏️ 私人委托]
  └─ 共建者看三个标签，冒险者只看后两个（无"创世工作"）
  
第二层（scope + 状态）：[全部] [🏕️ 营地] [✏️ 私人] [🔄进行中] [⏳待审核] [✅已完成]
  └─ 在第一层过滤结果上进一步筛选
```

两层同时生效：先选来源标签，再选状态标签，两层交集。

- **信息密度说明**（修复问题 1.2）：工作台是个人效率场景。`renderQuestCard` 标准模式的卡片高度较高（类型徽章 + meta行 + 操作行 + 展开区），可能不适合一次性浏览大量任务。初期使用标准卡片，如果用户反馈信息太散，后续加 `opts.compact: true` 模式——只渲染一行 header + 操作按钮，不渲染 meta 行和展开区。这是一个有意的设计决策，不是执行遗漏。

- 左边 NT 余额卡保持不变

### Step 3：改造成员档案

- 任务区改用 `renderQuestCard(t, {context:'member', showStatusEdit:true, showAssignee:true, taskIndex:idx})`
- 管理员可改状态（通过卡片下方的状态下拉框：待开始/进行中/已完成）
- **材料编辑不嵌入卡片**（修复问题 3.2）：任务卡片视图只显示任务名 + 状态下拉 + 负责人名。材料编辑仅在"材料表视图"中进行，两个视图通过现有切换按钮分开。不在卡片展开区嵌入材料表单——保持卡片简洁，材料编辑集中在材料表视图
- 保留材料表切换按钮（📋 任务 / 📦 材料表）

### Step 4：工会大厅保持结构

- 工会大厅的日期视图结构不变，只改卡片渲染为 `renderQuestCard(t, {context:'taskhall', showAssignee:true})`
- 选中不同的人 → 过滤对应的任务
- **工会大厅独有功能处理**（修复问题 3.1）：

| 功能 | 处理方式 |
|------|---------|
| 协作者头像 | 保留在 `renderQuestCard` 外部渲染——作为工会大厅日期视图的装饰层，不迁移入卡片 |
| 物品/材料展示 | 同上，保留在卡片外部渲染——从任务备注解析物品的逻辑保持在 `renderDailyView` 中 |
| 点击切换完成状态 | 通过 `opts.onToggleStatus` 回调实现——默认为 `dailyCheckTask(idx, role)`，点击卡片时触发 |

## 六、完成后的效果

```
同一个 "工笔画临摹" 任务：

村口看 → 卡片 + [领取]
主界面看 → 卡片 + [领取]/[提交]
砚仁看 → 卡片 + [提交] + NT余额
档案看 → 卡片 + [状态下拉]
工会看 → 卡片 + 负责人 + 日期

同一套样式，不同的操作按钮，不同的数据来源。
```

## 七、未来扩展备忘（修复问题 5.2）

当前 `renderQuestCard` 内部有少量基于 `ctx` 的条件分支（locked判断、展开回调选择、领取标签显示）。目前 5 个 context 在可控范围。如果未来增加到 7-8 个场景，条件分支会线性膨胀，建议将差异化逻辑重构为行为注入模式：

```js
opts.renderActions = function(t, idx, myClaim) { return '...'; }  // 自定义操作区渲染
opts.onToggle = function() { ... }  // 自定义展开/折叠回调
```

当前不改——5 个场景用映射表 + 状态推导足够应对。仅在超过 6 个 context 时启动重构。
