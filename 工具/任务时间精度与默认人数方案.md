# 任务时间精度 + 默认人数 + 发布确认 + 工作台增强

> 2026-07-10 | 待讨论，不执行

---

## 〇、你要解决的问题（4 个）

1. **截止时间只有天**——不能选「7/10 下午 2:30」。午饭前交、下午讨论前交这种场景需要小时分钟。
2. **人数默认 0**——容易忘记设。私人任务默认 1 人，营队任务默认 10 人。
3. **发布没有确认**——点了发布直接入库，没有预览校核。
4. **工作台看不到自己发的任务**——发出去之后找不到在哪看状态、没法撤回。

---

## 一、时间精度：从「天」到「分钟」

### 1.1 存储格式升级

| | 现在 | 改成 |
|------|------|------|
| 存储格式 | `"7/10"` | `"7/10 14:30"` |
| 没有时间时 | `"7/10"`（00:00:00） | `"7/10 23:59"` |
| 显示格式 |「7/10」 |「7/10 14:30」（有时）/「7/10」（仅日期） |

### 1.2 表单控件

原有 `<input type="date">` → 改为 `<input type="datetime-local">`，浏览器原生控件，支持日期 + 时分。

> **注意**：`datetime-local` 的值是 ISO 格式（`2026-07-10T14:30`），提交时需转换为存储格式 `"7/10 14:30"`（M/D HH:MM）。现有代码中已有 `datetime-local` 使用先例（`ntInputVal` 函数，line 5986），可参考其转换方式。

## 二、`parseMD` 升级：支持时间部分

### 2.1 解析流程

在 `parseMD` 函数内部，**现有正则之前**加一段时间匹配：

```
输入 "7/10 14:30" → new Date(当前年, 6, 10, 14, 30)
输入 "7/10"       → new Date(当前年, 6, 10)          （兼容旧格式，不变）
输入 ""（空）     → null（空 deadline，相当于不限截止）
```

### 2.2 兜底规则

用户只选日期不选时间 → 表单提交时自动拼接 `" 23:59"` → 存储为 `"7/10 23:59"` → 解析为当天最后一刻。

### 2.3 对现有函数的影响

| 函数 | 影响 | 说明 |
|------|:--:|------|
| `parseMD(s)` | 升级 | 加时间正则 |
| `daysBetween()` | **不变** | 仍按日历日 floor，时分不影响 |
| `daysUntil()` | **不变** | 同上 |
| `parseDate()` / `parseDateStr()` | **不变** | 薄封装自动继承 |
| `sortByDeadline()` | 简化 | 直接用 `parseMD(a) - parseMD(b)`，同天任务早截止排前面 |

## 三、人数默认值

### 3.1 区分两类任务

| 任务类型 | 默认人数 | 原因 |
|------|:--:|------|
| 私人委托（发布任务弹窗） | **1** | 私人间的委托，通常 1 对 1 |
| 营队任务（NT 管理发布） | **10** | 营队任务面向全体，需要多人接收 |

不设「不限人数」勾选框——直接填数字，想不限就填 0。保持简洁。

### 3.2 存量兼容

现有 `max_slots = 0` 语义是「不限」，不变。新发布的任务默认有值，用户可手动改为 0。

## 四、发布确认卡片（新功能）

### 4.1 现在 vs 改成

**现在**：填表 → 点「发布」→ `confirmPostTask()` 直接入库 → 弹窗关闭 → 一行小字「已发布：xxx」

**改成**：填表 → 点「发布」→ 弹出一张**确认卡片**→ 用户确认后才入库

### 4.2 卡片内容

```
┌──────────────────────────────────┐
│  📋 确认发布任务                  │
│                                  │
│  名称　　绘制营地地图              │
│  类型　　支线                     │
│  范围　　营地 / 私人               │
│  奖励　　NT 15                    │
│  人数　　不限（或「限 5 人」）      │
│  截止　　7/15 14:30               │
│  审核人　砚仁                      │
│  提交要求 笔记 / 照片≥1            │
│  备注　　请参考卫星图绘制           │
│                                  │
│  [✅ 确认发布]  [✎ 返回修改]       │
└──────────────────────────────────┘
```

### 4.3 实现方式

**适用范围**：私人委托发布（`confirmPostTask`）和营队任务发布（`confirmCampTask`）都使用确认卡片。

`confirmPostTask()` / `confirmCampTask()` 改为两步：

```
Step 1：收集表单数据 → 渲染确认卡片 → 显示在弹窗内
Step 2：用户点「确认发布」→ 写 data.tasks → 扣 NT → saveData → 关闭弹窗
        用户点「返回修改」→ 隐藏卡片 → 回到表单
```

不需要新弹窗——复用现有 `postTaskModal` / `campTaskModal`，在内部切换表单/卡片视图。

## 五、工作台新增「我发布的任务」（新功能）

### 5.1 现在工作台任务面板的结构

```
📋 任务 标签页
  ├── 🏛️ 营队审批（仅管理员可见）
  ├── ⏳ 待审核（有人提交了我发的/我审的任务）
  ├── 🏳️ 撤回审核
  ├── 📋 进行中（我领取的）
  ├── ⏳ 审核中（我提交的）
  └── 📦 已完成（我完成的）
```

**缺少**：「我发布出去的任务」——这些任务的状态（有人领了吗？谁在做了？）看不到。

### 5.2 新增段落

在「待审核」和「进行中」之间插入「📤 我发布的」段落。

**渲染方式**：直接复用 `renderQuestCard`（与任务大厅/进行中卡片相同的渲染函数），传入 `{context:'workspace', userName:currentUser.name, isMember:true}` 即可。`renderQuestCard` 已自带撤回按钮、人数显示、截止时间等信息，无需新建卡片布局。

展示效果示意：

```
📤 我发布的（3项）
  [与任务大厅相同的卡片样式，含撤回按钮、领取人数、截止时间等]
```

### 5.3 筛选逻辑

```js
var myPublished = tasks.filter(function(t) {
  return t.poster === currentUser.name
    && (t.status === 'active' || t.status === 'pending_approval');
  // 含营队任务，卡片上有 scope 标签区分。pending_approval 的也显示（发布者想知道审核进度）。
});
```

> **为什么包含营队任务**：营队审批区只显示 `pending_approval` 状态，一旦审批通过变为 `active`，营队任务就从审批区消失。如果"我发布的"也排除营队任务，发布者将完全看不到自己发布的营队任务的领取进度。

### 5.4 撤回功能

撤回按钮和审批逻辑**已存在**（`renderQuestCard` 中已有撤回按钮，`withdrawTask()` → `reviewWithdrawal()` 状态机完整：`active → withdrawing → withdrawn`，含 NT 退款审计）。

"我发布的"段落的核心价值是**聚合展示**——让发布者在一个地方看到自己发布的所有任务，快速发现哪些需要撤回。卡片复用 `renderQuestCard` 渲染，撤回按钮自然附带，不需要额外开发。

## 九、自查

- **少即是多**：只改 `parseMD` 一处底层。不引入精度切换 checkbox（`datetime-local` 默认 1 分钟步长已足够）。确认卡片复用现有弹窗，不新建 modal。不碰 `days` 函数族和 `Clock` 系统。
- **复用优先**：时间解析走 `parseMD`，天数计算走 `days` 函数族，撤回走已有 `withdrawing` 状态流转（`withdrawTask` + `reviewWithdrawal`）。「我发布的」段落复用 `renderQuestCard` 渲染——撤回按钮、人数显示、截止时间等全部自然附带，无需额外开发。
- **需求对齐**：4 个改动都对应明确的痛点——缺时间精度 → `datetime-local`；默认 0 不安全 → 默认 1/10；缺发布确认 → 确认卡片（私人+营地均覆盖）；缺发布后管理 →「我发布的」聚合视图。没有超出需求范围的"顺手优化"。
- **全局最优**：`parseMD` 改一处全局生效。确认卡片同时覆盖 `confirmPostTask` 和 `confirmCampTask`，保持两个发布入口体验一致。「我发布的」包含营队任务（`active` + `pending_approval`），避免营队审批通过后发布者追踪盲区。工作台新增段落不破坏现有结构。
- **长期主义**：旧新数据格式天然共存（纯日期字符串不受影响）。`renderQuestCard` 作为统一卡片渲染入口，后续卡片改版只改一处。撤回状态机完整（`active→withdrawing→withdrawn`），不会被未来的任务生命周期扩展打破。
