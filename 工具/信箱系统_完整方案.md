# 信箱系统 · 完整方案

> 从「被动查看」到「主动闭环」——统一通知中心，所有角色可用，一键跳转到处理位置
> 2026-07-07

---

## 〇、为什么需要信箱？

### 现在的问题

```
管理员：打开监察 → 看到"待审核 2 项" → 切到 NT管理 → 找是哪 2 项 → 找到 → 审核 → 切回监察 → 还有什么来着？
冒险者：不知道任务审核通过了没 → 切到"我的资料" → 翻找 → 哦通过了 → 不知道什么时候通过的
共建者：不知道有人逾期了 → 要主动去猎人公会翻 → 很容易漏
```

核心缺陷：**知道和行动是断开的**。你必须自己记住"有什么事、去哪个 tab、找哪个条目"。

### 信箱要做什么

```
📬 红点 → 点开 → 看到"陈若雨提交了用户访谈" → 点击 → 直接跳到 NT管理，已定位好 → 审核 → 完成
```

一条线走到底：**通知 → 理解 → 跳转 → 处理 → 闭环**。

## 一、信箱是什么

一个**弹出面板**，不是 mode 切换。放在 HUD 栏，所有角色都能点开。

```
┌─ HUD 栏 ───────────────────────────────────────────────┐
│  砚仁 (🧙管理员)          💎 30,000      📬 ③    ⚙️    │
└─────────────────────────────────────────────────────────┘
                                               ↑
                                          信箱入口
                                    红点数字 = 未读消息数
```

点开后面板从右侧滑出（或居中弹出），不遮挡当前 tab。读完关掉，继续在原 tab 工作。

## 三、交互设计

### 入口

```
HUD 栏右侧：📬 图标按钮
- 无未读：📬 灰色
- 有未读：📬 ③ 红色徽章
- 点击：弹出信箱面板
```

### 面板结构

```
┌─ 📬 信箱 ───────────────────── [✕ 关闭] ────────────┐
│                                                      │
│  [全部] [待处理] [已读]          ← 筛选 tab          │
│                                                      │
│  🔴 待处理                                           │
│  ├─ 🆕 陈若雨提交了「用户访谈」· 2天前  [去审核 →]  │
│  ├─ 🆕 小红「系统架构」逾期 4 天          [去看看 →]  │
│  └─ ⚠️ 小红进度掉队 (1/6)                [去看看 →]  │
│                                                      │
│  📋 已读（折叠）                                      │
│  ├─ 墨染提交了「编写文档」· 1天前                    │
│  └─ fc「代码审查」逾期 1 天                          │
│                                                      │
│  ───────────────────────────────────────              │
│  标记全部已读                                         │
└──────────────────────────────────────────────────────┘
```

### 消息行设计

每条消息：
- **图标**：类型图标（🆕新提交 / ⚠️逾期 / ✅通过 / ❌退回 / 💰到账 / ⏰临近）
- **正文**：谁 + 什么事 + 关键数字
- **时间**：多久前
- **新标记**：未读的左边有圆点 ●
- **操作**：右侧有跳转按钮 `[去审核 →]` / `[去看看 →]`

点击消息行本身 = 跳转。点击跳转按钮 = 同样跳转。

### 跳转行为

```
点击消息
  → 关闭信箱面板
  → switchTab(目标tab)
  → 定位 + 高亮目标条目（闪烁 3 次后恢复）
  → 自动展开折叠区（如果目标在折叠区内）
  → 标记该消息为已读
```

## 四、数据设计

### 消息数据结构

```js
// inbox_state 存 localStorage
{
  "messages": {
    "admin": [
      {
        id: "sub_陈若雨_用户访谈",
        type: "submitted",        // submitted | overdue | progress_low | pool_low | ...
        read: false,
        createdAt: "2026-07-05",
        data: {
          person: "陈若雨",
          task: "用户访谈",
          waitDays: 2,
          jumpTab: "ntadmin",
          jumpTarget: { person: "陈若雨", task: "用户访谈" }
        }
      },
      ...
    ],
    "adventurer_陈若雨": [...],
    ...
  },
  "lastOpened": "2026-07-07T15:30:00"
}
```

### 消息生成时机

消息**不是每次打开面板时实时计算的**（和之前监察不同）。消息在以下时机生成/刷新：

| 时机 | 触发 |
|------|------|
| 加载数据 | `loadData()` 后刷新所有消息 |
| 操作完成 | 审核通过、提交任务、分配任务后刷新相关消息 |
| 打开信箱 | 刷新时间戳、清理过期消息 |

### 消息去重和清理

- 同一件事只生成一条消息（如"陈若雨提交了用户访谈"只出现一次）
- 已读消息保留 7 天后清理
- 已处理的消息（如审核完毕）自动消失

## 六、分阶段执行

### 🟢 第一阶段：管理员信箱（核心闭环）

**目标**：管理员能用信箱完成"看到 → 跳转 → 处理"闭环。

| 任务 | 产出 |
|------|------|
| HUD 栏添加 📬 按钮 + 徽章 | 入口可见 |
| 弹出面板 UI | 面板可用 |
| 管理员消息生成（待审核、逾期、进度、NT池） | 消息可见 |
| 点击跳转 + NT管理/猎人公会/成员档案支持定位 | 闭环可用 |
| 已读/未读持久化 | 红点准确 |
| 移除旧的监察 mode 和按钮 | 清理干净 |

**验收**：管理员登录 → 看到 📬③ → 点开 → 看到待审核消息 → 点一条 → 跳到 NT管理并定位 → 审核 → 完。

### 🟡 第二阶段：冒险者信箱

**目标**：冒险者也能用信箱闭环。

| 任务 | 产出 |
|------|------|
| 冒险者消息生成（新任务、审核结果、截止临近、NT到账） | 消息可见 |
| 跳转到任务公告栏 + 我的资料（支持定位） | 闭环可用 |
| 按角色过滤消息（每个角色只看自己的） | 隔离正确 |

### 🔵 第三阶段：共建者 + NPC + 游客

**目标**：全覆盖。

| 任务 | 产出 |
|------|------|
| 共建者消息生成 + 跳转 | 共建者可用 |
| NPC/游客欢迎引导消息 | 全覆盖 |
| 消息自动清理（7天过期） | 不堆积 |

## 七、和现有监察的关系

| 现有监察 | 信箱系统 |
|---------|---------|
| 模式切换（独占一个 mode） | 弹出面板（不占 mode） |

## 八、为什么这样设计

### 不是"加一个功能"，而是"修一个断点"

现有的断点：**信息和行动是分离的**。你知道有事，但你的手和眼要自己去找到事在哪、怎么处理。

信箱修的就是这个断点：**把信息变成行动的入口**。

### 不是"管理员专用"，而是"每个角色都需要"

冒险者等审核结果、共建者看谁提交了、NPC 看有没有新任务——每个人都有自己的"需要知道"。统一入口比各自找 tab 翻看效率高得多。

### 不是"多一个东西要维护"，而是"少一个东西要记住"

没有信箱 → 管理员要记住：打开监察看数字 → 切 NT管理 → 找对应条目 → 审核 → 记住还有没有别的。有心智负担。

有信箱 → 打开 → 看到消息 → 点 → 到了。一条龙。不费脑子。

## 九、风险和对策

| 风险 | 对策 |
|------|------|
| 跳转定位不准确 | 定位用"人名+任务名"双重匹配，找不到时只切 tab，不崩溃 |
| 消息膨胀 | 已处理的消息自动消失，已读消息 7 天过期 |
| 面板频繁开关影响体验 | 面板是浮层，不关也能操作背后的 tab |
| 原有监察用户习惯改变 | 信箱面板内容比监察更直观，过渡自然 |
