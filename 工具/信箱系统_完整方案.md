# 信箱系统 · 完整方案（v2）

> 审查后修订版。核心改动：消息实时计算不预存、信箱替代监察、跳转统一传参、消息聚合。
> 2026-07-07

## 〇、为什么需要信箱？

### 现在的问题

管理员：打开监察 → 看到"待审核 2 项" → 切到 NT管理 → 找是哪些 → 审核 → 切回来 → 还有什么来着？
冒险者：不知道审核过了没 → 切到"我的资料" → 翻找 → 哦过了 → 错过截止日期的不知道

**知道和行动是断开的。** 必须自己记住"有什么事、去哪个 tab、找哪个条目"。

### 信箱做什么

```
📬 红点 → 点开 → "陈若雨提交了用户访谈" → 点 → 跳到 NT管理，已定位 → 审核 → 完
```

一条线：**通知 → 跳转 → 处理 → 闭环**。

### 和监察的关系

**信箱替代监察。** 监察面板的待审核/逾期提醒 → 收进信箱消息。监察的"全员进度"→ 本就该在成员档案和猎人公会里看，不重复。

旧的 `showOverseePanel()` 和 `builderModeBar` 里的监察按钮在信箱上线后移除。

## 一、消息如何生成——实时计算，不预存

### 为什么

审查报告指出：预生成消息存 localStorage 会带来三个问题：
- 每次操作都要触发消息生成，散布改动
- 消息可能和源数据不同步（任务被删了但消息还在）
- 需要额外的去重、清理、过期逻辑

### 替代方案

**打开信箱时当场计算。** 遍历 `data.tasks` + `data.finance`，判断哪些是"需要关注的"，渲染出来。

### 已读/未读：只存一个时间戳

```js
localStorage.inbox_lastOpened = "2026-07-07T15:30:00"
```

- 消息的 `createdAt`（提交时间、逾期时间等）> `lastOpened` → 🆕 新消息
- 点击跳转时 → 更新 `lastOpened`
- 只存一个字段，不需要消息级别的已读状态

### 数据流

```
用户点 📬
  → renderInbox()
    → 遍历 data.tasks（找待审核、逾期）
    → 遍历 data.finance（找 NT 到账）
    → 当场生成消息列表
    → 对比 localStorage.inbox_lastOpened 判断"新/旧"
    → 渲染面板
```

## 二、角色 × 消息类型

### 🧙 管理员

| 消息类型 | 来源 | 聚合 | 跳转 |
|---------|------|------|------|
| 待审核 | `claimants[].status='submitted'` | **按人聚合**："陈若雨提交了 2 项任务" | → NT管理 `{highlight:'review', person:'陈若雨'}` |
| 逾期 | `deadline < today && 未完成` | **按人聚合**："小红 3 项任务逾期（系统架构4天、彩蛋7天…）" | → 猎人公会 `{highlight:'task', name:'系统架构'}` |
| 进度异常 | 完成率 < 40% 且有任务 | 每人一条："小红仅完成 1/6（17%）" | → 成员档案 `{highlight:'member', name:'小红'}` |
| NT池预警 | 剩余 < 待审核总额 | 一条："NT 池仅剩 2,000，不足以覆盖 4,500 待审核" | → 资源金库 |

**消息上限**：每种类型最多显示 10 条，超过的折叠为"还有 5 条… [展开]"。

### 🧱 共建者

| 消息类型 | 来源 | 跳转 |
|---------|------|------|
| 待审核 | 同管理员 | → NT管理 |
| 逾期（自己所管） | 同管理员，只筛自己相关的 | → 猎人公会 |
| 新任务分配 | 自己被设为 `assignee` | → 猎人公会 `{highlight:'task', name:'海报设计'}` |

### ⚔️ 冒险者

| 消息类型 | 来源 | 跳转 |
|---------|------|------|
| 审核通过 | `claimant.status='completed'` | → 我的资料 |
| 审核退回 | `claimant.status='rejected'` | → 我的资料 `{highlight:'task', name:'编写文档'}` |
| 截止临近 | deadline 在 2 天内（自己领取的） | → 任务公告栏 `{highlight:'task', name:'前端优化'}` |
| NT 到账 | `finance` 新增 `task_reward` 且 `to` 为自己 | → 我的资料 |

**注意**：不推送"所有可领任务"——那会变成信息轰炸。只通知**和自己相关的**（已领取的截止、审核结果、到账）。

### 👥 NPC / 🏕️ 游客

推迟到 Phase 3。当前他们的界面（村口广场）还没建成交互功能，消息跳转没有目标。

## 三、入口和交互

### 入口

```
┌─ HUD 栏 ─────────────────────────────────────────────┐
│  砚仁 (🧙管理员)       💎 30,000      📬 ③      ⚙️  │
└──────────────────────────────────────────────────────┘
```

- 📬 放在 HUD 栏右侧，所有角色可见
- 红点数字 = 实时计算的未读消息总数
- 点击 → 弹出面板；再点或点 ✕ → 关闭面板

### 面板

```
┌─ 📬 信箱 ──────────────────────────────── [✕] ──────┐
│                                                      │
│  🔴 新消息                                           │
│  ├─ 🆕 陈若雨提交了 2 项任务（2天前）    [去审核 →] │
│  ├─ 🆕 小红 3 项任务逾期                [去看看 →]  │
│  └─ ⚠️ NT 池仅剩 2,000                   [去看看 →] │
│                                                      │
│  📋 更早（折叠）                                     │
│  ├─ 墨染提交了「编写文档」· 1天前                    │
│  └─ fc「代码审查」逾期 1 天                          │
│                                                      │
│  ───────────────────────────────────────              │
│  全部标为已读                                         │
└──────────────────────────────────────────────────────┘
```

- 消息分两组：🆕 新消息（展开）、📋 更早（折叠）
- 新/旧以 `lastOpened` 时间戳为界
- 点击消息行 = 跳转 + 关闭面板
- 点击"全部标为已读"= 更新 `lastOpened` 到当前时间

## 四、跳转机制——统一参数传递

### 不写 5 个 `highlightXxx()`

审查报告建议用统一的 `switchTab(name, params)`。各 tab 的渲染函数收到 `params.highlight` 后自己处理。

### 实现方式

```js
// 信箱中点击消息
function jumpFromInbox(tab, highlight) {
  closeInbox();                       // 关闭面板
  switchTab(tab, { highlight });      // 跳转 + 传参
  updateLastOpened();                 // 标记已读
}

// switchTab 扩展
function switchTab(name, opts) {
  // ... 现有逻辑 ...
  tabHighlight = opts && opts.highlight ? opts.highlight : null;
  // 渲染目标 tab 时，tab 自己读取 tabHighlight 做定位
}
```

### 各 tab 如何处理 highlight

| tab | `highlight` 参数 | 行为 |
|-----|-----------------|------|
| NT管理 | `{type:'review', person:'陈若雨'}` | 展开待审核列表，高亮陈若雨的审核卡 |
| 猎人公会 | `{type:'task', name:'系统架构'}` | 滚动到该任务卡片，边框闪烁 |
| 成员档案 | `{type:'member', name:'小红'}` | 直接打开小红的详情 |
| 任务公告栏 | `{type:'task', name:'前端优化'}` | 滚动到该任务，边框闪烁 |
| 我的资料 | `{type:'task', name:'编写文档'}` | 展开任务列表，高亮对应条目 |
| 资源金库 | 无 | 直接切 tab 即可 |

全局变量 `tabHighlight` 在渲染后自动清空，避免残留。

## 八、为什么这样设计

| 问题 | 答案 |
|------|------|
| 为什么消息不预存？ | 数据源（`data.tasks`）本身就在 localStorage，再存一份它的"投影"只会增加不一致风险和维护成本 |
| 为什么用 `lastOpened` 而不是逐条已读？ | 信箱不是聊天软件，100% 的消息准确率不值得 200% 的复杂度。"上次打开后新增的"这个粒度够用 |
| 为什么取代监察而不是并存？ | 两个东西做的是同一件事（提醒你有事），一个就够了。信箱做得更多（还能跳转） |
| 为什么冒险者不推送"所有可领任务"？ | 12 条可领任务 = 12 条消息 = 信息轰炸。只通知"和己相关的" |
| 为什么跳转统一传参而不是每 tab 写函数？ | 降低耦合。未来加新 tab 只需在渲染时读 `tabHighlight`，不需要信箱知道新 tab 的实现细节 |
