### 五个角色

### 核心原则

1. **「我的资料」→ 个人工作台**：有实际任务/流水操作的角色（管理员、共建者、冒险者）拥有个人工作台，整合营队工作、私人委托、流水、设置。NPC 和游客保持现有村口占位页，此次不改动。
2. **任务公告栏角色内增强**：公告栏现有的领取/提交/审核流程已经跑通，此次改动是在现有基础上增加 scope（营队/私人）和 visibility（谁能领取）过滤，不重建。
3. **管理员可预览任何角色**：扩展现有 `previewMode` 机制（已支持冒险者预览），增加共建者/NPC/游客视角。
4. **双轨制**：营队工作（组织 NT）和私人委托（个人 NT）隔离——不引入静态余额字段，继续通过 finance 流水表 + scope 标记计算。

### 设计决策说明

以下是审查过程中明确的几个关键取舍：

- **侧边栏 vs 卡片布局**：个人工作台采用侧边栏而非继续用卡片堆叠，原因是：(a) 营队/私人/流水/设置是互斥视图（不需要同时看到）；(b) 侧边栏恒定显示 NT 余额汇总，用户切换视图时不用滚回顶部看余额；(c) 侧边栏和全局 tab 栏是不同层级：tab = 模块切换，sidebar = 模块内部导航。如果实现复杂度超出预期，退化为卡片式 + 顶部子 tab 也是可接受的降级方案。
- **NPC/游客不在此次改动范围**：NPC 和游客目前只有村口占位页（升级入口），给他们加上完整的任务公告栏需要定义新的交互模型（他们不能"领取"任务，只能浏览？），需求尚不明确，推迟到后续版本。
- **余额从流水计算，不引入静态字段**：`camp_nt` 和 `personal_nt` 不作为存储字段，而是从 `data.finance` 流水按 scope 过滤后求和得到。避免数据冗余和不一致。
- **单文件架构暂不拆分**：dashboard.html 目前约 6250 行，此次改动预计增加 1000-2000 行。考虑到项目当前只有一个人维护、部署方式是直接打开 HTML 文件，暂不拆分。当文件超过 10000 行或开始多人协作时，优先拆分策略为：CSS → 独立文件，JS → 按模块分文件（auth.js / tasks.js / finance.js / ui.js），通过 `<script>` 标签加载。

### 营队工作 vs 私人委托 —— 双轨制

```
🏕️ 营队工作（scope: camp）
├─ 来源：管理员/共建者在 NT管理 发布
├─ NT 来源：营队 NT 池（组织经费）
├─ 审核：发布者（或指定审核人）审核
├─ NT 到账：营队 NT 余额（camp_nt）
└─ 流水标签：🏕️ 营队

✏️ 私人委托（scope: personal）
├─ 来源：任何人可发布
├─ NT 来源：发布者的个人 NT
├─ 审核：可选
├─ NT 到账：个人 NT 余额（personal_nt）
└─ 流水标签：✏️ 私人
```

### 为什么分开

- 营队 NT 是组织经费，不能和个人消费混
- 流水分开 → 对账清晰
- 收入来源分开 → 知道哪些是工资、哪些是外快
- 税务/统计可以分别计算

## 三、任务公告栏 —— 现有 questboard 增强

### 定位

现有 `renderQuestBoard()`（dashboard.html 第 1944 行）已实现：按类型过滤（主线/支线/共创/隐藏）、按状态过滤（可领取/已领取）、任务卡片展示、领取/提交/放弃操作。

此次改动为**增量增强**，不改动已有的领取/提交/审核流程：

1. 对管理员和共建者也开放此 tab（当前仅冒险者可见），他们可以浏览和领取公开任务
2. 增加 scope 和 visibility 过滤
3. 任务卡片上显示 scope 标记（🏕️营队 / ✏️私人）

### 任务 visibility 字段

visibility 基于**角色**过滤（而非硬编码期数），可扩展：

| visibility 值 | 谁能领 | 说明 |
|--------------|--------|------|
| `all` | 所有人 | 默认值 |
| `adventurer` | 仅冒险者 | — |
| `builder` | 共建者及以上 | 管理员也可领 |
| `npc` | 仅 NPC（在地伙伴） | 预留，此次不用 |
| `visitor` | 仅游客 | 预留，此次不用 |

> **为什么不按期数？** 原设计 `season_4`、`season_5` 等硬编码值每新增一期就需要改代码。如需限制「只允许第 4 期冒险者领取」，可在任务上设置 `min_season: 4` 字段，由领取逻辑校验当前用户的 `season` 属性，而不是把期数写进 visibility。

### 发布时设置

在 NT 管理或猎人公会发布任务时，可设置：
- **scope**：营队 / 私人
- **visibility**：谁能领（下拉单选，默认 `all`）
- **reviewer**：审核人（默认 = 发布者；现有 `reviewTask` 已用 `confirmed_by` 记录审核人，此处用 `reviewer` 替代 `confirmed_by` 作为发布时指定的默认审核人）
- **max_slots**：人数上限（现有字段，保持不变）
- **points**：NT 奖励（现有字段，保持不变）

> **原则**：余额从 `data.finance` 流水实时计算，不做静态存储。`data` 对象和 `users`（localStorage）的现有字段保持不变，仅做增量添加。

### data.tasks 扩展（现有字段全部保留）

```
{
  scope: 'camp',           // 新增：'camp' | 'personal'，默认 'camp'
  visibility: 'all',       // 新增：'all' | 'adventurer' | 'builder' | 'npc' | 'visitor'，默认 'all'
  reviewer: '',            // 新增：审核人名字，空 = 发布者自己审
  min_season: 0,           // 新增：最低期数限制，0 = 不限。领取时校验用户 season >= min_season
  // … 现有字段不变：name, type, category, points, max_slots, deadline, status, claimants, poster, note …
}
```

### data.finance 扩展（现有字段全部保留）

```
{
  scope: 'camp',           // 新增：'camp' | 'personal'，默认 'camp'
  // … 现有字段不变：type, from, to, amount, task_name, date, confirmed_by …
}
```

### NT 余额计算规则（不新增静态字段）

个人工作台显示的 NT 余额通过以下方式**实时计算**，不存储在 `data.members` 中：

- **营队 NT 余额** = `data.finance` 中 `scope === 'camp'` 且 `to === 当前用户` 的 `amount` 之和 − `from === 当前用户` 的 `amount` 之和
- **个人 NT 余额** = 同上，但 `scope === 'personal'`
- **合计 NT** = 两者之和

> 现有 `updateWorldHUD()`（第 1521 行）已按此模式计算总 NT，只需增加 scope 过滤维度即可。

### data.members 扩展（现有字段全部保留）

### 设计要点

1. **为什么余额走流水而不设静态字段**：现有 `data.finance` 已是单一事实来源。如果额外在 `data.members[name].camp_nt` 存一份，每次交易需要同时更新两处（流水 + 余额字段），一处遗漏即产生数据不一致。不如始终从流水计算。
2. **`badges` 勋章暂不加**：勋章系统的设计和展示逻辑（如何获取、何时颁发、UI 如何展示）尚未定义，不应提前在数据结构中预留字段。
3. **`id` 系统唯一 ID 暂不加**：现有系统以 `name` 为唯一标识（`data.members[name]`、`users[name]` 均为以名字为 key 的对象），引入 `id` 意味着所有查找逻辑需要重构。当确实需要支持改名功能时再加。
