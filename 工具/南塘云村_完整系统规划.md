## 〇、一句话说清

**南塘云村 = 实景地图（地基）+ 田间管理 + 住宿管理 + 物品追踪 + 任务系统 + 卫生维护 + 天气提醒 + 社交互动。**

线上线下双向映射。线下的空间、物品、作物、人，在线上有一个实时同步的数字副本。远端的人通过这个副本看见南塘、参与南塘。

## 三、核心数据架构（最关键的部分）

### 3.1 一层数据，多个视图

不是每个子系统各存一套数据。而是**共享一个空间数据模型**，各子系统是它的不同「视图」。

```
                    ┌─────────────────────┐
                    │   空间数据模型 (Spatial) │
                    │   grid + spaces + objects │
                    └──────────┬──────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
    🥬 田间视图           🏠 住宿视图          📦 物品视图
   "哪些格子是菜地"      "哪些房间有人住"     "每个空间有什么物品"
   "每块地种了什么"      "入住/离开日期"      "数量/新鲜度/位置"
```

### 3.2 空间定义（从实景地图延伸）

```js
// ===== 空间数据 (nt_map_data) =====
// 这是实景地图编辑器已有的数据结构，在此之上扩展
{
  boards: {
    "world": {   // 方位图 (15×20)
      grid: [...],       // 每个格子的填色+边框（已有）
      objects: [...]     // emoji 标注物体（已有）
    },
    "dorm": {    // 宿舍 (22×30)
      grid: [...],
      objects: [...]
    },
    "coop": {    // 合作社 (22×28)
      surroundings: [...],  // 周边层（所有楼层共用）
      floors: {
        "1F": { grid: [...], objects: [...] },
        "2F": { grid: [...], objects: [...] },
        "3F": { grid: [...], objects: [...] }
      }
    },
    "field": {   // 菜地 (11×15)
      grid: [...],
      objects: [...]
    }
  }
}
```

### 3.3 空间语义层（新增——给地图上的区域赋予「意义」）

```js

  "dorm-101": {
    id: "dorm-101",
    name: "宿舍 101",
    board: "dorm",            // 属于哪个地图板

    gridArea: {               // 在地图上占据的格子范围
      x1: 3, y1: 2, x2: 7, y2: 6
    },
    type: "bedroom",          // bedroom | kitchen | bathroom | hallway | studio | storage | outdoor | field
    capacity: 2,              // 最多容纳人数（床位）
    floorPlan: "1F",          // 所属楼层
    photos: [],               // 标准照 URL 列表（远期）
    status: "clean",          // clean | needs_cleaning | cleaning
    lastCleaned: "2026-07-05",
    cleaningSchedule: "weekly" // weekly | biweekly | monthly
  },
  "field-A": {
    id: "field-A",
    name: "菜地 A 区",
    board: "field",
    gridArea: { x1: 0, y1: 0, x2: 4, y2: 4 },
    type: "field",
    plots: [                  // 细分的种植格
      { id: "plot-A1", crop: "番茄", planted: "2026-06-15", days: 55, status: "growing" },
      { id: "plot-A2", crop: "玉米", planted: "2026-05-20", days: 80, status: "ready" }

```

## 四、每个子系统详细设计

### 4.0 空间地图（地基——已在实景地图编辑器中实现）

**已有能力**：
- 四张网格地图（方位图/宿舍/合作社/菜地）
- 格子填色（路/田/水/房）+ 边界（墙/窗/门/围栏）
- emoji 物体标注 + 分类
- 编辑/查看双模式
- 楼层切换（合作社 1F/2F/3F）
- 周边层 + 楼层数据合并

**云村需要新增的**：
- 空间语义层（`nt_spaces`）：给网格上的区域赋予「名字」和「类型」
- 点击空间 → 弹出信息面板（物品/作物/住宿/任务）
- 小人图标叠加（谁在哪个空间）

**和地图编辑器的关系**：
- 地图编辑器负责「画地图」（编辑网格）
- 云村负责「用地图」（在网格上叠加语义数据）
- 两者共享 `nt_map_data`，云村只读
- 云村的 `nt_spaces` 是对地图网格的「命名层」

### 4.2 住宿管理

```
① 管理员定义房间
   nt_spaces["dorm-101"] = {
     name: "宿舍 101", type: "bedroom", capacity: 2
   }

② 有人入住 → 登记
   nt_stay_data.records.push({
     person: "王五", space: "dorm-101",
     checkIn: "7/3", checkOut: "7/6",
     dailyRate: 70, total: 210, paid: false
   })

③ 地图上显示
   宿舍 101 格子 → 🟡 已入住（王五 7/3-7/6）
   空房间 → 🟢 可入住

④ 退房时
   管理员标记 paid: true
   房间状态恢复 🟢

⑤ 费用汇总
   本月住宿收入：NT xxx
   待付：王五 NT 210、张三 NT 140
```

### 4.3 物品仓库

```
物品全程追踪：

  来源          →    位置      →   状态       →   去向
  ────────         ──────        ──────        ──────
  田间收获         厨房冰箱       新鲜          被吃掉
  采购             储物间         存放中        被使用
  捐赠             画室           已开封        过期丢弃
  制作产出         展览厅         变质⚠️

  每个物品都有：
  - 名称、数量、单位
  - 所在空间（关联 nt_spaces ID）
  - 存储方式（冷藏/常温/冷冻）
  - 收获/购入日期
```

```
     │ 共享 nt_session           │                        │ 共享 nt_users

     │                           │                        │ 读写 nt_spaces
```

**关键原则**：
- dashboard.html 是「管理后台」——管理员/共建者用
- 实景地图.html 是「空间编辑器」——共建者编辑空间布局
- cloud-village.html 是「云村入口」——游客/NPC/所有人查看和互动
- 三者通过 localStorage key 契约打通，不需要互相引用代码

## 八、技术约束 & 注意事项

1. **同源策略**：三个 HTML 必须在同一域名/本地文件夹下，才能共享 localStorage
2. **数据一致**：所有写入 localStorage 的操作，必须在 try/catch 中处理 JSON 解析错误
3. **冲突处理**：多个标签页同时写同一个 key → 最后写入者胜。对于高频写入的场景（如实时状态），需要考虑 debounce
4. **存储上限**：localStorage 通常 5-10MB。地图网格数据 + 物品 + 日志累积可能超出。Phase 4 需要考虑数据归档策略
5. **天气 API**：和风天气免费版每天 1000 次调用，足够用。需要用户注册获取 API Key
6. **地图嵌入**：Phase 1 用 iframe 嵌入实景地图是最简单的方案。后续可以改为直接引用 JS 模块
