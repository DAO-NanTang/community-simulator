# 50-100 人实时协作部署方案

> 2026-07-10 | v3 审查修订：明确数据安全保障 + 砍掉过度设计 + 补充回退方案

---

## 〇、当前瓶颈分析

| 组件 | 当前实现 | 100 人并发时的问题 |
|------|---------|------|
| 数据存储 | 浏览器 localStorage | 每人数据隔离，看不到别人的操作 |
| 用户认证 | localStorage 明文存密码 | 无安全性，换浏览器绕过 |
| 数据同步 | 无——纯本地 | 你发布的任务别人看不到 |
| NT 余额 | 本地计算 | 两人同时审核 → 数据覆盖 |
| **操作响应** | `localStorage.setItem()` 同步写入 | **迁移后如果改成 `await fetch()`，每次点击要等 50-100ms 网络往返** |
| 文件大小 | 652KB 单文件 | 静态文件多快都不影响 |

**两个核心结论**：
1. 从 localStorage → 服务器是必须的，否则根本不能多人协作
2. **但简单把 `localStorage.setItem()` 替换成 `await fetch()` 会让每次点击变慢**——需要用「乐观更新」解决

### 为什么不能省掉后端（数据安全角度）

有人可能会想：能不能把 JSON 文件放到 GitHub Gist 或腾讯云 COS 上，大家共用这一个文件？——**不能**。原因：

| 风险 | 后果 | 真实概率 |
|------|------|:--:|
| 两人同时保存 → 后写覆盖先写 | 审核结果消失、NT 对不上 | **高**（多人协作必然发生） |
| 写文件到一半网络中断 | 整个 data 对象损坏、所有人白屏 | **中**（移动网络/弱网环境常见） |
| 审核的 3 步只完成了 2 步 | 状态改了但 NT 没发，或 NT 发了但没记流水 | **中**（无事务保护） |
| 存储服务限流/故障 | 所有人操作失败、数据写不进去 | **低**（但后果严重——全员停摆） |

**SQLite + 原子事务是解决以上问题的"最低可行方案"**。没有比它更简单同时又能保障数据安全的方案了。

## 一、架构选型

### 最终方案：SQLite (WAL) + Node.js Express + SSE 实时推送

```
浏览器 ──→ Nginx（静态文件 + SSE 直通 + API 反向代理）
               │
               ├── dashboard.html（静态文件，一次加载）
               │
               ├── /api/* ──→ Node.js Express（业务逻辑）
               │                 │
               │                 ├── data.db（SQLite，WAL 模式）
               │                 └── EventBus（SSE 广播）
               │
               └── /api/stream ──→ SSE 长连接（每人一条，实时接收变更）
```

### 为什么还是 SQLite？

| 对比维度 | SQLite (WAL) | PostgreSQL |
|---------|:--:|-----------|
| 100 人并发读 | ✅ WAL 模式下无限并发读 | 更强 |
| 100 人并发写 | ✅ 串行化写入，<5ms/次，几乎不排队 | 更强 |
| 安装 | 零配置 | 需要安装数据库服务 |
| 备份 | 复制一个文件 | 需要 pg_dump |
| 运维 | 零运维 | 需要维护数据库进程 |
| 本项目适用性 | ✅ 完全够用 | 100 人还是过度设计 |

**SQLite WAL 模式是关键**：默认的 DELETE 模式下，读和写互斥——一个人写数据时其他人读不了。开了 WAL 后，读和写可以同时进行，100 人同时读 + 偶尔写完全无压力。

### 为什么不选 WebSocket 而选 SSE？

| | SSE (Server-Sent Events) | WebSocket |
|------|:--:|:--:|
| 浏览器支持 | 所有浏览器原生支持 `EventSource` | 需要库或手写 |
| 协议 | HTTP，复用 Nginx 配置 | 独立协议 `ws://`，Nginx 需额外配置 |
| 方向 | 服务器 → 浏览器（正是我们需要的） | 双向（我们用不上客户端→服务器这条） |
| 自动重连 | `EventSource` 自带 | 需要手动实现 |
| 代码量 | 后端 ~30 行，前端 ~20 行 | 后端 ~80 行，前端 ~60 行 |
| 100 连接资源 | 几乎无开销 | 几乎无开销 |

**结论**：客户端→服务器的数据走普通 HTTP POST（按钮点击），服务器→所有客户端的广播走 SSE。刚好匹配需求，最简单。

## 二、数据准确性保障

> 这是 100 人场景下最容易出问题的部分，需要多管齐下。

### 2.1 三层防护体系

```
第一层：SQLite WAL + 事务
  ↓  确保单个写入的原子性
第二层：乐观锁（版本号）
  ↓  防止「A 和 B 同时改了同一条数据，后写的覆盖先写的」
第三层：服务端原子操作
  ↓  审核 = 改任务状态 + 扣 NT + 记流水 → 要么全成功，要么全失败
```

### 2.2 第一层：SQLite WAL 模式

```
-- 启动时执行（只需一次）
PRAGMA journal_mode=WAL;     -- 读写不互斥
PRAGMA synchronous=NORMAL;   -- 安全性和性能的平衡点
PRAGMA busy_timeout=5000;    -- 写锁等待最多 5 秒
```

**效果**：
- WAL 模式下，100 个人同时读取数据不会被一个人的写入阻塞
- 写入串行化（SQLite 的硬限制），但单次写入 <5ms，100 人同时写的极端情况总耗时 <500ms
- `busy_timeout=5000` 让等待写入的请求排队而不是直接报错

### 2.3 第二层：乐观锁（行级版本号）

**问题场景**：管理员 A 和 B 同时打开任务列表，A 审核了任务 #3，B 审核了任务 #7。两个操作修改的是同一个 `data` 对象的数组。如果直接全量覆盖，后写的会把先写的审核结果弄丢。

**方案：每次修改带版本号**

```
数据库记录：
{
  "id": "tasks",
  "data_json": "{...所有任务...}",
  "version": 42
}

A 审核任务 #3：
  POST /api/sync/review  { taskIdx: 3, action: "approve", version: 42 }
  → 后端检查：数据库 version == 42？✅ 匹配
  → 修改数据 → version 变成 43 → 返回 { ok: true, newVersion: 43 }

B 审核任务 #7（几乎同时）：
  POST /api/sync/review  { taskIdx: 7, action: "approve", version: 42 }
  → 后端检查：数据库 version == 42？❌ 已经是 43 了
  → 返回 409 Conflict → 前端自动重试（拿最新数据 + 重新应用改动 + 再提交）
```

**实际冲突率极低**：100 人场景，每人的操作间隔至少几秒，两次写操作撞在同一个 5ms 窗口内的概率 ≈ 0。乐观锁是最后一道保险，不是常用路径。

### 2.4 第三层：服务端原子事务

**最危险的场景**：审核通过一个任务涉及三个操作：
1. 改任务状态 → 「已认领」
2. 给认领人加 NT → `finance` 数组追加入账记录
3. 记流水 → `transactions` 数组追加

**三个操作必须原子化**——如果第 2 步成功了但第 3 步崩溃了，NT 余额就对不上了。

```
// 后端：一个 SQLite 事务包裹所有操作
db.transaction(() => {
  // 1. 更新任务状态
  updateTaskStatus(taskIdx, 'approved');
  // 2. 创建入账记录
  createFinanceRecord({ user, amount, type: 'task_reward' });
  // 3. 追加流水
  addTransaction({ user, amount, reason, timestamp });
})();
// 要么三个全成功，要么全部回滚 → 数据永远不会不一致
```

### 2.5 数据准确性底线

| 场景 | 防护手段 | 会不会丢数据？ |
|------|------|:--:|
| 两人同时审核不同任务 | 乐观锁 + 增量操作 | ❌ 不会 |
| 两人同时审核同一个任务 | 乐观锁 → 先到先得，后者 409 | ❌ 不会 |
| 服务器中途崩溃 | SQLite WAL 崩溃恢复 | ❌ 不会（WAL 自动回滚未完成事务） |
| NT 余额计算错误 | 服务端原子事务计算 | ❌ 不会 |
| 网络断开导致请求丢失 | 前端收到错误 → 重试 | ❌ 不会 |

## 三、实时同步策略（SSE 替代轮询）

### 3.1 对比

| | 30 秒轮询 | 3 秒轮询 | SSE 推送 |
|------|:--:|:--:|:--:|
| 同步延迟 | ≤30s | ≤3s | **<100ms** |
| 服务器负载 | 每 30s × 100 人 = 3.3 次请求/秒 | 每 3s × 100 人 = 33 次/秒 | 100 条长连接，0 次轮询 |
| 带宽浪费 | 每次都传全量数据 | 每次都传全量数据 | 只传变更部分 |
| 代码复杂度 | 低 | 低 | 中（约 50 行净增） |
| 用户体验 | 差 | 勉强 | ⭐ 接近实时 |

### 3.2 SSE 工作原理

```
每个浏览器打开页面时：
  1. 建立一条 GET /api/stream?token=xxx 的长连接（一直保持打开）
  2. 服务器不关闭这条连接，有新数据就往里写

当用户 A 做任何操作：
  A 点击按钮 → POST /api/sync/xxx → 服务器处理 → 写入 DB
                                              ↓
                              服务器遍历所有 SSE 连接 → 推送变更事件
                                              ↓
                              B、C、D... 所有在线的人收到事件 → 局部刷新 UI
```

### 3.3 前端代码（约 20 行）

```javascript
// api.js —— 新增 SSE 监听
function connectSSE(token) {
  const es = new EventSource('/api/stream?token=' + token);

  es.onmessage = function(event) {
    const change = JSON.parse(event.data);
    // 只刷新受影响的区域
    switch (change.type) {
      case 'task_created':
        appendTaskRow(change.task);           // 追加一行，不重绘整个列表
        break;
      case 'task_reviewed':
        updateTaskRow(change.idx, change);    // 只更新那一行
        refreshStats();                        // 只刷新统计数字
        break;
      case 'member_updated':
        updateMemberCard(change.name, change);
        break;
      case 'finance_changed':
        refreshFinanceTable();
        refreshStats();
        break;
    }
  };

  // EventSource 自动重连，但重连期间可能错过事件
  es.onerror = function() {
    // 断线重连后，拉一次增量数据补漏
    setTimeout(() => pullMissedChanges(lastKnownVersion), 2000);
  };

  return es;
}
```

### 3.4 后端代码（约 40 行）

```javascript
// server.js —— SSE 端点
const clients = new Set();

app.get('/api/stream', (req, res) => {
  // 设置 SSE 必需的响应头
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
    'X-Accel-Buffering': 'no',  // 告诉 Nginx 不要缓冲
  });

  clients.add(res);
  req.on('close', () => clients.delete(res));

  // 每 30 秒发心跳，保持连接（防止 Nginx/浏览器超时断开）
  const heartbeat = setInterval(() => res.write(': heartbeat\n\n'), 30000);
  req.on('close', () => clearInterval(heartbeat));
});

// 广播函数：任何数据变更后调用
function broadcast(change) {
  const msg = 'data: ' + JSON.stringify(change) + '\n\n';
  for (const client of clients) {
    client.write(msg);
  }
}

// 在每次 API 写入后调用
app.post('/api/sync/review', (req, res) => {
  // ... 更新数据库 ...
  broadcast({ type: 'task_reviewed', idx: req.body.idx, ...result });
  res.json({ ok: true });
});
```

### 3.5 断线重连 + 补漏机制

```
正常状态：SSE 连接保持，变更实时推送（<100ms）

断线（网络波动、服务器重启）：
  1. EventSource 自动重连（浏览器内置行为）
  2. 重连成功后，前端调 GET /api/data → 拿到最新全量数据
  3. 覆盖本地 data 对象 → updateAll() 全量刷新一次 → 恢复到最新状态

心跳：每 30 秒发一个注释帧（: heartbeat），保持连接不超时
```

> 为什么不做增量补漏（`/api/changes?since=ver`）？SSE 断线概率低（Nginx 24h 超时 + EventSource 自动重连），重连后全量拉取一次（~50KB）远少于维护变更日志的代码量（~50 行后端 + ~20 行前端）。用复杂度换一个几乎不会触发的优化不值得。

## 四、操作流畅性：乐观更新

> 这是迁移到服务器后最容易忽略的问题。

### 4.1 问题

| 操作 | localStorage 时代 | 简单 API 迁移 | 用户体感 |
|------|:--:|:--:|------|
| 点击「审核通过」 | 0ms（同步写磁盘） | ~80ms（网络往返） | **每点一下顿 80ms，非常不舒服** |
| 发布任务 | 0ms | ~80ms | 同上 |
| 修改成员信息 | 0ms | ~80ms | 同上 |
| 打开弹窗 | 0ms（纯前端） | 0ms（不涉及 API） | ✅ 不受影响 |
| 切换 Tab | 0ms（纯前端） | 0ms（不涉及 API） | ✅ 不受影响 |
| 页面首次加载 | 0ms（读磁盘） | ~100ms（网络请求） | ⚠️ 首次加载多 100ms，可接受 |

**核心差异**：按钮点击从 0ms → 80ms，用户感觉到的是「不跟手」。

### 4.2 解决方案：乐观更新

**原则**：用户操作 → **立刻更新 UI** → 后台发 API → 失败了再回滚。

```
用户点击「审核通过」
  │
  ├── 1. 立即更新 UI（0ms）
  │      - 任务行变绿
  │      - NT 余额变化
  │      - 按钮变灰
  │      - 统计数字刷新
  │      - 用户感觉：瞬间完成 ✅
  │
  └── 2. 后台发送 POST /api/sync/review
         │
         ├── 成功（99.9%）→ 什么都不做，UI 已就位
         │     （SSE 广播给其他人，别人也立刻看到）
         │
         └── 失败 409 冲突（0.1%）→ 弹窗提示「操作冲突，已自动刷新」
               → 重新拉取最新数据 → 自动合并 → UI 更新为正确状态
```

### 4.3 乐观更新实现（前端改动）

```javascript
// 审核通过 — 乐观更新版
async function approveTask(taskIdx) {
  // 1. 乐观更新：立刻改 UI
  const oldState = snapshotTaskState(taskIdx);  // 备份当前状态，用于回滚
  applyTaskApprovedUI(taskIdx);                  // 任务变绿、NT 变化、统计刷新
  showPendingIndicator(taskIdx);                 // 行尾显示一个微小的 ⌛ 旋转图标

  try {
    // 2. 后台发 API
    const result = await api.syncReview({
      idx: taskIdx,
      action: 'approve',
      version: data._version
    });

    // 3. 成功：更新版本号，移除 ⌛
    data._version = result.newVersion;
    hidePendingIndicator(taskIdx);

  } catch (err) {
    // 4. 失败：回滚 UI + 提示用户
    if (err.status === 409) {
      restoreTaskState(taskIdx, oldState);       // 恢复原来的样子
      const latest = await api.getData();        // 拿最新数据
      mergeNewData(latest);                      // 合并
      updateAll();                               // 全量刷新（这次是真的）
      showToast('操作冲突，数据已刷新，请重试');
    } else {
      restoreTaskState(taskIdx, oldState);
      showToast('网络错误，请重试');
    }
  }
}
```

### 4.4 乐观更新的适用范围

| 操作 | 乐观更新？ | 理由 |
|------|:--:|------|
| 审核通过/退回 | ✅ 乐观 | 冲突率极低，用户期望瞬间反馈 |
| 发布新任务 | ✅ 乐观 | 同上，几乎没有冲突可能 |
| 修改成员信息 | ✅ 乐观 | 同一成员不太可能被两人同时编辑 |
| NT 转账 | ⚠️ 半乐观 | 先乐观更新 UI，但加一个余额检查：如果服务器返回「余额不足」，回滚 |
| 注册/登录 | ❌ 不乐观 | 必须等服务器确认，不能先让用户「以为登录成功」 |

### 4.5 操作延迟总结

| 操作 | 体感延迟 | 说明 |
|------|:--:|------|
| 点击按钮（审核、发布、编辑等） | **0ms** | 乐观更新，和 localStorage 时代一样快 |
| 打开弹窗 | **0ms** | 纯前端，不涉及 API |
| 切换 Tab | **0ms** | 纯前端 |
| 首次加载页面 | ~100ms | 初始拉取数据，一次性的 |
| 错误回滚 | ~200ms | 极少发生，发生时弹窗提示 |

## 五、100 人规模瓶颈分析

### 5.1 各组件承受力

| 组件 | 100 人负载 | 是否瓶颈 | 说明 |
|------|------|:--:|------|
| **Nginx 静态文件** | 100 × 652KB 同时加载 | ❌ 否 | 静态文件走 CDN/缓存，首次后浏览器缓存 |
| **Node.js Express** | 100 SSE 连接 + 偶发 API 请求 | ❌ 否 | 事件驱动，单核轻松扛 10000 并发连接 |
| **SQLite 读取** | WAL 模式，无限并发读 | ❌ 否 | 每次读 <5ms |
| **SQLite 写入** | 串行化，单次 <5ms | ❌ 否 | 100 人同时写极端情况 500ms 排完，但实际不会同时写 |
| **SSE 广播** | 每次写入后遍历 100 个连接发送 | ❌ 否 | Node.js 异步 I/O，100 次 res.write() 几乎是瞬间 |
| **服务器内存** | 100 SSE 连接 + SQLite 缓存 | ❌ 否 | ~150MB，1G 内存绰绰有余 |
| **带宽** | SSE 心跳 + 偶发 JSON 推送 | ❌ 否 | 3Mbps 足够，SSE 空闲时只有心跳 |

### 5.2 真正的瓶颈（诚实评估）

100 人规模下，**唯一的理论瓶颈是 SQLite 写入串行化**。但实际场景：

```
真实操作频率估算（100 人在线）：
  - 浏览任务、看数据（只读）：95% 的时间
  - 审核通过/退回：平均每人每小时 5-10 次 → 总共 500-1000 次/小时 ≈ 0.3 次/秒
  - 发布新任务：平均每人每小时 1-2 次 → 总共 100-200 次/小时 ≈ 0.06 次/秒
  - 修改成员信息：几乎为零

写入总频率：<1 次/秒
SQLite 写入能力：>100 次/秒（有索引的情况下）

结论：写入负载是 SQLite 上限的 1%，完全不用担心。
```

### 5.3 什么时候需要升级到 PostgreSQL？

| 指标 | SQLite 上限 | 本项目 100 人 |
|------|:--:|:--:|
| 并发读 | 无限（WAL） | ~100 |
| 写入/秒 | ~200 | <1 |
| 数据库大小 | 建议 <1GB | <10MB |
| 同时写入者 | 1（串行） | 几乎不会同时 |

**升级阈值**：同时在线 >500 人，或写入频率 >50 次/秒。在此之前 SQLite 绰绰有余。

```nginx
server {
    listen 80;
    server_name your-domain.com;
    root /opt/gnt-server/public;

    # 静态文件
    location / {
        try_files $uri /dashboard.html;
    }

    # API 代理
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # SSE 端点 —— 关键配置！
    location /api/stream {
        proxy_pass http://127.0.0.1:3000;

    }
```

## 八、潜在问题与对策

> 用户问：「会出现什么问题？」——诚实回答。

### 8.1 SSE 连接断开

| 问题 | 影响 | 对策 |
|------|------|------|
| 用户网络波动 | SSE 断开，收不到实时更新 | EventSource 自动重连 + 重连后调 `/api/changes` 补漏 |
| Nginx 超时断开 | 同上 | `proxy_read_timeout 86400s`（24 小时） |
| 服务器重启 | 所有 SSE 断开 | pm2 自动重启 + 所有客户端自动重连 |

### 8.2 浏览器连接数限制

**每个域名同时最多 6 个 HTTP/1.1 连接**（浏览器硬限制）：
- SSE 占用 1 个
- 剩余 5 个给 API 请求
- 正常情况下够用（用户不会同时发 5 个 API 请求）
- 如果有大量图片加载，可以考虑把图片放到 CDN 子域名

### 8.3 内存泄漏（SSE 连接管理）

`clients` Set 中死连接（用户关闭了标签页但 `close` 事件没触发）会累积。对策：
- 心跳检测：每 30 秒发心跳，如果 `res.write()` 失败（连接已断开），从 Set 中移除
- pm2 设置内存上限（150MB 自动重启），作为兜底

### 8.4 初始数据迁移

从 localStorage 到服务器的数据迁移需要一个策略：

```
方案：管理员导出 → 服务器导入

1. 在现有 dashboard.html 上加一个「导出数据」按钮
   - 读取 localStorage 中所有数据
   - 下载为一个 data.json 文件

2. 管理员部署服务器后运行：
   node migrate.js --from=data.json

3. 迁移完成后，管理员用新系统登录验证数据完整性

4. 旧系统加一个横幅：「系统已迁移到 xxx.com，请使用新地址访问」
```

### 8.5 单点故障

整个系统在一台服务器上，服务器挂了所有人都用不了。对 100 人团队：

| 故障 | 恢复时间 | 数据安全 |
|------|:--:|------|
| 进程崩溃 | pm2 自动重启（<2 秒） | WAL 保护，不丢数据 |
| 服务器宕机 | 手动重启（<30 分钟） | 磁盘数据完好 |
| 磁盘损坏 | 从备份恢复（<1 小时） | 最多丢一天的数据 |

**对于 100 人的共创营场景，单点故障是可接受的。** 不需要主从复制、负载均衡等方案。如果后续需要高可用，最简单的做法是每天备份 + 准备一台备用服务器。

### 8.6 用户隔离

所有 100 人读写的同一份 `data` 对象。需要注意：

- 权限控制在后端做（普通成员不能审核任务）
- JWT token 中带角色信息，后端每个 API 检查权限
- 不需要做多租户隔离（就是一个营地）

## 十二、补充说明（审查后追加）

### 12.1 数据安全底线：为什么这套方案不会丢数据

| 保护层 | 防什么 | 实现 |
|------|------|------|
| SQLite WAL | 写操作中途崩溃 | WAL 自动回滚未完成事务，数据库不会损坏 |
| 原子事务 | 审核=改状态+发NT+记流水，三步只完成两步 | `db.transaction()` 包裹，要么全成功要么全回滚 |
| 乐观锁 | 两人同时改同一条数据 | 版本号不匹配 → 409 → 前端自动重试 |
| 每日备份 | 磁盘损坏/误删 | crontab 每天备份，7 天轮转，最多丢一天数据 |
| pm2 守护 | Node.js 进程崩溃 | 自动重启，<2 秒恢复 |

这五层保护加起来，数据安全性**远高于** localStorage（目前方案）和任何"零后端共享文件"方案。

### 12.2 回退方案：服务器 → 单机

如果服务器出问题，可以随时退回到 localStorage 单机模式：

1. 服务器上 `cp data.db backup.db` → 所有数据完好
2. 写一个导出脚本 `export.js`：读取 SQLite → 输出 `data.json`
3. 旧版 dashboard.html 加载 `data.json` → 恢复单机运行
4. 所有用户的数据（tasks/members/finance）都在那个 JSON 里，一个不少

迁移不是单向的。数据格式保持不变（JSON），只换了存储位置。

### 12.3 全量 JSON 存储的权衡

当前设计把整个 `data` 对象作为一个 JSON 字段存在 SQLite 中（而非拆分为 tasks/members/finance 等多个关系型表）。

**利**：前端适配成本最低（数据结构完全不变）、迁移简单（直接导出 JSON）

**弊**：每次读取返回全量数据（~50KB）、无法用 SQL 做聚合查询（如"本月每人 NT 收入"需要前端遍历）

**远期升级路径**：当 `data` 对象超过 500KB 或需要服务端生成报表时，新增关系型表（`tasks` / `finance` / `members` 各自独立），写一个迁移脚本将 JSON 拆入各表。API 接口不变——前端无感知。

### 12.4 局部刷新函数与全量渲染的共存

方案中提到的 `updateTaskRow()` / `appendTaskRow()` / `updateMemberCard()` 等局部刷新函数是**新增代码**（约 40 行），与现有 `renderWorkspace()` / `renderQuestBoard()` 的全量 innerHTML 替换并存：

- **SSE 事件来时**：调用局部刷新函数（仅更新受影响的行，不碰其他内容）
- **用户切换 Tab 时**：调用全量渲染函数（现有逻辑不变）
- **冲突回滚时**：调用 `updateAll()`（确保所有视图一致）

两者不冲突——SSE 负责"别人改了东西我立刻看到"，Tab 切换负责"我去看看别的页面"。

### 12.5 规模说明：50 还是 100？

方案同时提到 50 人和 100 人。诚实地说：SQLite + Express 在这个数量级上没有区别——都不构成瓶颈。写 50 还是 100 主要影响服务器配置推荐（1 核够 50 人，2 核更宽裕）。实际部署时按当前营地规模选择即可，后续扩容只需升级云服务器配置，不动代码。
