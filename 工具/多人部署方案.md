# 50 人并发部署方案

> 2026-07-10 | 从单用户 localStorage → 多人服务器

---

## 〇、当前瓶颈分析

| 组件 | 当前实现 | 50 人并发时的问题 |
|------|---------|------|
| 数据存储 | 浏览器 localStorage | 每人数据隔离，**看不到别人的任务/流水** |
| 用户认证 | localStorage 明文存密码 | 换个浏览器就能绕过，**无安全性** |
| 数据同步 | 无——纯本地 | 你发布的任务别人**看不到** |
| NT 余额 | 本地计算 | 两个人同时审核 → **数据覆盖** |
| 文件大小 | 652KB 单文件 | **不影响并发**——静态文件多快都行 |

**核心结论**：当前系统在服务器上不会卡顿（静态文件），但**根本上不支持多用户协作**。50 人同时访问需要每个人看到同一份数据，这必须引入后端。

---

## 一、架构选型

### 推荐方案：SQLite + Node.js Express（轻量级）

```
浏览器 ──→ Nginx（静态文件 + 反向代理）
               │
               ├── dashboard.html（前端，改动 ~200行）
               │
               └── /api/* ──→ Node.js Express（后端，~500行）
                                │
                                └── data.db（SQLite 单文件）
```

### 为什么选 SQLite 而不是 PostgreSQL/MySQL？

| 对比维度 | SQLite | PostgreSQL |
|---------|--------|-----------|
| 安装 | 零配置，npm install 即用 | 需要安装数据库服务 |
| 50 人并发读写 | ✅ 足够（SQLite 支持 ~100 并发读） | 更强但用不上 |
| 备份 | 复制 data.db 一个文件 | 需要 pg_dump |
| 服务器要求 | 1 核 512MB 足够 | 建议 2 核 2GB |
| 运维 | 零运维 | 需要维护数据库进程 |
| 本项目适用性 | ✅ 完全够用 | 过度设计 |

**50 人场景下 SQLite 完全不会成为瓶颈**。等用户规模超过 200 人再考虑升级。

## 二、需要改什么（按改动量排序）

### 层面一：前端改动（~200 行）

**核心策略**：把所有 `localStorage.getItem/setItem` 和 `data = JSON.parse(...)` 替换为 API 调用。其他 UI 逻辑不动。

改动清单：

| 当前代码 | 改为 |
|---------|------|
| `localStorage.getItem('camp_data')` | `await api.getData()` |
| `localStorage.setItem('camp_data', JSON.stringify(data))` | `await api.saveData(data)` |
| `localStorage.getItem('nt_users')` | `await api.getUsers()` |
| `saveUsers(users)` | `await api.saveUsers(users)` |
| `localStorage.getItem('nt_invite_codes')` | `await api.getInviteCodes()` |
| `saveInviteCodes(codes)` | `await api.saveInviteCodes(codes)` |
| `encodePassword(pwd, name)` | 后端处理，前端只传明文 |
| `JSON.parse(session)` | `await api.validateSession(token)` |

**新增**：
- 登录时拿到 `token`，存入 `sessionStorage`（不是 localStorage）
- 每次 API 调用带 `Authorization: Bearer <token>`
- 数据变更后不调用 `updateAll()` 的全量刷新——改为只刷新当前可见 tab

### 层面二：后端 API（~500 行新代码）

单文件 `server.js`，启动命令 `node server.js`：

```
POST   /api/login          # 登录 → 返回 token
POST   /api/register       # 注册
GET    /api/data           # 获取营地数据（data对象）
POST   /api/data           # 保存营地数据（全量同步）
GET    /api/users          # 获取用户列表
POST   /api/users          # 保存用户
GET    /api/invite-codes   # 获取邀请码
POST   /api/invite-codes   # 保存邀请码
POST   /api/sync           # 增量同步（只传变更的 task/finance/member）
```

### 层面三：数据库（自动创建，无需手动建表）

SQLite 数据库结构（由后端首次启动时自动创建）：

```
users        — 用户表（name, password_hash, role, avatar_seed, uid, season, created）
camp_data    — 营地数据表（id, data_json TEXT, updated_at）
invite_codes — 邀请码表（code, type, created_at）
sessions     — 登录会话表（token, username, created_at, expires_at）
```

## 三、数据同步策略（解决 50 人并发核心问题）

### 问题：多人同时修改数据怎么办？

场景：管理员 A 审核通过任务 X，同时管理员 B 审核通过任务 Y。两人都拿到了 `data` 的副本，修改后各自写回——后写的人会覆盖先写的人。

### 方案：乐观锁 + 增量同步

每条数据记录加 `version` 字段：

```
写入时：WHERE version = 我读到的版本 → 成功 → version + 1
        WHERE version ≠ 我读到的版本 → 冲突 → 重试（重新读取+合并+写入）
```

对于本项目，最简实现：

1. `data` 对象存在数据库里，带一个 `version` 数字
2. 前端保存时带 `version` 参数
3. 后端检查：`version` 匹配 → 写入成功；不匹配 → 返回 409 Conflict
4. 前端收到 409 → 重新拉取最新数据 → 合并本地改动 → 再次保存

**对于本项目的实际场景**（50 人、任务系统），冲突率很低——任务 A 和任务 B 是不同的数组元素，不会互相覆盖。乐观锁主要保护 `finance` 数组和 `members` 对象。

### 优化：增量同步

当前 `updateAll()` 每次都全量保存整个 `data` 对象。改为增量：

| 操作 | 发送数据 | API |
|------|---------|------|
| 发布任务 | `{task: {...}}` | `POST /api/sync/task` |
| 审核通过 | `{idx: 3, claimant: '宏心', action: 'approve'}` | `POST /api/sync/review` |
| 修改成员 | `{name: '宏心', role: '核心成员'}` | `POST /api/sync/member` |

增量同步后，前端不需要在每次操作后全量保存——只传变更部分，后端负责合并。

## 四、性能预估（50 人同时在线）

| 场景 | 操作 | SQLite 耗时 | 网络耗时 | 总耗时 |
|------|------|:--:|:--:|:--:|
| 页面首次加载 | GET /api/data（~50KB JSON） | <5ms | ~50ms | <100ms |
| 发布任务 | POST /api/sync/task | <2ms | ~30ms | <50ms |
| 审核通过 | POST /api/sync/review | <3ms | ~30ms | <50ms |
| 50 人同时刷新 | 50 × GET /api/data | ~50ms | 各自独立 | 无排队 |

**结论**：50 人并发下，SQLite + Express 的响应时间在 100ms 以内，用户完全无感知。

## 五、部署方案

### 最低配置服务器

| 配置 | 要求 |
|------|------|
| CPU | 1 核 |
| 内存 | 512 MB |
| 磁盘 | 1 GB（含 data.db + 备份） |
| 系统 | Ubuntu 20.04 / Debian 11 |
| Node.js | v18+ |

### 部署步骤

```bash
# 1. 安装 Node.js
curl -fsSL https://deb.nodesource.com/setup_18.x | bash -
apt-get install -y nodejs nginx

# 2. 创建项目目录
mkdir -p /opt/gnt-server
cd /opt/gnt-server

# 3. 初始化
npm init -y
npm install express better-sqlite3 bcryptjs jsonwebtoken cors

# 4. 上传文件
# - dashboard.html → /opt/gnt-server/public/
# - server.js      → /opt/gnt-server/
# - 图片文件       → /opt/gnt-server/public/

# 5. 配置 Nginx
# - 静态文件：/opt/gnt-server/public/
# - API 代理：/api/* → http://localhost:3000

# 6. 启动
node server.js
```

### Nginx 配置（约 15 行）

```
server {
    listen 80;
    server_name your-domain.com;
    root /opt/gnt-server/public;
    
    location / {
        try_files $uri /dashboard.html;
    }
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
    }
}
```

## 七、不做（明确排除）

- ❌ **WebSocket 实时推送**：50 人的团队协作不需要毫秒级实时。30 秒轮询足够，复杂度降低 10 倍。
- ❌ **Redis 缓存**：SQLite 直接读够快。加缓存反而引入一致性问题。
- ❌ **Docker/K8s**：单机部署，不需要容器化。`node server.js` 足够。
- ❌ **前端框架重写**：dashboard.html 不改 UI，只改数据层。React/Vue 对这个项目是过度设计。
- ❌ **OAuth/第三方登录**：当前密码登录够用。后续可加但不在本次范围。
