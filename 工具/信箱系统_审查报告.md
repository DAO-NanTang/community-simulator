# 《信箱系统_完整方案.md》审查报告

> 审查对象：[信箱系统_完整方案.md](信箱系统_完整方案.md)
> 对照文件：dashboard.html（当前 6,300+ 行）+ [监察面板_重构方案.md](监察面板_重构方案.md)
> 审查日期：2026-07-07

---

## 总体判断

**方向是对的——信息到行动的闭环是真实存在的痛点。** 管理员现在确实要"记住数字 → 切 tab → 找条目"，这个心智负担存在。信箱作为统一通知入口的思路，比监察仪表盘更进一步——它不仅告诉你"有什么事"，还帮你"直接跳过去"。

但有三个实施层面的问题需要重新考虑：消息存储方式增加了不必要的复杂度、跳转定位逻辑散布太广、和监察方案的关系没有完全理清。下面逐条说。

---

## 一、逐原则审查

### 原则1：少即是多

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 新增元素数量 | 🟡 | HUD 按钮 + 弹出面板 + 消息引擎 + 5 处跳转 + 移除监察。改动量不小 |
| 消息预生成 | 🔴 | §四设计了一套完整的消息预生成机制——`inbox_state` 存 localStorage、在多个时机触发生成、去重清理。这套机制的复杂度和信箱的收益不成比例 |
| 分阶段合理 | 🟢 | §六的三阶段拆分是合理的，第一阶段只做管理员信箱，范围可控 |

**核心问题**：消息不需要预生成和存储。

当前方案是"消息在操作时生成 → 存到 localStorage → 打开信箱时读取"。但这意味着：
- 每次操作（审核、提交、分配）都要触发消息生成
- 消息可能和实际数据不同步（比如消息生成了但任务被删了）
- 需要额外的去重、清理、过期逻辑

**更简单的做法**：打开信箱时**实时计算**。信箱打开 → 遍历 `data.tasks` 和 `data.finance` → 当场判断哪些是"需要关注的" → 渲染出来。消息不需要存，因为数据源（`data.tasks`）本身就有所有信息。已读状态只需存一个时间戳（"上次打开信箱的时间"），比之前更新的就是"新"的。

**裁决：🟡 勉强通过。** 方向和分阶段是好的，但消息存储方案可以砍掉一半复杂度。

---

### 原则2：复用优先

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 复用现有数据源 | 🟢 | `data.tasks`、`data.finance`、`claimants` 等数据不动，正确 |
| 复用现有渲染 | 🟡 | 面板 UI 是从零写的弹出层，信箱面板本身没有复用现有 tab 框架 |
| 跳转逻辑复用 | 🟡 | 每个目标 tab 都要加 `highlightXxx()` 函数。能否用更简单的方式——比如 `switchTab('ntadmin')` 后让目标 tab 自己处理定位，而不是信箱去控制？ |
| 移除监察 | 🔴 | §五说"移除现有监察"，但监察面板重构方案刚被用户修改过（加入了折叠+红点机制）。两个方案在竞争同一个位置 |

**关键追问**：信箱和监察是什么关系？

当前的 [监察面板_重构方案.md](监察面板_重构方案.md) 设计了一个"折叠+红点+已读追踪"的系统，和信箱的"消息+红点+已读+跳转"非常相似。它们是：
- A) 监察是信箱的简化版（只展示、不跳转）→ 直接用信箱替代监察 ✓
- B) 监察和信箱是两个东西（监察=管理员仪表盘，信箱=全角色通知）→ 两者并存？
- C) 信箱替代监察，但把监察的"全员进度"作为信箱的一个折叠区？

方案 §七提到了"迁移策略"，但没有明确回答：**监察按钮到底删不删？删了之后，原来监察里的"全员进度"在哪里看？**

**裁决：🟡 勉强通过。** 复用数据源是对的，但和监察方案的关系必须说清楚。

---

### 原则3：需求对齐

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 管理员消息 | 🟢 | 待审核、逾期、进度掉队——都有明确场景 |
| 共建者消息 | 🟢 | 所管任务逾期、新分配——合理 |
| 冒险者消息 | 🟢 | 审核结果、新任务、NT到账——是冒险者真实需要知道的 |
| NPC/游客消息 | 🟡 | "欢迎引导"和"升级邀请"——目前 NPC/游客只看到占位页，还没有可互动的界面。这个阶段给他们做消息，消息跳转的目标（南塘广场、升级页面）还不存在 |

**追问**：冒险者的"新任务可领"消息是什么触发条件？方案写的是 `claimants 有空位且未领取`——但这意味着所有有空位的任务都会给所有冒险者发消息？如果公告栏有 12 个可领任务，冒险者会收到 12 条消息。这反而变成了信息轰炸。应该改为只通知"和自己相关的"（比如被导师触发、或者之前完成过同类型任务）。

**裁决：🟡 通过。** 消息类型都有需求来源，但 NPC/游客的时机过早，冒险者"新任务"的触发条件需要限定。

### 原则4：全局最优

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 全角色覆盖 | 🟢 | 五种身份都有消息类型，不只是管理员 |
| 跳转定位 | 🔴 | 5 个 tab 各加一个 `highlightXxx()` 函数——这是散布式的改动。未来加新 tab 也要记得加定位支持。能否用统一的事件机制？比如 `switchTab('ntadmin', {highlight: {person:'陈若雨', task:'用户访谈'}})` → 目标 tab 的渲染函数收到参数后自己处理 |
| 数据一致性 | 🟡 | 消息预生成后如果源数据变了（比如任务被删除），消息会指向不存在的目标。跳转定位使用了"人名+任务名"双重匹配——但如果任务被改名，也会失效 |
| 移除监察按钮 | 🟡 | 删了监察按钮后，builderModeBar 只剩"创世"和"冒险者视角"两个按钮，略显空旷。可以考虑把"冒险者视角"移到别处，或者保留一个精简的监察入口 |

**裁决：🟡 通过。** 全局思路是对的，但跳转定位建议用统一参数传递而非每 tab 单独实现。

---

### 原则5：长期主义

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 消息类型可扩展 | 🟢 | 消息结构有 `type` 字段，新增类型只需加新 case |
| 数据清理 | 🟢 | 7 天自动过期 + 已处理自动消失，设计合理 |
| 跳转脆弱性 | 🟡 | "人名+任务名"双重匹配在任务改名或人名变更时会失效。建议跳转时传 task index（`data.tasks` 中的索引）而非名称——索引在单次会话中稳定 |
| 消息膨胀 | 🟡 | 如果 20 个冒险者每人提交 5 个任务，管理员信箱有 100 条待审核消息。方案没有说分页或聚合。建议同类型消息按人聚合——"陈若雨提交了 3 项任务"而不是 3 条单独消息 |
| 和未来多世界的关系 | 🟡 | Phase 2 多世界架构后，消息需要带 `world_id`。方案没有提这个前提假设 |

**裁决：🟡 通过。** 扩展性设计到位，但跳转方式、消息聚合、多世界预留需要补充。

## 二、总结

| 原则 | 评级 | 关键问题 |
|------|:--:|------|
| 少即是多 | 🟡 | 消息预生成+存储砍掉，改为实时计算 |
| 复用优先 | 🟡 | 和监察方案的关系必须明确 |
| 需求对齐 | 🟡 | NPC/游客时机过早，冒险者"新任务"触发需收窄 |

| 长期主义 | 🟡 | 补充消息聚合 + 多世界预留 + 跳转用索引 |

## 三、修正建议

### 必须修改

1. **消息改为实时计算，不预生成存储。**
   打开信箱时遍历 `data.tasks` + `data.finance`，当场生成消息列表。已读状态只存一个 `lastOpened` 时间戳。砍掉 `inbox_state.messages` 数组、去重逻辑、清理逻辑。

2. **明确信箱和监察的关系。**
   二选一：要么信箱替代监察（监察按钮删掉，进度总览收进信箱的一个折叠区），要么信箱和监察并存（监察保留，信箱做通知跳转）。不能两个方案都留着但关系模糊。

3. **收窄冒险者"新任务"消息的触发范围。**
   不是所有可领任务都通知，只通知"被导师触发的新任务"和"同类型有更新"。

### 建议修改

4. **跳转用 `switchTab(name, params)` 传参数**，各 tab 的渲染函数接收 `params.highlight` 后自己处理定位，而不是写 5 个 `highlightXxx()`。

5. **同类型消息按人聚合**。"陈若雨提交了 3 项任务"是一条消息，不是三条。

6. **补充消息数量上限**。超过 N 条时折叠或分页，避免 100 条消息淹没管理员。

7. **NPC/游客信箱推迟到 Phase 3**。他们的界面（村口）还没建好，消息跳转目标不存在。

---

## 四、最终结论

**方案可以执行，但先把上面三条必须修改的做完了再动工。**

最大的简化点：不要预生成消息存 localStorage。信箱打开时实时查 `data.tasks` 算出来就行——数据源本身已经在 localStorage 里了，不需要再存一份它的"投影"。这个简化会让代码量减少约 30%，维护难度减半，而且天然避免了数据不一致。
