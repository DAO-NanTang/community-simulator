## 二、候选详述

### 候选 1：`saveData()` —— 统一数据持久化 🔴

**现状**
- 保存惯用法 `safeStorage.setItem('camp_data', JSON.stringify(data))` **散落 109 处**。
- `updateAll()`（约 L6143）结尾也保存一次；因此大量"`safeStorage.setItem(...); updateAll();`"两连写会**存两遍**。
- 已有 `safeStorage` 包装（try/catch 防 Edge 拦截），但**没有业务层的保存入口**。

**统一为**
```js
function saveData() {
  safeStorage.setItem('camp_data', JSON.stringify(data));
}
```
109 处直写 → `saveData()`。

**收益**
- 保存有唯一入口：将来加**防抖、脏标记、写失败提示、飞书同步触发、数据版本号**只改一处。
- 消除"忘了存"和"存两遍"。
- 与 `Clock`/`recordTransaction` 同一哲学。

**风险**：极低。纯包装、1:1 机械替换。

**开放问题（待讨论）**
- (a) `saveData()` 是否顺带触发飞书同步（把 `updateAll` 里的 `autoSaveToFeishu` 也收进来）？还是保持最小、只写本地？
- (b) 是否借此**去掉与 `updateAll()` 的重复保存**（即 setItem 紧跟 updateAll 的地方删掉前者）？还是本期只做机械替换、双保存留到下一步？

### 候选 2：`parseMD()` —— 统一日期解析（含修 bug）🟠

**现状（有坑）**
- 四个定义、三个名字，都解析 "M/D"：
  - `parseDL`（定义**两次**：L2717、L2733），被调用 **7 次** → 返回 `Date | null`
  - `parseDate`（L7116），被调用 **11 次** → 返回 `毫秒数(getTime) | Infinity`
  - `parseDateStr`（L7129），被调用 **15 次** → 返回 `毫秒数 | Infinity`
- **年份硬编码不一致**：parseDL 用 **2026**，parseDate/parseDateStr 用 **2025**。同一 "7/10" 走不同函数得到不同年份 → 潜在 bug。
- **返回契约不同**：一个返回 Date/null（用于日期运算），两个返回毫秒/Infinity（用于排序比较）。**不能盲目合并。**

**统一思路**
- 建一个底层 `parseMD(s)`：年份用 `Clock` 推导当年（不再硬编码 2025/2026），返回 `Date | null`。
- 保留两个"契约适配"薄封装：比较排序处要毫秒/Infinity 的，用 `parseMD(s)?.getTime() ?? Infinity` 收口。
- 三处调用点分别切到统一底层。

**收益**：消重复 + **修掉 2025/2026 不一致** + 跨年不再失准。

**风险**：中。必须逐一确认 33 处调用的**返回类型期望**，尤其"Infinity 用于把无效日期排到最后"这类语义不能丢。

**开放问题**
- (a) 年份规则：**选定「当前年」**（用 Clock 推导当年）。不选「就近年」——它引入了隐式逻辑，在年初判断去年 12 月日期时容易出错。当前业务如有跨年场景（如 12 月发布的任务 1 月截止），手动指定年份即可，不交给解析函数隐式处理。
- (b) 是否把返回统一为 `Date|null`，调用处自行 `.getTime()`？还是保留两种适配封装减少改动量？→ **保留两种薄封装**（一个返回 Date|null，一个返回毫秒/Infinity），这样做最少改动，且各自语义清晰。

### 候选 4：`daysBetween(a, b)` / `daysSince(x)` —— 统一天数差 🟡

**现状**
- `86400000`（一天的毫秒数）散落 **15 处**，每次手写计算。
- 分为以下几种模式：

| 模式 | 示例代码 | 出现次数 |
|------|------|:--:|
| 紧急度判断 | `(d-td)/86400000 <= 2` | 5 |
| 注册天数 | `Math.ceil((Date.now() - new Date(u.created)) / 86400000)` | 4 |
| 等待天数 | `Math.floor((today - earliest) / 86400000)` | 2 |
| 剩余天数 | `Math.floor((deadline - today) / 86400000)` | 1 |
| 获取次日 | `new Date(now.getTime() + 86400000)` | 1 |
| 7天截断 | `new Date(Clock.ms() - 7 * 86400000)` | 1 |
| 营地天数 | `Math.round((end - start) / 86400000)` | 1 |

- **注**：5 处紧急度判断中的 `parseMD(t.deadline)` 已在候选 2 中统一。

**统一为**
```js
function daysBetween(a, b) {
  // 统一按 UTC+8 日历日计算天数差；a, b 为 Date 或 "YYYY-MM-DD" 或 "M/D" 字符串
  var toDate = function(x) {
    if (x instanceof Date) return x;
    if (typeof x === 'number') return new Date(x);
    var d = parseMD(x); if (d) return d;
    return new Date(x + 'T00:00:00');
  };
  var da = toDate(a), db = toDate(b);
  if (isNaN(da.getTime()) || isNaN(db.getTime())) return 0;
  var TZ = 8 * 60 * 60000;
  return Math.floor((da.getTime() + TZ) / 86400000) - Math.floor((db.getTime() + TZ) / 86400000);
}
function daysSince(x) { return daysBetween(Clock.today(), x); }
function daysUntil(x) { return daysBetween(x, Clock.today()); }
```

- `daysBetween(a, b)`：a 和 b 之间的日历日差（整数，a 晚于 b 为正）
- `daysSince(x)`：从 x 到今天过了多少天（正数=过去，负数=未来）
- `daysUntil(x)`：从今天到 x 还剩多少天（正数=未来，负数=过期）

**替换对照**

| 现状 | 替换为 | 处数 |
|------|------|:--:|
| `(d-td)/86400000 <= 2 && (d-td)/86400000 >= 0` | `daysUntil(dl) <= 2 && daysUntil(dl) >= 0` | 5 |
| `Math.ceil((Date.now() - new Date(u.created))/86400000)` | `Math.abs(daysSince(u.created))` — 注：`daysSince` 可返回负数 | 4 |
| `Math.floor((new Date(today+'T00:00:00') - new Date(s.earliestDate+'T00:00:00')) / 86400000)` | `daysSince(s.earliestDate)` | 1 |
| `Math.floor((new Date(t.deadline+'T00:00:00') - new Date(today+'T00:00:00')) / 86400000)` | `daysUntil(t.deadline)` | 1 |
| `Math.floor((new Date(today+'T00:00:00') - new Date(t.deadline+'T00:00:00')) / 86400000)` | `daysSince(t.deadline)` | 1 |
| `new Date(now.getTime() + 86400000)` | `new Date(now.getTime() + 86400000)` — **不动**（次日计算，不是天数差） | 1 |
| `new Date(Clock.ms() - 7 * 86400000)` | `new Date(Clock.ms() - 7 * 86400000)` — **不动**（时间截断，不是天数差） | 1 |
| `Math.round((endD - startD) / 86400000)` | `Math.abs(daysBetween(endD, startD))` | 1 |

**实际替换量**：15 处中 **12 处替换**，3 处不动（次日计算/时间截断/营地天数——语义不同）。

**收益**：消除魔术数字 86400000、UTC+8 日历日口径统一、少写易错的毫秒运算、与 `Clock` 配套。

**风险**：低。`daysBetween` 返回整数（floor），而现状有些用 `Math.ceil`。注册天数的 4 处需要加 `Math.abs()` 包裹（因为 `daysSince` 对过去日期返回正数，但语义上注册天数永远是正数）。

### 候选 7（新发现）：时区一致性 —— `today` 变量 → `Clock.today()` 🟡

**现状**
- `var today = new Date().toISOString().slice(0,10)` 这个模式散落 **11 处**（用作日期比较的基准日变量）。
- 另外 ~37 处 `new Date().toISOString().slice(0,10)` 是记录时间戳（`created_at`、`posted_at` 等），**本轮不动**——它们的语义是"记录操作发生的 UTC 时刻"，不是"今天是什么日子"。

**为什么需要改**
- `new Date().toISOString()` 返回 **UTC 时间**。例如北京时间早上 6:00，UTC 是前一天 22:00 → `slice(0,10)` 得到的是**昨天的日期**。
- `Clock.today()` 已经正确实现了 UTC+8 日历日（用 `getUTCDate()` + `TZ_MIN` 偏移），但**目前没有任何调用者**（0 处使用！）。

**替换方式**：严格 1:1，只替换用作"今天"变量的那一行：
```
var today = new Date().toISOString().slice(0,10);
→
var today = Clock.today();
```

**涉及行**（11 处 `today` 变量声明 + 若干独立的 `new Date().toISOString().slice(0,10)` 比较用）：
- `today` 变量声明：共约 11 处，直接替换
- 独立比较用（如 `m.time >= new Date().toISOString().slice(0,10)`）：共约 3 处，替换为 `m.time >= Clock.today()`

**不动**（~37 处）：
- 所有 `created_at`、`posted_at`、`listed_at`、`sold_at`、`changed_at` 等记录时间戳
- 文件下载名中的日期
- 这些是数据记录，UTC 时间是可以接受的

**收益**：修复一个隐性的时区 bug——凌晨 0-7 点用户看到的"今天"日期是昨天。

**风险**：极低。`Clock.today()` 返回 `"YYYY-MM-DD"` 字符串，和 `new Date().toISOString().slice(0,10)` 格式完全一致，下游比较逻辑不变。

### 自审（少即是多 / 复用 / 需求对齐 / 全局 / 长期）

- **少即是多**：候选 4 只做 12 处实际替换（3 处语义不同的不动）；候选 7 只替换 ~14 处 today 变量/比较（37 处时间戳不动）。两个候选都不新增功能，只消除重复和修复隐式 bug。
- **复用优先**：候选 4 复用 `parseMD`（候选 2 成果）和 `Clock`（已有时间源）；候选 7 复用已有的 `Clock.today()`。
- **需求对齐**：候选 4 对应 15 处散落的魔术数字；候选 7 对应 Clock 设计意图（UTC+8 日历日）未被实际使用的问题——Clock 写了但没人用，等于白写。
- **全局最优**：两个候选都建立在 1/2/3 的成果之上（`saveData` → `Clock` → `parseMD` → `daysBetween` → `Clock.today` 收编），形成一条完整的依赖链。每一步都是下一步的基础。
- **长期主义**：`daysBetween`/`daysSince`/`daysUntil` 是时间计算的标准入口，未来任何地方需要算天数差都不再需要手写毫秒数。`Clock.today()` 的全面采用确保所有"今天"判断都是 UTC+8 口径。

**特别注意**：
- 注册天数的替换：现状用 `Math.ceil((Date.now() - new Date(u.created)) / 86400000)` → 改为 `Math.abs(daysSince(u.created))`。`daysSince` 对过去日期返回正数（`Clock.today()` 晚于 `u.created`），`Math.abs` 是保险，实际不需要但无害。
- 第 3064 行的 `new Date(now.getTime() + 86400000)` 是获取"明天"的 ISO 日期用于日期选择器默认值，不是天数差计算，本轮不动。
- 第 4077 行的 `new Date(Clock.ms() - 7 * 86400000)` 是计算 7 天前的时间戳，用于截断通知列表，不是天数差计算，本轮不动。
