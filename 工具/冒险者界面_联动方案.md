## 〇、数据结构设计

### 0.1 任务对象扩展

每条任务在现有字段基础上新增：

```js
{
  // ── 现有字段（不变）──
  name: "M7 闸口②白描审核",
  type: "主线",           // 主线/支线/共创/隐藏
  category: "课程",        // 宣传/课程/生活/财务/管理
  points: 70,              // NT 奖励
  assignee: "",            // 发布者（NT管理员）
  deadline: "6/15",
  status: "active",        // active / closed
  source: "第四期NT任务系统",

  // ── 新增字段 ──
  max_slots: 20,           // 最多多少人领（0=不限）
  poster: "砚仁",          // 谁发布的
  posted_at: "2026-07-10",
  claimants: [             // 领取记录
    {
      name: "陈若雨",
      claimed_at: "2026-07-10",
      status: "completed",    // triggered | in_progress | submitted | completed | rejected | dropped
      submitted_at: "2026-07-12",
      submit_note: "已提交3张临摹作品",
      reviewed_by: "砚仁",
      reviewed_at: "2026-07-13",
      review_note: "通过"
    }
  ]
}
```

**claimants[].status 完整取值表**：

| status | 含义 | 谁设的 | 触发条件 |
|--------|------|--------|---------|
| `triggered` | 导师已触发（仅隐藏任务） | NT管理员 | 导师在管理面板把冒险者加到隐藏任务的 claimants |
| `in_progress` | 进行中 | 冒险者 | 冒险者点「领取」后 |
| `submitted` | 已提交，待审核 | 冒险者 | 冒险者点「提交完成」后 |
| `completed` | 审核通过 | NT管理员 | 管理员点「确认通过」后 |
| `rejected` | 退回修改 | NT管理员 | 管理员点「退回修改」后 |
| `dropped` | 已放弃 | 冒险者 | 冒险者点「放弃委托」后 |

### 0.2 流水记录结构

```js
// 存在 data.finance（NT 流水数组）
{
  type: "task_reward",       // task_reward | plaza_delegation | lodging | transfer
  from: "系统",               // NT 来源
  to: "陈若雨",               // NT 去向
  amount: 70,
  task_name: "M7 闸口②白描审核",
  task_id: 7,                 // data.tasks 中的索引
  date: "2026-07-13",
  confirmed_by: "砚仁",
  note: ""
}
```

### 2.2 交互流程

**点击「⚔️ 领取委托」**：
```
① 检查是否已登录（currentUser）
② 检查是否已领取（claimants 里有没有同名，含 triggered/dropped）
③ 检查是否已满（max_slots > 0 && claimants.filter(c => c.status !== 'dropped').length >= max_slots）
④ 检查任务状态是否 active
⑤ 全部通过 → claimants.push({ name, claimed_at: now, status: "in_progress" })
⑥ 保存 → 刷新公告栏 + 我的资料
```

**点击「提交完成」**：
```
① 弹出输入框：「描述你完成的内容」（可选）
② claimants 中该用户的记录 status → "submitted"
③ submit_note 写入
④ 保存 → 刷新
⑤ NT管理员那边出现待审核项
```

### 2.3 筛选逻辑

| 筛选项 | 逻辑 |
|--------|------|
| 全部 | 不筛选 |
| 主线 | `task.type === '主线'` |
| 支线 | `task.type === '支线'` |
| 共创 | `task.type === '共创'` |
| 隐藏 | `task.type === '隐藏'` + 当前用户在 claimants 中（任意 status） |
| 可领取 | `claimants` 中没有当前用户 且 未满 |
| 已领取 | `claimants` 中有当前用户（status 不为 dropped） |

### 2.4 隐藏任务触发机制

**流程**：

```
NT管理员在管理面板中，对某个隐藏任务点「🔓 触发」
    │
    ├─ 选择目标冒险者（下拉列表或输入姓名）
    │
    ├─ claimants.push({
    │     name: "陈若雨",
    │     claimed_at: now,
    │     status: "triggered"
    │   })
    │
    └─ 保存 → 冒险者的公告栏「隐藏」筛选中出现该任务
         → 冒险者可以点「⚔️ 领取委托」
         → 领取后 status 从 triggered → in_progress
```

**筛选逻辑**：隐藏任务只在公告栏显示给 claimants 中存在的冒险者（status 为任意值：triggered / in_progress / submitted / completed / rejected）。别人看不到。

**触发入口**：NT 管理员面板 → 已发布任务列表 → 隐藏任务行末的「🔓 触发」按钮。

### 2.5 放弃委托

**场景**：冒险者领了任务后做不完，想放弃。

**流程**：
```
冒险者点「🏳️ 放弃委托」
    │
    ├─ confirm('确定放弃该委托？')
    │
    ├─ claimants 中该用户的记录 status → "dropped"
    │
    └─ 保存 → 刷新
         → 任务从「我的资料」进行中移除，进入「已放弃」分类
         → 公告栏「已领取」中不再显示该任务
         → 名额释放：该用户不再占用 max_slots 计数
```

**名额计算**：`claimants.filter(c => c.status !== 'dropped').length`，不把 dropped 计入已领人数。

**UI 按钮**：在「我的资料」和「公告栏」的「已领取」任务卡片上，与「提交完成」并列显示「🏳️ 放弃」。

### 3.3 审核→转账流

```
管理员点「✅ 确认通过」
    │
    ├─ ① 更新 claimants[].status = "completed"
    ├─ ② claimants[].reviewed_by = currentUser.name
    ├─ ③ claimants[].reviewed_at = 当前时间
    │
    ├─ ④ 写入流水：
    │     data.finance.push({
    │       type: "task_reward",
    │       from: "系统",
    │       to: 冒险者名字,
    │       amount: task.points,
    │       task_name: task.name,
    │       date: 当前日期,
    │       confirmed_by: currentUser.name
    │     })
    │
    └─ ⑤ 保存 → 刷新面板
         → 冒险者的「我的资料」里这条变成已完成
         → 冒险者的 NT 余额增加
```

**退回修改**：
```
管理员点「❌ 退回修改」
    │
    ├─ ① claimants[].status = "rejected"
    ├─ ② claimants[].review_note = 退回原因
    └─ ③ 冒险者看到状态变回「已退回」，可以修改后重新提交
         （重新提交时 status 从 rejected → submitted）
```

### 3.4 NT 池保护

确认前检查：
```js
var totalIssued = data.finance
  .filter(function(tx) { return tx.type === 'task_reward'; })
  .reduce(function(sum, tx) { return sum + tx.amount; }, 0);

var poolTotal = data.budget.nt_total_pool || 30000;

if (totalIssued + task.points > poolTotal) {
  return alert('⚠️ NT 池余额不足！剩余 ' + (poolTotal - totalIssued) + ' NT');
}
```

> ⚠️ **单世界假设**：当前 `data.finance` 混着所有期的数据，NT 池统计未按世界 ID 过滤。在单世界场景下这是正确的；若未来上线多世界，需在遍历流水时加上 `world_id` 过滤条件。

### 3.5 面板挂载位置

> **决策：NT 管理员面板作为共建者 tab 栏中的一个独立 tab。**

当前共建者 tab（dashboard.html:494-501）：
```
🌍 创世终端 / 👥 冒险者档案 / 📋 猎人公会 / 📅 世界时间线 / 💰 资源金库 / 🧾 通关结算
```

改造后的共建者 tab：
```
🌍 创世终端 / 👥 成员档案 / 📋 猎人公会 / 🎯 NT管理 / 📅 世界时间线 / 💰 资源金库 / 🧾 通关结算
```

- tab 名：`ntadmin`
- tab 可见条件：`currentUser.role === 'builder'` 且当前用户在 Step3 被分配了 NT 角色
- 渲染函数：`renderNTAdmin()`
- `switchTab('ntadmin')` → 调用 `renderNTAdmin()`

**为什么不挂在猎人公会的子视图？**NT 管理面板和猎人公会（任务编排）是两种不同的工作——NT 管理面向冒险者的任务板，猎人公会面向共建者的协同工作。独立 tab 更清晰。

## 七、称号/勋章自动计算

```js
function getAchievements(userName) {
  var tasks = data.tasks || [];
  var achievements = [];

  // 统计该用户完成的任务
  var completed = tasks.filter(function(t) {
    var c = (t.claimants || []).find(function(cl) { return cl.name === userName; });
    return c && c.status === 'completed';
  });

  var allMain = tasks.filter(function(t) { return t.type === '主线'; });
  var doneMain = completed.filter(function(t) { return t.type === '主线'; });

  // 主线通关
  if (allMain.length > 0 && doneMain.length === allMain.length) {
    achievements.push({ name: '主线通关', icon: '🏅', desc: '完成全部主线任务' });
  }

  // 全勤猎人
  if (completed.length >= 15) {
    achievements.push({ name: '全勤猎人', icon: '🎖️', desc: '完成 15 项以上任务' });
  }

  // 支线达人
  var doneSide = completed.filter(function(t) { return t.type === '支线'; }).length;
  if (doneSide >= 6) {
    achievements.push({ name: '支线达人', icon: '⭐', desc: '完成 6 项以上支线任务' });
  }

  // 可以在 data.members[name].achievements 里存手动授予的称号
  var manual = (data.members && data.members[name] && data.members[name].achievements) || [];

  return achievements.concat(manual);
}
```
