## 二、逐原则审查

### 原则1：少即是多

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 改动范围 | 🟡 | 声称"只改 tab-members"，但 `renderMembers()` 是全剧核心函数，改了它会影响任务大厅（`renderDailyView` 也依赖 `getStaffNames()` 从 `data.members` 取人） |
| 二级导航 | 🔴 | 新增了"游客"类型，但在 [身份系统梳理.md](身份系统梳理.md) 中只有三种身份：NPC、冒险者、共建者。**"游客"没有需求来源**，属于多余的顺手添加 |
| 密码管理 | 🟡 | 密码修改功能在 memberModal 里增加是合理的，但 `nt_users` 系统已有注册/登录流程，密码修改应复用它已有的 `encodePassword()`，方案确实这样做了 |

**裁决：** 删除"游客"类型。目前只有 NPC / 冒险者 / 共建者 三种。

---

### 原则2：复用优先

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 复用 memberModal | 🟢 | 弹窗HTML已存在（第1085-1108行），方案正确复用它 |
| 复用 getPersonMaterials() | 🟢 | 材料表函数已存在，共建者详情可直接调用 |
| 复用 `encodePassword()` | 🟢 | 已在第1178行定义 |
| 复用 `getUsers()` / `saveUsers()` | 🟢 | 已在第1179-1180行定义 |
| 二级导航按钮样式 | 🟢 | 用的是已有的 `.btn` / `.btn.pri` 类 |
| `renderMembers()` 改造 | 🔴 | 方案说"从 `nt_users` 获取所有该类型用户"。但当前 `renderMembers()` 构建的 `staffData` 是 `{name: {tasks, points, done, ...}}` 结构，如果从 `nt_users` 拿人但没有对应任务，那 `m.tasks.length` 就是 0，`renderMemberDetail()` 仍然能正常显示"暂无任务"。**但** `nt_users` 里的 role 字段用的是 `'builder'` / `'adventurer'` / `'npc'`，而 `data.members[name].role` 可能是中文角色名（如"统筹"）。**两套 role 体系不一致，会产生混淆。** |

**裁决：** 第1.5节需要明确：`nt_users` 的 `role` 字段（`'builder'/'adventurer'/'npc'`）用于身份筛选，`data.members` 的 `role` 字段（如 `'统筹'`）用于显示。两者不冲突，但需要在代码中分开处理。

### 原则3：需求对齐

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 修 `showEditMember()` | 🟢 | 有3处调用无定义，明确bug |
| 二级导航筛选 | 🟡 | 身份系统已明确有三种身份，筛选功能合理。但"游客"无需求来源 |
| 共建者详情加任务+材料 | 🟢 | 用户明确要求共建者看到自己的任务和材料表 |
| 密码修改 | 🟡 | 用户提到"密码1500"，有登录需求。但成员档案弹窗里改密码是否是正确的位置？这应该是用户自己的"设置"功能，而非管理员在编辑成员时改别人的密码 |

**裁决：** 删除"游客"；密码修改功能保留但需确认：管理员应该有权限改成员密码吗？如果是，保留；如果不是，挪到用户自己的设置界面。

---

### 原则4：全局最优

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 改名影响 | 🔴 | `saveMember()` 改名逻辑（第73-76行）：`data.members[newName] = data.members[oldName]; delete data.members[oldName]` — **但没有更新 `data.tasks` 中的 `assignee` 字段！** 如果某个任务 assignee 是旧名字，改名后任务就找不到人了。`data.finance_cny.transactions` 中的 `person` 字段也一样。 |
| `renderMembers()` 数据源切换 | 🔴 | 方案说从 `nt_users` 取人。但当前 `staffData` 完全从 `data.tasks` 构建（第1544-1556行）。如果一个人注册了但在 `data.tasks` 中没有任务，他仍然应该出现在成员档案中。方案是对的，但需确保：① 头像列表从 `nt_users` 取，② 详情中的任务统计仍从 `data.tasks` 取。**两者共存，不能互相替代。** |
| 全局变量 `memberFilter` | 🟡 | 使用全局变量控制筛选状态，没有持久化。如果用户刷新页面，筛选状态丢失回到默认值。不过这是可接受的行为。 |
| 与 `getStaffNames()` 的关系 | 🟡 | 很多地方调用 `getStaffNames()` 来判断一个人是工作人员还是玩家。如果 `nt_users` 中的 role 和 `data.members` 中的 role 不同步，会导致判断不一致。 |

**裁决：** `saveMember()` 改名时必须同步更新 `data.tasks[].assignee` 和 `data.finance_cny.transactions[].person`。`renderMembers()` 需要保持双数据源：头像列表从 `nt_users` 取，任务统计从 `data.tasks` 取。

---

### 原则5：长期主义

| 检查点 | 评级 | 说明 |
|--------|------|------|
| `nt_users` 与 `data.members` 同步 | 🔴 | 方案没有系统性处理两个数据源的同步问题。如果 `nt_users` 有一个人但 `data.members` 没有，或者反过来，该怎么处理？方案提到了"从 `nt_users` 取人 + 从 `data.members` 补充"，但从 `nt_users` 取来的人在 `data.members` 中可能根本不存在。 |
| 数据迁移 | 🔴 | 现有 `data.members` 中的成员没有 `nt_users` 中的 `password` 字段。方案没有提供迁移路径。用户已有的 localStorage 数据打开后会怎样？ |
| 硬编码的角色名 | 🟡 | `memberFilter` 用字符串 `'builder'/'adventurer'/'npc'` 硬编码匹配。如果未来加新身份，需要改多处。建议定义常量。 |
| 成员 Modal 缺少 `editMemberOldName` 的重置 | 🟡 | 每次打开 modal 需要正确设置 `editMemberOldName`。方案在 `showEditMember()` 中做了，但关闭时没有清理。如果用户改名后不保存，下次打开可能是脏数据。不过这是小问题。 |

**裁决：** 需要补充数据迁移/初始化逻辑：首次使用时，对现有 `data.members` 中没有 `nt_users` 条目的成员自动生成基础用户记录。

方案只提了 `showEditMember()` 不存在，但以下函数也被调用却不存在：

| 缺失函数 | 调用位置 |
|----------|---------|
| `saveMember()` | memberModal 第1105行 `<button onclick="saveMember()">` |
| `deleteMember()` | memberModal 第1104行 `<button onclick="deleteMember()">` |

方案中只写了 `saveMember()` 的改造，但没有写 `deleteMember()`。

## 四、总结

| 原则 | 评级 | 关键问题 |
|------|------|---------|
| 少即是多 | 🟡 | "游客"类型无需求来源，应删除 |
| 复用优先 | 🟡 | `nt_users.role` 和 `data.members.role` 是两套体系，需明确区分 |
| 需求对齐 | 🟡 | 密码修改位置需确认 |
| 全局最优 | 🔴 | 改名时不同步 `data.tasks.assignee` — 会产生僵尸任务 |
| 长期主义 | 🔴 | `nt_users` 和 `data.members` 的数据同步没有系统性方案 |

## 五、修正建议

### 必须修改（阻断级）

1. **改名同步更新任务和流水** — `saveMember()` 改名时遍历 `data.tasks` 和 `data.finance_cny.transactions` 更新 `assignee` / `person` 字段
2. **补充 `deleteMember()` 函数** — 方案遗漏了这个缺失函数
3. **删除"游客"类型** — 当前需求中不存在
4. **补充数据初始化逻辑** — 现有 `data.members` 中没有 `nt_users` 条目的成员，需要自动生成基础用户记录（无密码，标记为待注册）

### 建议修改（重要级）

5. **明确 `nt_users.role` 和 `data.members.role` 的分工** — 前者管身份筛选（builder/adventurer/npc），后者管角色显示（统筹/宣传/NT）
6. **`renderMembers()` 双数据源逻辑** — 伪代码写清楚：头像列表来源 → 任务统计来源 → 合并规则
