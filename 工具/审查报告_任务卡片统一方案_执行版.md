#### ⚠️ 改动 1.3 — showStatusEdit 存在两个问题

**问题 1.3a — 状态更新函数签名不一致**

执行版第 66 行：
```js
h += '<select onchange="data.tasks['+idx+'].status=this.value;localStorage.setItem(...);'+(_refreshFn[ctx]||'')+'" ...>';
```

这直接操作 `data.tasks[idx].status`，与现有代码 [dashboard.html:3988-3992](dashboard.html#L3988-L3992) 使用 `updateTaskStatus(memName, taskName, status)` 函数的方式不一致。现有函数在修改状态后还会调用 `updateAll()` 重新计算统计：

```js
function updateTaskStatus(memName, taskName, status) {
  var t = data.tasks.find(function(x) { return x.name === taskName; });
  if (t) t.status = status;
  updateAll();
}
```

执行版的 onchange 只调用了 `localStorage.setItem` + 重渲染函数，没有调用 `updateAll()`。这意味着：
- 成员统计数据不会更新
- NT 余额不会重算
- 进度条不会刷新

**建议**：onchange 中改为调用 `updateTaskStatus` 或在重渲染后追加 `updateAll()` 调用（取决于 `_refreshFn` 触发的渲染函数是否已经包含 `updateAll()`）。

**严重程度**：🟠 中（会导致统计不同步）

var statuses = ['待开始','进行中','已完成'];

但 `data.tasks` 中实际可能出现的状态值还包括 `'已确认'`、`'active'`、`'待确认'`（见 [dashboard.html:4027](dashboard.html#L4027) 任务编辑表单）。当前成员档案的下拉框使用 `'待开始'`/`'进行中'`/`'已完成'`（[dashboard.html:3979](dashboard.html#L3979)），这三个值对于成员档案场景是正确的。但如果未来 `showStatusEdit` 被复用到其他场景（如工会大厅），状态值可能不够用。

#### ❌ 改动 2.3 — 来源过滤逻辑不完整且引用了不存在的变量（阻塞级）

**位置**：执行版第 180-185 行

```js
if (wsSourceFilter === 'work') {
  list = myCampTasks.filter(function(e){ return !e.claim || e.claim.status===''; })
    .concat(myCampTasks.filter(function(e){ return e.task.assignee === currentUser.name; }));
}
```

**三个问题**：

1. **`myCampTasks` 不存在**：同上，这个变量在代码中未定义。实际代码中，`allMyTasks` 是一个包含所有 `{task, idx, claim, scope}` 对象的数组，按 scope 区分营地（`scope==='camp'`）和个人（`scope==='personal'`）。

2. **只处理了 `'work'` 一种标签**：`'quest'` 和 `'personal'` 的过滤逻辑完全缺失。如果用户点击 `📋 大厅委托` 或 `✏️ 私人委托`，`wsSourceFilter` 不等于 `'work'`，整个 if 块跳过，`list` 保持原样——这意味着三个标签没有实际过滤效果。

3. **过滤逻辑的业务语义有歧义**：
   - `!e.claim || e.claim.status===''` — 这个条件试图匹配"有 assignee 但没有 claimant"的任务。但在 `allMyTasks` 的构建逻辑中，如果任务是 assignee 匹配且没有 claimant，会创建一个 `{status:'in_progress'}` 的合成 claim（[dashboard.html:2541](dashboard.html#L2541)）。所以 `e.claim` 永远不为空。
   - 区分"创世工作"和"大厅委托"的核心依据应该是**任务是否来自 claimant 主动领取**，还是**任务是否仅通过 assignee 分配**。当前数据模型中没有明确的"来源"字段来区分这两种情况，需要更清晰的定义。

**建议**：

先定义三个来源标签的精确过滤语义：

| 标签 | 数据含义 | 过滤条件 |
|------|---------|---------|
| 🏛️ 创世工作 | 创世终端直接分配给我的 | `e.task.assignee === name` 且 `e.claim` 是合成创建的（即原 claimant 中无此人） |
| 📋 大厅委托 | 我从大厅主动领取的 | `e.claim` 是从 `t.claimants` 中匹配到的真实记录 |
| ✏️ 私人委托 | 私人发布的任务 | `e.scope === 'personal'` |

然后在 `allMyTasks` 构建时打标记录来源，而非事后推断：

```js
allMyTasks.push({ task: t, idx: i, claim: myClaim, scope: scope, source: sourceType });
```

其中 `sourceType` 在构建时就能确定：是 assignee 直接分配（`'work'`），还是 claimants 匹配（`'quest'`），还是 personal scope（`'personal'`）。

**严重程度**：🔴 高（按此执行会导致三标签过滤功能完全失效）

#### ⚠️ 改动 3.1 — taskIndex 的防御价值被 self-nullify

**位置**：执行版第 212 行

var idx = data.tasks.indexOf(t);

### Step 4：工会大厅卡片统一

#### ❌ 改动 4.1 + 4.2 — 事件冒泡冲突（阻塞级）

**位置**：执行版第 243-265 行

```js
// 改动 4.1：cardHTML 来自 renderQuestCard（卡片 header 有 onclick 展开/折叠）
var cardHTML = renderQuestCard(t, {...});

// 改动 4.2：外层 div 包裹 dailyCheckTask
return '<div onclick="dailyCheckTask('+idx+',\''+role+'\')" style="cursor:pointer">'+avatarHTML+cardHTML+matHTML+'</div>';
```

**问题**：
- `renderQuestCard` 渲染的卡片 header 有自己的 `onclick`（[dashboard.html:2755](dashboard.html#L2755)）：切换展开/折叠 + 调用重渲染函数
- 外层 div 有自己的 `onclick`：`dailyCheckTask(idx, role)`
- 点击卡片 header 展开/折叠时，事件会**冒泡**到外层 div，同时触发 `dailyCheckTask`
- `dailyCheckTask` 会修改任务状态并调用 `renderDailyView(role)` 重渲染整个视图

**后果**：用户在工会大厅点击卡片展开/折叠时，任务状态会被意外切换（待开始→进行中→已完成 循环），且整个工会大厅会重渲染而非仅刷新卡片。这是明显的功能性 bug。

**修复方案**：在 `renderQuestCard` 的 header onclick 中增加 `event.stopPropagation()`：

```js
// 在 renderQuestCard 第 2755 行，onclick 开头插入：
onclick="event.stopPropagation();window._cardExpanded[...]=...;..."
```

或者，将 `dailyCheckTask` 的绑定改为仅在卡片外部区域触发（非 header 区域），但这更复杂且不可靠。

**推荐**：在 Step 1 中就修改 `renderQuestCard` 的 header onclick，增加 `event.stopPropagation()`。这是一个稳健的改动，不会影响 village/questboard/workspace 场景（这些场景外层没有额外的 onclick）。

**严重程度**：🔴 高（导致工会大厅中展开/折叠卡片会意外改变任务状态）

## 三、额外发现：改动 1.3 的重渲染调用链分析

改动 1.3 的 status onchange 中 `'+(_refreshFn[ctx]||'')+'` 会触发重渲染。对于各 context：

| Context | _refreshFn 值 | 调用的函数 | 是否包含 updateAll | 
|---------|--------------|-----------|-------------------|
| village | `renderVillageQuestHall()` | 直接 | ❌ 不包含 |
| questboard | `renderQuestBoard()` | 直接 | ❌ 不包含 |
| workspace | `renderWorkspace()` | 直接 | ❌ 不包含 |
| member | `renderMemberDetail('name')` | 间接调用 `renderMemberDetailFor` | ❌ 不包含 |
| taskhall | `renderDailyView('staff')` | 直接 → 调用 `updateStats()` | ⚠️ 调用了 updateStats 但不是 updateAll |

而 `updateTaskStatus`（[dashboard.html:3987-3992](dashboard.html#L3987-L3992)）调用的是 `updateAll()`——这是更全面的更新（包含 NT 余额重算、统计刷新等）。直接在 onchange 中操作 `data.tasks[idx].status` + 重渲染，不会触发 `updateAll()`，这意味着：

1. 成员统计数据（`staffData`/`playerData`）不会自动刷新——下次手动调用 `renderMembers()`/`renderPlayers()` 时才会更新
2. NT 余额不会重算
3. 如果状态改为"已完成"，NT 发放逻辑不会被触发

**建议**：改动 1.3 的 onchange 中增加 `updateAll()` 调用，或者改为调用现有的 `updateTaskStatus` 函数（需要 taskName 而非 idx）。

**严重程度**：🟠 中（会导致统计不同步，但不会丢失数据——刷新页面或切换标签后会恢复正确状态）
