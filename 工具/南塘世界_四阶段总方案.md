## 一、Phase 1：统一身份 · 单世界贯通 ✅ 已完成（2026-07-07）

### 1.0 目标

五种身份（管理员/共建者/冒险者/南塘NPC/游客）都能正常注册/登录，系统自动路由到正确界面。在**单世界**（第四期）下，管理员能管理、共建者能协作、冒险者能做任务、NPC/游客能在村口占位页停留。

### 1.1 核心原则

- **单文件不改结构**：仍然是 `dashboard.html` 一个文件，`localStorage` 做数据存储
- **不动数据结构**：`data.tasks` / `data.members` / `data.finance` 等字段保持和现有 `camp_data` 一致
- **管好自己的边界**：代码只管单世界场景，不为多世界预留接口（那是 Phase 2 的事）

#### ⚠️ 安全边界说明

本阶段的"认证"本质是**本地分身份使用**，不是真正的安全认证。密码存储用的是 `encodePassword`（可逆编码，非加密哈希），登录态 `nt_session` 存的是明文 `{ name, role }`（无 token、无过期时间）。它解决的是"多人共用一台电脑时互相不误看对方的界面"，而不是"防止恶意访问"。真正的访问控制需要后端服务器。在此之前，不要用本系统存储真正的敏感数据。

## 二、Phase 2：多世界架构

### 2.0 目标

数据从「一个全局 `data`」变为「每个世界独立存储」。管理员能创建新世界（如第五期），所有人在顶部看到世界切换栏，切换世界后数据完全隔离。

### 2.1 核心改动

#### 数据层

```
现在：                           Phase 2 后：
localStorage                     localStorage
  camp_data (一个对象)     →       nt_world_1100 (第四期)
                                   nt_world_5700 (第五期)
                                   nt_world_plaza (村口) [Phase 3]
                                   nt_worlds_index ["plaza","1100","5700"]
```

#### 代码层

| 改动点 | 说明 |
|--------|------|
| 全局 `data` → `worldData` | 约 100+ 处读写需要改为引用当前世界的 `worldData` |
| `localStorage.setItem('camp_data', ...)` → `saveWorld(id)` | 约 25+ 处持久化调用 |
| 新增 `loadWorld(id)` | 从 localStorage 加载指定世界到 `worldData` |
| 新增 `switchWorld(id)` | 保存当前 → 加载目标 → 刷新全部 UI |
| 新增 `createWorld(name, code)` | 以当前世界为模板创建新世界 |
| `updateAll()` | 改为只更新当前世界 |
| `loadData()` / `exportData()` | 支持「导入到哪个世界」/「导出哪个世界」 |
| Wizard 五步向导 | 创建的内容写到当前世界 |
| 顶部世界切换栏 UI | 新 HTML 元素 |

**世界切换栏**（顶部，管理员/共建者/冒险者可见）：

```
┌─ 🌍 南塘世界 ──────────────────────────────────────────┐
```

- 共建者/管理员：看到全部世界 + 创建按钮
- 冒险者：看到村口 + 自己所属的世界
- NPC/游客：不显示切换栏（只有村口）

### 2.4 数据迁移

首次加载 Phase 2 时：
1. 检测 `camp_data` 存在 + `nt_worlds_index` 不存在 → 触发迁移
2. `camp_data` → `nt_world_1100`
3. 创建 `nt_worlds_index = ["plaza", "1100"]`
4. 创建空 `nt_world_plaza`（为 Phase 3 预留）
5. 保留旧的 `camp_data` 不删除（安全备份）

**幂等保护**：迁移脚本检查目标 key（`nt_world_1100`）是否已存在，已存在则跳过整个迁移流程，不覆盖已有数据。即使用户打开了多个标签页，也只会迁移一次。

## 三、Phase 3：南塘村口

### 3.0 目标

把目前 NPC/游客看到的「占位页」替换为真正的「南塘村口」——一张实景地图 + 日常委托 + P2P 流水 + 住宿登记。村口是所有人的第一站和碰头地。

### 3.1 村口是什么

村口不是一个 tab，是一个**独立空间**（和第四期、第五期平级）。数据存储在 `nt_world_plaza`。

所有人——不管什么身份——都能进村口。但不同身份看到的内容不同：
- 管理员：看到全部 + 管理面板
- 共建者/NPC：看到全部 + 可发委托
- 冒险者：看到委托板 + 可接单
- 游客：看到委托板（只读）+ 排行榜

### 3.2 村口包含的功能

| 功能 | 说明 | 复杂度 |
|------|------|:--:|
| 实景地图 | 南塘底图 + 地点标注 + 点击弹出面板 | 中 |
| 日常委托板 | 打扫/帮厨/浇花等，P2P 委托 | 中 |
| P2P 流水 | 委托方→接单方 NT 划转 | 中 |
| 住宿登记 | 入住/离开/日费率/待付跟踪 | 中 |
| 村口 NT 榜 | 统计村口内委托流水的 NT 排名 | 小 |
| 公告编辑 | 管理员可编辑顶部公告 | 小 |
| 村口管理面板 | 确认交付/登记住宿/转让管理权 | 大 |

> **关于实景地图**：底图将另做一个静态网页来生成，届时替换。Phase 3 先用文字 + emoji 坐标做一个简易版地图，预留一个地图组件接口（`renderPlazaMap()`），后续替换底图时只需改这一个函数，不动其他代码。

### 3.3 数据结构（存于 `nt_world_plaza`）

```js
{
  meta: { id: "plaza", name: "南塘村口", type: "plaza" },
  tasks: [],                    // 日常委托
  finance_plaza: {              // P2P 流水
    transactions: [
      { from, to, amount, reason, status, date, confirmed_by }
    ]
  },
  stay_records: [               // 住宿
    { person, check_in, check_out, days, daily_rate, total, paid }
  ],
  plaza_admin: "砚仁",          // 村口管理员
  bulletin: "欢迎来到南塘...",   // 公告
  plaza_map: {                  // 实景地图（Phase 3 先用文字坐标，后续替换底图）
    image: "",                  // 底图 URL/base64，目前留空，由独立静态网页生成后填入
    spots: [{ id, name, x, y, description }]
  }
}
```

### 3.4 交互流程

#### 日常委托 P2P 流程

```
① 共建者发布「打扫画室」NT 50
② 王五点「🙋 接单」→ status = accepted
③ 王五完成后点「✅ 完成」→ status = completed
④ 管理员点「确认交付」→ status = confirmed → NT 从共建者划给王五
⑤ 双方流水各记一笔
```

> **注意**：以上为 happy path。完整状态机还需定义以下分支，待 Phase 3 详细设计时补全：
> - **委托方余额不足**：接单人已完成委托但委托方 NT 不够 → 系统如何处理？
> - **接单人放弃**：accepted 状态的委托需要 `abandon` 动作回到待接状态
> - **管理员退回**：confirmed 前管理员可退回 → 需定义 `rejected` 状态
>
> 建议的完整状态：`pending → accepted → completed → confirmed`（正常）或 `pending → accepted → abandoned`（放弃）或 `completed → rejected`（退回）。

#### 住宿登记

```
① 管理员登记：王五 7/3 → 7/6，3 天，NT 70/天，共计 NT 210
② 王五退房时交给管理员 NT 210
③ 管理员点「标记已付」→ paid = true
```

## 六、第一阶段实施提示

当前 Phase 1 的 `dashboard.html` 已经是一个 **6,200+ 行的单文件应用**。接下来的实施原则：

1. **每改一处跑一遍**：改完一个函数 → 刷新浏览器 → 确认不报错
2. **用 git 做 checkpoint**：每完成一个小功能 commit 一次
3. **不碰数据结构**：`data.tasks` / `data.members` / `data.finance` 等字段不动
4. **先在共建者/管理员视角测，再切冒险者视角测**：冒险者视角是预览模式，和真实登录不完全一样
5. **CSS 优先于 JS**：控制可见性用 CSS class（如 `body.builder-mode`），减少 JS 里写 `element.style.display`
