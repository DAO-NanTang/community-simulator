## 〇、关键发现：被删除的代码

在审查前，发现 `dashboard_backup_20260704_031135.html` 中 **`showEditMember()`、`saveMember()`、`deleteMember()` 三个函数曾经完整存在**（第1704-1759行），但在当前 `dashboard.html` 中被删除了。

| 备份已有的逻辑 | 方案 | 结论 |
|-------------|------|------|
| `saveMember()` 改名时更新 `data.tasks[].assignee` 和 `.confirmer` | ❌ 未包含 | **回归！** |
| `deleteMember()` 完整实现（清任务分配、删 finance by_person） | ❌ 未提及 | **遗漏！** |

### 原则4：全局最优

这是问题最严重的一环：

| 检查点 | 评级 | 说明 |
|--------|------|------|
| 改名同步 `data.tasks` | 🔴 **阻断** | 备份版本的 `saveMember()` 第1728行有 `data.tasks.forEach(t => { if (t.assignee === oldName) t.assignee = name; if (t.confirmer === oldName) t.confirmer = name; })`，**方案代码没有这段逻辑**。改名后旧名字的任务将变成僵尸任务，找不到负责人。 |
| 改名同步 `data.finance_cny` | 🔴 **阻断** | `data.finance_cny.transactions[].person` 和 `data.finance_cny.by_person[name]` 都不会被更新。改名后财务流水找不到人。 |
| `nt_users.role` vs `data.members.role` | 🟡 | 两套 role 体系：`nt_users.role`（`'builder'/'adventurer'/'npc'/'visitor'`）用于身份筛选；`data.members.role`（`'统筹'/'宣传'` 等中文角色名）用于显示。方案未明确区分，容易混淆。 |
| `memberFilter` 全局变量 | 🟡 | 不持久化，刷新后回默认值。可接受但不优雅。 |

**裁决：🔴 不通过。** 改名逻辑是**回归性缺陷**——备份版本有，方案反而丢了。

### 原则5：长期主义

| 检查点 | 评级 | 说明 |
|--------|------|------|
| `deleteMember()` 缺席 | 🔴 | 不实现的话，删除按钮永远是死按钮——这是明显的半成品 |

| 硬编码角色名 | 🟡 | `['builder','adventurer','npc','visitor']` 硬编码在 `filterMembers()` 中，与其他地方的角色定义重复。不影响功能，但新增身份时需改多处 |

**裁决：🔴 不通过。** 缺少 `deleteMember()`、数据同步策略不清晰。

## 三、修正要求

### 必须修改（阻断级）

1. **基于备份版本恢复，而非重写。**
   [dashboard_backup_20260704_031135.html](dashboard_backup_20260704_031135.html) 第1704-1759行有完整的三个函数。在备份版本基础上增加：
   - `nt_users` 同步（备份版缺失）
   - 密码修改（备份版缺失）
   - `data.finance_cny.transactions[].person` 改名同步（备份版也缺失）

2. **恢复 `saveMember()` 中的改名逻辑。**
   必须包含（等价于备份版第1728行）：
   ```js
   if (oldName && oldName !== newName) {
     data.tasks.forEach(t => {
       if (t.assignee === oldName) t.assignee = newName;
       if (t.confirmer === oldName) t.confirmer = newName;
     });
   }
   ```

3. **补充 `deleteMember()` 函数。**
   备份版第1747-1759行已有完整实现，直接恢复。需额外增加：
   - 同步删除 `nt_users` 中的记录
   - 同步清理 `data.finance_cny.transactions` 中的 `person` 引用

4. **补充改名时同步财务记录。**
   遍历 `data.finance_cny.transactions` 更新 `person` 字段，以及重命名 `data.finance_cny.by_person` 的 key。

### 建议修改（重要级）

5. **明确 `nt_users.role` 和 `data.members.role` 的分工。**
   在方案文档中写清楚：
   - `nt_users.role`（`'builder'/'adventurer'/'npc'/'visitor'`）→ 用于身份筛选和登录权限
   - `data.members.role`（`'统筹'/'宣传'` 等中文角色名）→ 用于显示

6. **补充 `renderMembers()` 数据合并伪代码。**
   明确三种情况的处理：
   - 人在 `nt_users` 且在 `data.members` → 合并显示，`data.members` 的字段优先
   - 人在 `nt_users` 但不在 `data.members` → 用默认值创建 member 基础信息
   - 人在 `data.members` 但不在 `nt_users` → 用 `migrateExistingMembers()` 已有的逻辑自动补充

7. **将方案中 `saveMember()` 的描述从"改造"改为"恢复并增强"。**
   准确反映当前代码中该函数不存在的实际情况。
