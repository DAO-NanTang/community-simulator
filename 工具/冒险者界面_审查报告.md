## 总体判断

**这份方案和前面几份不一样——它是能拿来施工的。** 数据结构设计得干净（扩展现有字段，不改现有结构），交互流程每一步都写了判断逻辑（§2.2 领取五步校验、§3.3 审核→转账六步流程），验收走查（§8）从头到尾走通了完整链路。

但它有三类缺口：**跟现有代码的接头处没交代**、**两个交互功能缺了定义**、**一个面板没说明挂哪**。下面逐条说。

---

## 🟢 做得好的部分

### 数据设计——扩展而非替换

方案在现有 `data.tasks` 上追加字段（`max_slots`、`poster`、`posted_at`、`claimants[]`），不改已有字段。JS 对象天然支持任意属性，不会破坏现有代码。`claimants[]` 的设计尤其好——每个领取者的状态独立追踪，一个任务被多个人领的场景天然支持。

### 流水——复用现有 `data.finance` 数组

方案把 NT 流水写入 `data.finance`，这正是现有代码已经在用的 NT 流水数组（dashboard.html 第 1471 行：`data.finance = Array.isArray(loaded.finance) ? loaded.finance : []`）。新增 `type: "task_reward"` 字段来区分流水类型，和现有的 `data.finance_cny.transactions`（RMB 流水）互不干扰。复用得当。

### 交互流程——每一步都有判断条件

§2.2 的领取委托五步校验（是否登录→是否已领→是否满额→任务状态→写入）、§3.3 的审核→转账六步流程（更新状态→写审核信息→写流水→保存→刷新），都不是"大概这样"的描述，而是给出了具体到字段的伪代码。实施者能直接翻译成 JS。

### NT 池保护——考虑到了

§3.4 在确认通过前检查 `totalIssued + task.points > poolTotal`，防止超额发放。这个细节很多方案会漏掉。

## 🟡 需要补上的缺口

### 缺口 1：`switchTab()` 不知道怎么处理新 tab

**问题**：方案 §6.1-6.2 把冒险者 tab 从五个改成四个，名字也变了：

| 旧 tab 名 | 新 tab 名 | 对应的渲染函数 |
|-----------|----------|---------------|
| `taskhall`（我的委托） | `profile`（我的资料） | `renderMyProfile()`（新增） |
| `map`（冒险任务） | `questboard`（任务公告栏） | `renderQuestBoard()`（新增） |
| `timeline` | `timeline` | 不变 |
| `budget` | — | **去掉** |
| `settlement` | — | **去掉** |
| — | `leaderboard`（排行榜） | 未定义 |

但 `switchTab()` 函数（第 1517 行）当前是：

```javascript
if (name === 'taskhall') {
  if (currentUser && currentUser.role === 'builder' && !previewMode) {
    renderDailyView('staff'); renderTasks(); renderCompleteness();
  } else {
    renderDailyView('player');
  }
}
if (name === 'map') renderDailyView('player');
```

方案没说这段代码要改成什么样。实施者需要知道：

- `name === 'profile'` → 调 `renderMyProfile()`
- `name === 'questboard'` → 调 `renderQuestBoard()`
- `name === 'leaderboard'` → 调什么？（方案没定义）
- `name === 'taskhall'` 和 `name === 'map'` 这两个旧 case 还要保留吗？（共建者的猎人公会还在用 `taskhall`）

**建议**：在方案里加一节"`switchTab()` 改造"，把新旧 case 的对应关系写清楚。

### 缺口 2：NT 管理员面板挂在哪

**问题**：方案 §三 详细设计了 NT 管理员面板（发布任务 + 待审核 + NT 池统计），但没有说它在现有 tab 结构里的位置。

当前共建者 tab（dashboard.html:494-501）：
```
🌍 创世终端 / 👥 冒险者档案 / 📋 猎人公会 / 📅 世界时间线 / 💰 资源金库 / 🧾 通关结算
```

NT 管理员面板应该：
- 是猎人公会（`tab-taskhall`）里的一个子视图？
- 是共建者模式栏（`builderModeBar`）里的一个新模式？
- 是新加的一个独立 tab？

三种方案各有影响范围，方案必须选一种。**这个决定会影响 HTML 加什么元素、`switchTab()` 加什么 case、共建者能不能同时看到猎人公会和 NT 管理面板。**

### 缺口 3：隐藏任务的触发机制没定义

**问题**：方案 §2.3 说：

> 隐藏任务不显示在所有人的公告栏里。只有被导师触发后，该冒险者才能在自己的公告栏看到。

但 `claimants[]` 的结构是：

```js
claimants: [{ name, claimed_at, status, submitted_at, ... }]
```

数组里只有"已经领取了任务的人"。没有"被导师触发但还没领取"的状态。

导师怎么触发一个隐藏任务给特定冒险者？是：
- 在 NT 管理面板里把某个冒险者加到该隐藏任务的 claimants 里（status = "triggered"）？
- 还是另有 UI？
- 触发后，冒险者的公告栏怎么知道要显示这个隐藏任务？筛选逻辑是 `claimants 中有当前用户`——如果触发时 status 是 "triggered"，那筛选条件需要改成 `claimants 中有当前用户且 status ≠ 'completed'`。

**这个逻辑差一环就闭环了。** 补上"触发"操作的定义和 status 的取值。

### 缺口 4：没有"放弃委托"

**问题**：冒险者可以领取→提交，但不能放弃。

场景：冒险者领了一个 NT 15 的支线任务，后来发现做不完，想放弃。现在没有这个操作——任务会永远卡在 `in_progress` 状态，占着名额（如果有限额的话），也影响冒险者的进度统计。

**建议**：加一个 `status: "dropped"`，以及对应的 `claimants` 更新逻辑和 UI 按钮。

### 缺口 5：排行榜 tab 只有名字

**问题**：方案 §6.2 加了"🏆 排行榜" tab，但全文没有描述它显示什么、数据来源是什么、渲染函数叫什么。实施者拿到方案，不知道该写一个什么样的排行榜——是按 NT 总量排？按完成任务数排？按成就数排？显示多少人？能不能筛选？

**要么给排行榜写一节内容（哪怕简单表格），要么先不加这个 tab，等下一版方案。**

## 🔧 跟现有代码的冲突点

### 冲突 1：`renderDailyView('player')` 在多处被调用

`renderDailyView('player')` 当前在以下位置被调用（grep 结果）：

| 位置 | 调用场景 |
|------|---------|
| 第 581-582 行 | 攻略地图 tab 的日期选择器 `onchange` |
| 第 1639 行 | `switchTab('map')` |
| 第 1644 行 | `switchTab('taskhall')` 的冒险者分支 |
| 第 1773 行 | `refreshDailyViews()` |
| 第 1809 行 | 某个刷新逻辑 |
| 第 2001 行 | `renderStrategyMap()` 兼容函数 |

方案 §6.4 说"不再由 taskhall tab 调用，改为 `renderMyProfile()` 和 `renderQuestBoard()`"。**但其他地方还在调用它。** 尤其是 `refreshDailyViews()` 和攻略地图的日期选择器——这些要不要改？

**建议**：方案里加一个清单，列出 `renderDailyView('player')` 的每个调用点，标注"保留/替换/删除"。

### 冲突 2：去掉冒险者的"资源金库"和"通关结算"——确认可以？

方案 §6.2 的冒险者 tab 去掉了这两个。当前冒险者可以看到资源金库（`switchTab('budget')`，第 1638 行）和通关结算（`switchTab('settlement')`，第 1639 行）。去掉后冒险者完全看不到自己的 NT 余额和结算信息了吗？"我的资料"里有 NT 余额和流水，所以功能还在——但需要在方案里明确一句。

### 冲突 3：NT 池统计在单世界场景下可能算错

方案 §3.4 计算 `totalIssued` 时遍历 `data.finance` 中所有 `type === 'task_reward'` 的记录。在当前单世界架构下，`data.finance` 混着所有期的数据。如果未来多世界上线，这个计算要加上世界 ID 过滤。方案没提这个前提假设。

## 审查原则逐项评分

| 原则 | 评分 | 说明 |
|------|------|------|
| **少即是多** | ✅ | 去掉日期选择器、加载导出、任务总览表、左侧人员列表——这些确实是冒险者不需要看到的东西。四个 tab 简洁聚焦 |
| **复用优先** | ✅ | 数据层扩展 `data.tasks` 而非替换；流水复用 `data.finance` 数组；NT 池数据来自 `data.budget.nt_total_pool`。不改已有字段，不建新 localStorage key |
| **需求对齐** | ✅ | 领取→提交→审核→转账的完整链路，每一步都能追溯到实际运营场景（NT管理员发任务，冒险者接任务，管理员审核确认，NT到账） |
| **全局最优** | 🟡 | NT 管理员面板没说明挂在现有 tab 结构的什么位置；`switchTab()` 没给出新 tab 名对应的 case；`renderDailyView('player')` 的多个调用点未清理 |
| **长期主义** | 🟡 | `claimants[]` 数组设计可扩展，但隐藏任务触发机制缺一个状态值；缺少"放弃委托"操作会在长期运营中积累僵尸任务；排行榜完全没有设计内容 |
