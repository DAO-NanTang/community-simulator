# 🥬 白菜拍卖行 · UI / 原理 / 交互方案

> 2026-07-09 | 待 AI 审查 | v1
> 归属：南塘云村 Hub 新增卡片（6 → 8 项之一）
> 定位：**线下口头竞拍 + 站内记录**。站内负责「发起拍卖会 / 报名冻结保证金 / 主持人录入成交 / 自动结算」，实际举牌竞价在集合地点线下进行。

---

## 一、核心原理（一句话）

> 谁都能发起一场**有固定时间和集合地点**的拍卖会；想参拍的人先在站内**冻结保证金**报名（可匿名）；到点后大家到集合地点**线下口头竞价**；结束后由**主持人**在站内录入「中标人 + 成交价」，系统**自动结算**（未中标退保证金、中标付款给卖家）。

设计取舍：
- 不做站内实时出价（避免纯前端并发/刷新竞争问题），把「价格发现」交给线下，站内只做**门槛（保证金）+ 通知 + 结算**这三件前端能做好的事。
- 保证金是**参拍资格门槛**，也是防止乱举牌的约束，对齐你说的「余额超过一定值才能竞价」。

## 四、数据结构

```js
data.auctions = [
  {
    id: 'auc_001',
    title: '工笔花鸟画',
    // 拍品来源
    item_source: 'inventory',      // 'inventory' | 'freeform'
    item_id: 'inv_xxx',            // item_source=inventory 时
    desc: '第四期共创营创作',        // 自由描述 / 备注
    seller: '若雨',                 // 拍品所有者（自由填写时=发起人）
    host: '文峰',                   // 主持人（发起人，可≠seller）
    start_price: 100,
    deposit: 20,                   // 保证金额度
    time: '2026-07-09T20:00',
    location: 'teahouse',          // 复用 getGameLocations 的 id
    note: '现场交割',
    status: 'signup',              // signup(报名中) | pending(待结算) | done(已成交) | passed(流拍)
    bidders: [                     // 报名者
      { name: '朝林', anon: false, deposit_tx: 'tx_aa' },
      { name: 'niya', anon: true,  deposit_tx: 'tx_bb' }
    ],
    winner: '',                    // 结算后填
    final_price: 0,
    created_at: '2026-07-09T10:00'
  }
];
```

匿名显示：名单里第 i 个匿名者渲染为「匿名#i」，`anon:true` 时对普通用户隐藏 `name`，仅 `host` / admin 可见真身。

## 五、交互 / 数据流

```
① 发起拍卖会
   填表 → 校验 → push data.auctions（status='signup'）
   若 item_source='inventory'：把该物品标记 reserved（不可同时在集市售卖）

② 报名参拍（冻结保证金）
   校验 calcNtTotal(me) ≥ deposit
   → recordTransaction(AUCTION_DEPOSIT, from:me, to:'拍卖行托管', amount:deposit)
   → auctions.bidders.push({name, anon, deposit_tx})
   → 我的余额 −deposit（冻结）

③ 开拍提醒（到点前）
   报名者信箱收到「🔨 拍卖会 20:00 在茶馆开始，请到场」

④ 线下口头竞价（站外，不在系统内）

⑤ 主持人录入成交
   选中标人 W + 成交价 P → [确认成交并结算] → status='done'
   结算（见第六节记账）

   或 [流拍] → status='passed' → 全体报名者退保证金
```

**状态机**：`signup → pending（到点后可录入）→ done / passed`
（`pending` 可省略，主持人任意时刻可录入；到点前录入给二次确认提示。）

## 六、记账设计（新增 3 个交易类型）

`TX_TYPES` 新增：

| type | from | to | amount | 说明 |
|------|------|-----|--------|------|
| `auction_deposit` | 参拍人 | 拍卖行托管 | 保证金 | 报名冻结 |
| `auction_refund`  | 拍卖行托管 | 参拍人 | 保证金 | 未中标/流拍退回 |
| `auction_settle`  | 中标人 | 卖家 | 成交价 | 成交付款 |

> 「拍卖行托管」是一个系统伪账户（与「系统」「任务池」「高琳姐」同类），用于持有冻结的保证金。`calcNtTotal` 天然反映冻结效果（报名后余额减少）。

**结算算法（成交）**——采用「全额退保证金 + 中标人付全款」，净额等价于「保证金抵扣」，但只需每人一条退款 + 一条成交，代码最简、原子完成无中间态风险：

```
for 每个报名者 b:
    recordTransaction(auction_refund, 托管 → b, deposit)   // 含中标人也先退
recordTransaction(auction_settle, 中标人 → 卖家, final_price)
若拍品绑定 inventory：item.status = 'sold'
```

净结果：卖家 +P；中标人 −P；其余 ±0。UI 上以「保证金抵扣、补付差额」话术呈现，符合直觉。

**流拍**：只做 `for 每个报名者: auction_refund(托管→b, deposit)`，status='passed'，inventory 解除 reserved。

## 七、提醒 / 信箱集成

复用已有 `computeInboxMessages` + 子函数模式，新增 `computeAuctionReminders()`：

| 触发 | 对象 | 文案 | 类型 |
|------|------|------|------|
| 开拍前 30 分钟 ~ 开拍 | 报名者 | 🔨「工笔花鸟画」拍卖会 20:00 在 🍵茶馆 开始 | `auction_soon` |
| 结算完成 | 中标人 | 🎉 你以 180 NT 拍得「工笔花鸟画」 | `auction_won` |
| 结算完成 | 未中标/流拍 | 💰 拍卖结束，保证金 20 NT 已退回 | `auction_refunded` |

`renderMsg` 走通用 `m.text` 分支（已在提醒系统改造中支持），只需在 `TYPE_ICON`/`TYPE_LABEL` 加 3 个条目。
