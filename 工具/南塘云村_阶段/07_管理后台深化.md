# 阶段七：管理后台深化

> **目标**：数据看得清、治理管得了。从"日常使用工具"升级为"社区治理平台"。
> **状态**：📝 待审查
> **依赖**：阶段一至六（所有数据源）

---

## 一、本阶段要解决什么问题

1. **NT去哪了** —— 流向图、发行/回收统计、各人收支明细
2. **东西去哪了** —— 物品消耗报表、浪费率、从田间到餐桌的追踪
3. **谁干了什么** —— 打扫完成率、委托完成率、活跃度排行
4. **历史回放** —— 任一时间段内南塘发生了什么
5. **治理工具** —— 提案、投票、规则修改

一句话验收标准：**管理员打开后台，能看到全局数据统计、NT流向图、物品消耗报表。**（治理系统和事件回放延后到后续版本）

```
- 会议使用：投影回放本周事件，辅助决策
- 对外展示：给访客看南塘的日常
```

## 九、存储升级方案
> ⚠️ **存储升级前置**：照片 base64 存储从阶段二就开始积累数据。不应等到阶段七才做存储升级——建议阶段三（打扫拍照）引入照片压缩+IndexedDB存储。迁移路径：①阶段三实现照片存IndexedDB ②阶段七前将旧localStorage照片迁移到IndexedDB ③localStorage保留非照片数据。

到阶段七，数据量可能很大。需要从 localStorage 迁移到更合适的方案：

```
数据分层存储策略：

  热数据（频繁读写）:
    - 用户会话、当前状态 → localStorage
    - 当前物品列表、当前空间状态 → localStorage
    - 最近的弹幕/通知 → localStorage

  温数据（偶尔读取）:
    - 本月事件流 → localStorage (压缩)
    - 本月记录 → localStorage (压缩)

  冷数据（归档）:
    - 历史事件流 → IndexedDB 或 JSON导出文件
    - 历史记录 → IndexedDB 或 JSON导出文件
    - 照片 → IndexedDB (支持更大存储)

  定期归档:
    - 每月1号自动归档上月数据
    - 照片压缩为缩略图（原图导出存档）
```

## 十一、待确认问题

1. **治理系统的权限**：谁可以发起提案？谁可以投票？目前设计是在地成员投票，需要确认。
2. **提案的法律效力**：提案通过后是自动执行（如修改打扫NT），还是需要管理员手动执行？
3. **数据归档**：归档到 IndexedDB 还是导出 JSON 文件？IndexedDB 更方便但不如文件可迁移。
4. **事件回放的性能**：如果一年有几千条事件，回放界面的性能需要优化（虚拟滚动、分页加载）。
5. **是否需要通知/邮件系统**：提案发起时通知所有投票人？打扫逾期时提醒管理员？
6. **阶段七的优先级**：数据统计可以早点做（甚至部分合并到阶段五），治理工具可以晚点。你怎么看？
