# 阶段三：卫生 + 脏污度 + 打扫审核

> **目标**：空间自动变脏，打扫有图有真相，审核通过获得贡献值+经验值（虚拟账本记账，月底NT结算）。
> **状态**：📝 待审查
> **依赖**：阶段一（身份系统）、阶段二（空间定义）

---

## 一、本阶段要解决什么问题

1. **空间状态可视化** —— 哪些地方该打扫了？一眼能看出来
2. **打扫有据可查** —— 打扫前拍照、打扫后拍照，管理员审核，不能糊弄
3. **打扫获得贡献值** —— 劳动有价值，自动发放奖励

一句话验收标准：**打开地图能看到每个空间的卫生状态，打扫需要拍前后照，管理员审核通过后自动获得贡献值+经验值。**

## 二、脏污度模型

### 2.1 核心概念

每个空间（由阶段二定义）都有一个**脏污度**数值，表示该空间的脏污程度。

```
脏污度范围：0（一尘不染）→ 100（极度脏乱）

  🟢 干净        0-30    不需要打扫
  🟡 轻度脏污    31-60   可以打扫，不紧急
  🔴 需要打扫    61-80   出现待打扫标记
  🚨 严重脏污    81-100  优先级最高，影响美观和体验
```

### 2.2 自动增长规则

系统每天自动为每个空间增加脏污度。不同空间类型增加速度不同：

| 空间类型 | 每日自动增长 | 说明 |
|----------|:---:|------|
| 卫生间 (bathroom) | +12~18 | 使用频率高，脏得快 |
| 厨房 (kitchen) | +8~15 | 做饭容易脏 |
| 画室 (studio) | +5~10 | 颜料、材料多 |
| 走廊 (hallway) | +5~10 | 人来人往 |
| 卧室 (bedroom) | +3~8 | 私人空间，较干净 |
| 储物间 (storage) | +2~5 | 很少使用 |
| 洗衣房 (laundry) | +3~8 | 中等 |
| 户外 (outdoor) | +1~3 | 自然因素 |

**增长逻辑**：
```
每天页面首次打开时（或定时检查）：
  lastCheckDate = nt_cleaning.lastCheckDate || today
  daysPassed = today - lastCheckDate
  
  if daysPassed >= 1:
    对每个空间：
      如果空间有 cleaningSchedule:
        每日增长 = 该空间类型的基准值（固定值，不使用随机浮动，避免争议）
        新脏污度 = min(100, 当前脏污度 + 每日增长 × daysPassed)
        更新空间脏污度
    更新 lastCheckDate = today
```

```js

### 3.5 审核不通过处理

```
审核不通过时：
  → 打扫记录状态: "rejected"
  → 脏污度不变（不重置）
  → 通知打扫人："你的厨房打扫未通过审核，原因：照片不够清晰，请重新打扫并拍照"
  → 该打扫任务重新出现在待办列表
  → 打扫人可以重新提交（拍新照片）
  → 不扣NT，也不奖励NT
```

## 四、排班系统（延后到阶段七）

排班系统（轮值名单、currentTurn 轮转、自动生成打扫任务）在 MVP 阶段过度设计。初期在地成员3-5人时，口头协调比系统排班更灵活。阶段三只做脏污度可视化和拍照审核，排班功能延后到阶段七。
例如：
  assignedTo = ["朝林", "小红", "大飞"], currentTurn = 1 (小红)
  小红打扫完成并审核通过 →
    currentTurn = (1 + 1) % 3 = 2 (大飞)
    nextDue = 今天 + 7天

频率 → 天数映射：
  daily → 1天
  every3days → 3天
  weekly → 7天
  biweekly → 14天
  monthly → 30天
```

### 4.3 自动生成打扫任务

```
系统每天检查：
  对每个有排班的空间：
    if (今天 >= nextDue) AND (脏污度 > 30):
      创建打扫提醒任务
      在地图该空间上显示 "🧹 该打扫了"
      当前轮值人的待办列表中出现此项

    if (脏污度 > 60) AND (未有人在打扫):
      自动生成一条 cleaning 类型的 nt_task
      在地图该空间上显示 "🚨 需要打扫"
      所有在地成员都能看到并接取（不限于轮值人）
```

## 六、NT 计算逻辑

### 6.1 打扫相关NT

| 事件 | NT变动 | 获得者 | 条件 |
|------|:---:|--------|------|
| 打扫审核通过 | +10 | 打扫人 | 管理员确认前后照片有效 |
| 打扫审核不通过 | 0 | — | 脏污度不变，重新打扫 |
| 逾期未打扫 | 0 | — | 不惩罚，但地图上🚨标记一直亮着 |
| 连续打扫奖励 | +2×连续次数 | 打扫人 | 同一人连续打扫同一空间（可选，阶段7加入） |

### 6.2 防刷机制

```
防止刷NT的打扫：
  1. 必须有前后对比照片，不能只拍一张
  2. 照片必须不同（简单的像素比较，防止同一张照片传两次）
  3. 管理员审核是人工壁垒——照片里确实干净了才能通过
  4. 脏污度 < 30 时不能提交打扫（"已经很干净了，不需要打扫"）
  5. 同一人不能审核自己提交的打扫（如果管理员也打扫了，需要另一个管理员审核）
```

## 十、待确认问题

1. **脏污度增长速率**：上述数字（厕所+12~18等）是估的。你们实际感受是哪个空间脏得最快？频率大概几天需要打扫一次？
2. **打扫NT奖励**：10 NT 合适吗？不同空间类型是否不同价？（厕所比储物间更难打扫，是否给更多？）
3. **拍照方式**：是调用手机相机拍照，还是上传照片文件？建议先用文件上传（简单），后续可接相机 API。
4. **管理员不在线时**：如果管理员几天不上线，打扫审核积压怎么办？是否需要"自动通过"机制（比如24小时后自动通过）？还是必须人工审核？
5. **打扫频率**：上述5个频率（每天/每3天/每周/每两周/每月）够用吗？
