## 一、总体评价

整体架构设计扎实，**"地图 + 账本"双核心模型**贯穿始终，事件流作为单一真相来源（source of truth）的思路是正确的。七个阶段的依赖关系清晰，每个阶段有明确的验收标准。文档本身可读性强，数据结构定义完整。

### 📄 README_索引.md

| 原则 | 判断 | 问题 |

| 复用优先 | ✅ | `nt_users` 与 dashboard 共享设计合理 |
| 需求对齐 | ⚠️ | **欠费后是否限制操作**列为"待确认问题"（[01:527]），但后续阶段未回应该问题。如果在阶段一不决策，后续阶段的行为（欠费人能否接委托？能否获得打扫NT？）会出现歧义。 |
| 全局最优 | ⚠️ | **NT 池模型与各人余额的耦合**：[01:345] `nt_nt_pool` 中每人余额存在 `nt_users[name].ntBalance`，而 NT 流水存在 `nt_nt_pool.transactions`。`nt_events` 也有 `ntChange` 字段。同一笔 NT 变动存在三处记录——任意一处不同步就会出现对账困难。 |
| 长期主义 | ⚠️ | **`nt_nt_pool` 统计字段冗余**。`circulating`（流通中）是"自动算"的字段，但它是从 `totalIssued - totalRecycled` 推导，而非从各人余额加总得出。如果手动转账记录不一致，这两个值会漂移。建议要么只存原始数据（流水），统计值始终实时计算；要么在每次写入时原子更新。 |
| 长期主义 | ⚠️ | **管理员初始设置未决**：[01:528] 问"写死在代码里还是首次启动引导创建"——如果写死，多人部署时会冲突；如果引导创建，需要考虑 nt_users 为空的检测逻辑和安全性（任何人都能抢注管理员？）。 |

### 📄 03_卫生脏污度与打扫审核.md

| 原则 | 判断 | 问题 |
|------|:---:|------|
| 少即是多 | ⚠️ | **排班系统复杂度偏高**。轮值名单 + currentTurn 轮转 + nextDue 计算 + 自动生成打扫任务——这是一个完整的调度子系统。如果初期只有 3-5 个在地成员，口头排班可能比系统排班更灵活。建议阶段三先做脏污度可视化和拍照审核，排班延后到阶段七。 |
| 少即是多 | ⚠️ | **脏污度随机浮动**。[03:58-59] 每日增长 = 基准值 ± 随机浮动(2-3)——随机性增加了不可预测性，但对于一个以"公平感"为目标的系统，随机浮动可能引起争议（"为什么厨房今天涨了 14 而昨天只涨了 8？"）。建议初期用固定值。 |
| 复用优先 | ⚠️ | **脏污度数据重复存储**。`nt_spaces[spaceId].dirtiness`（[02:124]）和 `nt_cleaning.spaces[spaceId].dirtiness`（[03:73]）存储同一数据。如果两处不同步，以哪个为准？建议只存一处，另一处通过 spaceId 引用读取。 |
| 需求对齐 | ⚠️ | **"自动通过"机制待确认**。[03:507] 问"管理员几天不上线怎么办"——这是核心体验问题，但列为待确认而非设计决策。如果打扫审核 24 小时无人处理自动通过，那照片审核就失去了意义；如果必须人工审核，管理员的在线率就是系统瓶颈。 |
| 全局最优 | ⚠️ | **阶段三的自动打扫任务依赖阶段四的数据结构**。[03:338-339] 说"自动生成一条 cleaning 类型的 nt_task"，但 `nt_tasks` 是阶段四才定义的。这形成一个跨阶段的前向依赖——阶段三的实现需要先引入阶段四的 `nt_tasks` 结构，哪怕阶段四还没做。 |
| 长期主义 | ⚠️ | **防刷机制中"照片必须不同"**。[03:389] 说"简单的像素比较"——这是不可靠的。同一个人从同一角度拍同一空间的两张照片，像素级对比可能判定为"不同"（光线变化、轻微位移），也可能被绕过（拍两个不同角落而非同一位置的前后对比）。建议改为依赖人工审核而非自动像素比较。 |

**具体位置**：
- [03_卫生脏污度与打扫审核.md:73] 与 [02_空间地图与物品标记.md:124] — 脏污度重复存储
- [03_卫生脏污度与打扫审核.md:338-339] — 前向依赖阶段四的 nt_tasks
- [03_卫生脏污度与打扫审核.md:389] — 像素比较防刷不可靠

### 📄 07_管理后台深化.md

| 原则 | 判断 | 问题 |
|------|:---:|------|
| 少即是多 | ⚠️ | **治理系统（提案/投票）可能是过早设计**。[07:296-376] 提案系统包含发起/分类/投票选项/投票人范围/时长/通过阈值——这是一个完整的 DAO 治理模块。如果南塘目前只有几个在地成员，口头协商比链上投票更高效。建议数据统计先做，治理系统等社区规模达到需要时再设计。 |
| 少即是多 | ⚠️ | **事件回放的"播放/暂停/倍速"交互**。[07:258-280] 像视频播放器一样的事件回放界面——如果数据量小（几十条事件），直接滚动列表就够了；如果数据量大（几千条），逐条播放没有实际用处。回放的价值在于"看到一段时间内的变化"，缩放/筛选比播放/暂停更重要。 |
| 复用优先 | ✅ | 所有统计数据均来自前六个阶段的数据，不引入新的数据采集 |
| 需求对齐 | ✅ | 每个统计模块（NT总览/物品消耗/打扫统计/委托统计）都对应明确的治理需求 |
| 全局最优 | ⚠️ | **数据导出的 CSV 格式**。[07:423] 提到导出 CSV——但如果数据是嵌套结构（物品有 events 数组，记录有 comments 数组），CSV 无法直接表达。导出格式的选择应该基于数据结构，而非预先决定。 |
| 长期主义 | ⚠️ | **存储升级方案中提到 IndexedDB 但未定义迁移路径**。[07:435-455] 从 localStorage 到"热/温/冷"三层存储的升级是架构变更，需要定义：何时触发迁移？迁移期间数据不一致怎么办？旧数据是否需要转换格式？如果阶段七才做存储升级，前六个阶段积累的照片 base64 数据可能已经超出 localStorage 限制，导致升级前系统就不可用了。 |
| 长期主义 | ✅ | 归档策略（每月1号自动归档、照片缩略图）考虑周全 |

## 三、跨文件共性问题

以下问题跨越多个阶段文件，需要从架构层面统一处理：

### 🔴 问题 1：NT 多处记账，缺乏单一真相来源

| 存储位置 | 记录内容 | 文件 |
|----------|----------|------|
| `nt_events[].ntChange` | 事件附带的 NT 变动 | README |
| `nt_nt_pool.transactions` | NT 流水 | 阶段一 |
| `nt_users[name].ntBalance` | 各人余额 | 阶段一 |

三处记录同一笔 NT 变动，任一处出错都会导致对账困难。**建议**：`nt_events` 为唯一事件记录（不含 `ntChange`，只记录事实），`nt_users[name].ntBalance` 为当前余额（通过事件流计算得出或原子更新），`nt_nt_pool.transactions` 在阶段一后废弃，统一使用事件流查询。

### 🔴 问题 2：多处数据重复存储

| 数据 | 存储位置 1 | 存储位置 2 |
|------|-----------|-----------|
| 脏污度 | `nt_spaces[].dirtiness` (阶段二) | `nt_cleaning.spaces[].dirtiness` (阶段三) |
| 用户位置 | `nt_users[].currentSpace` (阶段一) | `nt_social.presences` (阶段六) |
| 互动数据 | `nt_journal.entries[].likes/likedBy` (阶段五) | `nt_social` (阶段六) |

**建议**：每个数据只在一处存储，其他模块通过 ID 引用读取。`nt_spaces` 为空间定义的唯一存储，脏污度统一存在 `nt_cleaning`；用户位置统一存在 `nt_users`；阶段六互动数据统一存在 `nt_social`。

### 🟡 问题 3：照片存储策略贯穿全阶段但从未解决

阶段二、三、五、六都涉及照片（物品拍照、打扫前后照、记录照片），且都使用 base64 存储。localStorage 5MB 限制在阶段三（拍照审核）就会成为瓶颈，但直到阶段七才提出存储升级方案。

**建议**：在阶段二或阶段三就引入照片压缩 + IndexedDB 存储，不要等到阶段七。照片的存储策略是基础设施，不应跟随应用层阶段推进。

### 🟡 问题 4：所有定时逻辑依赖"打开页面"

- 阶段二的过期检查："打开页面时触发" [02:278]
- 阶段三的脏污度增长："每天页面首次打开时" [03:51]
- 阶段三的打扫到期检查："系统每天检查" [03:328]
- 阶段四的委托到期：无触发机制说明

纯前端 localStorage 应用无法在后台运行定时任务，这是技术限制。但文档应在设计层面承认这个限制，并明确：如果用户连续 N 天不打开页面，所有时间相关逻辑都会在下次打开时一次性追赶。这个行为需要和用户明确沟通。

### 🟡 问题 5：跨阶段前向依赖

- 阶段三引用 `nt_tasks`（阶段四才定义）[03:338]
- 阶段一的 `nt_users` 预留了 `currentSpace`/`status`（阶段六才使用）[01:93-94]
- 阶段五的点赞字段（`likes`/`likedBy`）和阶段六的互动系统重叠

**建议**：每个阶段的数据结构只定义本阶段需要的字段。后续阶段如果需要新字段，通过"扩展"而非"预留"来添加。预留字段在几个月后才实现时，很可能已经不符合当时的需求。

### 🟡 问题 6：NT 经济的发行/回收平衡未分析

粗略估算（4 个在地成员场景）：
- **月发行量**：注册 400NT（4人×100，一次性）+ 打扫 160NT（4人×4次×10NT）+ 记录 360NT（4人×30天×3NT/天）+ 物品 ~80NT = 约 1000NT/月（不含注册）
- **月回收量**：住宿费 8400NT（4人×30天×70NT/天）

这个模型是严重通缩的——一个在地成员如果只靠打扫和发记录，每月赚 ~130NT，但住宿费要 2100NT/月。如果没有委托（阶段四）的大量 NT 流转，在地成员根本无法维持住宿。

**建议**：在阶段一或 README 中增加 NT 经济平衡的粗略分析，确认各参数（注册赠送、打扫奖励、日费率）是否在合理范围内。当前的数字（注册 100、住宿 70/天）可能来自实际经验，但从文档看不出是否经过验证。

## 四、审查总结

### 整体符合度

| 原则 | 评分 | 说明 |
|------|:---:|------|
| 少即是多 | ⭐⭐⭐ | 弹幕、排班、治理系统存在过度设计倾向；头像选择器、记录发布表单可简化 |
| 复用优先 | ⭐⭐⭐⭐ | 核心数据模型（事件流、空间、用户）复用良好；主要问题在数据重复存储（脏污度、位置、互动） |
| 需求对齐 | ⭐⭐⭐½ | 每阶段有明确的验收标准；主要问题是一些"待确认"的决策实际上影响多个阶段的设计 |
| 全局最优 | ⭐⭐⭐ | 存在跨文件数据重复、前向依赖、缺少架构层面的照片/存储/NT经济决策 |
| 长期主义 | ⭐⭐⭐ | 事件流模型好；但 localStorage 限制、NT 经济平衡、数据迁移策略需要更早解决 |

### 最需要优先解决的 5 个问题

1. **NT 经济参数验证**（全局/长期主义）——日费率 70NT vs 日常收入 ~4-6NT/天，需要确认是否合理
2. **照片存储方案前置**（长期主义）——不应等到阶段七才解决，阶段二就该确定
3. **数据重复存储消除**（全局最优）——脏污度、位置、互动数据统一存储位置
4. **争议处理机制**（需求对齐）——委托确认僵局需要超时/仲裁设计
5. **NT 多处记账统一**（全局最优）——`nt_events` 作为唯一事件来源，NT 变动通过查询计算

### 设计亮点（值得保留）

- **地图 + 账本双核心模型**：空间维度和时间维度互补，概念清晰
- **事件流作为 backbone**：所有操作都产生事件，天然支持回放和审计
- **`ntChange` 归零的互动事件设计**：弹幕、留言、点赞不产生 NT，保持经济系统和社交系统解耦
- **委托预扣机制**：发布时预扣 NT，防止发布人赖账
- **退宿欠费宽容处理**：允许欠费退宿但记录欠款，兼顾现实灵活性和账务清晰
- **阶段间清晰的依赖链**：1→2→3→4→5→6→7 的线性依赖使开发计划可控
