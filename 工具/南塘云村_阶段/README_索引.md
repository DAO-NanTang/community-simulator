## 〇、总体架构：地图 + 账本

在进入各阶段之前，先理解整个系统的核心模型：

```
┌──────────────────────────────────────┐
│         南塘云村 · 数字孪生           │
│                                      │
│  ┌────────────┐    ┌────────────┐   │
│  │   地图      │    │   账本      │   │
│  │  (空间索引) │    │  (时间索引)  │   │
│  │            │    │            │   │
│  │ 从地图点进去 │←──→│ 从账本查过来 │   │
│  │ 看到空间现在  │    │ 看到物品的    │   │
│  │ 有什么       │    │ 完整一生     │   │
│  └────────────┘    └────────────┘   │
│                                      │
│  地图 = 在哪？         账本 = 发生了什么？  │
│  地图 = 空间视图        账本 = 时间视图    │
└──────────────────────────────────────┘
```

**核心概念**：每一件实物在系统里有一个对应的数字记录（数字孪生）。维护这个数字孪生就是 NT 的来源。云村民看的就是这个数字孪生慢慢被填满、变化的过程。

```

  ↓
阶段7：管理员能看到全局数据、回放历史、治理决策
```

## 二、贯穿全文的核心概念

### 2.1 数字孪生

每一件实物（冰箱里的白菜、仓库里的工具、田里的番茄）在系统里有一个对应的数字记录。这个记录不是静态的——它从被创建到被消耗，经历完整的生命周期。

### 2.2 NT（南塘积分）

NT 是系统内的价值单位。**每一分 NT 都对应一个真实发生的事件**——放了一件物品、打扫了一个空间、记录了日常、完成了委托。

NT 的来源（收入）：
- 注册赠送（一次性）
- 标记物品 +2NT
- 消耗物品 +1NT
- 打扫审核通过 +10NT
- 发布在地记录 +3NT
- 完成委托（按委托金额）
- 被打赏

NT 的支出：
- 住宿费（自动扣除）
- 发布委托（预付）
- 打赏他人

> ⚠️ **NT 经济平衡**：当前参数需要实际运营数据验证。粗略估算（4个在地成员）：月发行约 1,000 NT（不含注册一次性），月回收（住宿费）需验证是否合理。住宿费建议使用**浮动比例制**（每月收入的 X%），而非固定日费率，以避免通缩。参见 [南塘NT经济系统_完整设计.md](../南塘NT经济系统_完整设计.md)。

### 2.3 事件流模型

**核心存储是一个事件流。** 每个物品、每个空间的当前状态，是所有历史事件的最终结果。

```
事件流 (nt_events):

时间 | 类型 | 目标 | 操作人 | NT变动 | 照片 | 备注
-----|------|------|--------|--------|------|-----
D1 10:00 | item_add | 冰箱·上层 | 小杨 | +2 | - | 放入白菜×2
D1 14:00 | check_in | 宿舍101 | 大飞 | -70/天 | - | 入住3天
D3 09:00 | comment | 白菜#003 | 云小明 | 0 | - | "快烂了！"
D5 08:00 | expire_warn | 白菜#003 | 系统 | 0 | - | 还有2天过期
D6 12:00 | item_consume | 白菜#003 | 大飞 | +1 | 📷 | 吃掉了
D6 12:05 | chore_done | 厕所 | 小杨 | +10 | 📷📷 | 打扫完成
D6 15:00 | journal | 厨房 | 阿楠 | +3 | 📷 | "今天包饺子"
```

好处：
- **所有历史可追溯**——"上周那批白菜谁吃掉的？"一查就知道
- **数据可以回放**——云村民看到"这几天冰箱的变化史"
- **NT可核验**——每一分NT都有对应的真实事件

> ⚠️ **设计决策**：`ntChange` 字段在事件流中为可选字段，仅用于快速查看。**NT 余额的权威计算方式是通过事件流实时汇总**（`sum(nt_events.filter(actorId===user).ntChange)`），而不是单独存储在 `nt_users[name].ntBalance` 中。如果两处数字不一致，以事件流汇总结果为准。`nt_users[name].ntBalance` 仅作为缓存值，每次写入时从事件流重新计算。
>
> ⚠️ **定时逻辑标注**：纯前端 localStorage 应用无法在后台运行定时任务。以下定时行为均在**页面打开时一次性追赶**：物品过期检查、脏污度每日增长、打扫到期、委托到期。如果用户连续 N 天不打开页面，所有时间相关逻辑会在下次打开时集中执行。此行为是技术限制，非设计缺陷。

## 五、如何使用这些阶段文档

### 审查流程

1. **先读本索引**，理解整体架构和贯穿概念
2. **按顺序读每个阶段文件**（每个文件独立可读，但建议按顺序）
3. **审查每个阶段时，关注这些方面**：
   - 数据模型是否完整？（缺什么字段？）
   - 交互流程是否合理？（操作步骤能否更少？）
   - NT计算是否公平？（奖励够不够？有没有漏洞？）
   - 和前后阶段的衔接有没有断裂？
   - 有没有现实中无法执行的设计？
4. **修改时，回到对应的阶段文件修改**，然后检查是否影响其他阶段

### AI审查清单

审查每个阶段文件时，AI应回答以下问题：

- [ ] 本阶段的数据结构是否和前后阶段一致？
- [ ] 事件流中的事件类型是否覆盖了本阶段所有操作？
- [ ] NT流入流出是否平衡？（有没有无限刷NT的漏洞？）
- [ ] 权限设计是否有缺口？（谁可以做什么？）
- [ ] 交互流程是否有死胡同？（操作后能否撤销？出错怎么办？）
- [ ] 用户体验是否流畅？（步骤数、等待时间、反馈及时性）
- [ ] 是否有未定义的边界情况？（空房间、过期物品、离线用户……）

## 六、地图编辑器与云村的关系

```
┌─────────────────────┐
│  实景地图编辑器       │  ← 管理员用来「画地图」
│  (实景地图.html)      │     编辑网格颜色、边界、物体
│                      │     输出 → nt_map_data
└────────┬────────────┘
         │ 共享 nt_map_data (localStorage)
         ▼
┌─────────────────────┐
│  南塘云村            │  ← 所有人用来「用地图」
│  (cloud-village.html)│     读取网格 → 叠加 nt_spaces 语义层
│                      │     点击空间 → 弹出物品/任务/状态面板
│                      │     小人叠加 → 谁在哪个空间
└─────────────────────┘
```

**阶段2之前**：地图编辑器要先完成，画好南塘的地图（方位图、宿舍、合作社各层、菜地）。
**阶段2开始**：云村加载地图，管理员在地图上定义空间（框选格子→命名→设类型），之后所有功能都是围绕这些空间展开。
