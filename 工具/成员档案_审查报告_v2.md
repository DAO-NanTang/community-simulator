## 🟡 需要修正的四个问题

### 问题 1：`renderMembers()` 伪代码没实现自己的合并规则表

**位置**：§2.6（第 280-313 行）

**问题**：方案第 270-277 行画了一张三源合并表，第三种情况明确写了：

> 人在 `data.members` 但不在 `nt_users` → `migrateExistingMembers()` 已自动补充到 `nt_users`（idempotent），取合并后结果

但伪代码（第 289-293 行）只遍历了 `nt_users`：

```js
Object.keys(users).forEach(function(name) {
  var user = users[name];
  if (user.role !== memberFilter) return;
  if (!staffData[name]) staffData[name] = { tasks: [], points: 0, done: 0, active: 0, todo: 0 };
});
```

**如果 `migrateExistingMembers()` 因为幂等保护跳过了新加入的人**（当前逻辑是 `Object.keys(users).length > 0` 就整批跳过），那些在 `data.members` 里有但没有 `nt_users` 账号的人就不会出现在档案里。伪代码和合并规则表在第三种情况上不一致。

**修正**：在遍历 `nt_users` 之后加一轮 `data.members` 的补充：

```js
// 补充：data.members 中有但 nt_users 中没有的人
Object.keys(members).forEach(function(name) {
  if (staffData[name]) return; // 已经在 nt_users 遍历时加过了
  var inferredRole = isStaff2(name) ? 'builder' : 'adventurer';
  if (inferredRole !== memberFilter) return;
  staffData[name] = { tasks: [], points: 0, done: 0, active: 0, todo: 0 };
});
```

### 问题 2：`renderMemberDetail()` 要改的是第 2691 行，不是第 1475 行的空壳

**位置**：§2.7

**问题**：当前 dashboard.html 里 `renderMemberDetail` 有两个定义：

| 行号 | 内容 |
|------|------|
| 1475 | `function renderMemberDetail(name) {}` — 空壳 |
| 2691 | `function renderMemberDetail(name) { var m = staffData[name]; renderMemberDetailFor('member', name, m); }` — **实际生效的版本** |

JS 的函数声明提升 + 覆盖规则让第 2691 行的版本成为实际运行的代码。方案 §2.7 说要在这里加共建者任务清单和材料表，**必须标注改的是 2691 行的版本**，否则实施者可能改到 1475 行的空壳去，改了等于没改。

**修正**：在 §2.7 加一行行号标注，并给出改造后的函数完整代码（目前方案只给了材料表那段 `if (memberFilter === 'builder')` 片段，缺少函数签名和原有逻辑的保留）。

### 问题 3：`saveMember()` 只刷新 `renderMembers()`，可能漏刷关联视图

**位置**：§2.2（第 168 行）

**问题**：备份版本的 `saveMember()` 末尾调的是 `updateAll()`（备份第 1743 行），刷新所有视图。方案版本改成 `localStorage.setItem` + `renderMembers()`，更精准——但跳过了 `recalcFinanceByPerson()` 和 `renderBudget()`。

方案在改名逻辑里手动补调了 `recalcFinanceByPerson()`（第 123 行），这个是好的。但如果用户**只改头像或角色、不改名**，`recalcFinanceByPerson()` 就不会被触发。虽然改头像不需要重建 by_person 汇总，但如果后续有人新增了依赖 `updateAll()` 刷新的逻辑，这里就是坑。

**修正**：两种选一种：
- A（更安全）：保留 `updateAll()` 调用，接受多做一点不必要的工作
- B（更精准）：在 `saveMember()` 末尾显式列出需要刷新的函数：`renderMembers()` + `recalcFinanceByPerson()` + 任何其他依赖 `data.members` 的视图

### 问题 4：`editingAvatarStyle` 和 `editingAvatarSeed` 没有显式声明

**位置**：§2.1（第 85-86 行）

```js
editingAvatarStyle = info.avatar_style !== undefined ? info.avatar_style : 0;
editingAvatarSeed = info.avatar_seed !== undefined ? info.avatar_seed : 0;
```

用 `grep` 在当前 dashboard.html 搜 `var editingAvatar`——无结果。这两个变量是隐式全局变量。虽然 JS 允许，但按照代码规范应在文件顶部显式声明。

**修正**：在方案里加一行，在全局变量声明区加上：
```js
var editingAvatarStyle = 0, editingAvatarSeed = 0;
```
