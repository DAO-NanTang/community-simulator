## 〇、背景：为什么需要做这件事

### 系统里现在有五个"存档"入口，各自为政

dashboard.html 经过长期迭代，积累了大量功能。其中跟"查看历史/归档"相关的入口至少有五个，它们是在不同时期独立添加的，从未被统一设计过。后果是：**同一个已完成的任务，在不同入口看到的归档状态不一样**。

具体五个入口：

| 入口 | 在哪 | 数据来源 | 怎么判断"已归档" | 当前能用吗 |
|------|------|----------|-----------------|-----------|
| 工作台「📦 存档」 | `renderWorkspace` 内 | `data.tasks` 实时数据 | `getTaskDoneDate()` + 超过 7 天 | ✅ 刚做好 |
| 任务大厅「📦 已归档」 | `renderQuestBoard` 内 | `data.tasks` 实时数据 | `t.status === '已完成' \|\| '已确认'` | ⚠️ 能用，但判断标准有问题 |
| 村庄「📦 往期」 | `renderVillageQuestHall` 内 | `data.archived_periods` 快照 | 营期完结时的全量快照 | ❌ 按钮有，但点开显示"即将开放" |
| Hub「📚 档案室」 | `renderArchiveHub` | `users` + `members` | 成员目录 + 打赏/财务案卷 | ✅ 能用（看的是成员档案，不是任务） |
| NT管理「📦 完结本期」 | `renderNTAdmin` 内 | 管理员操作 | `closePeriod()` 批量转角色 | ✅ 能用（管理员功能） |

注意：Hub 档案室和 NT 管理完结本期各自用途明确、逻辑自洽，**不在本次改动范围内**。本次只处理前三个入口的统合。

### 核心矛盾举例

假设任务 A 在 7 月 1 日被审核通过（`claimant.reviewed_at = '2026-07-01'`），距今 9 天。

- 工作台「📦 存档」：`getTaskDoneDate` 算出来是 7 月 1 日，`daysSince > 7`，所以**出现在存档区** ✅
- 任务大厅「📦 已归档」：检查 `t.status`，发现它的值是 `'completed'`（新格式），不等于 `'已完成'`，所以**不出现在归档区** ❌

同一个任务，两边结论不一致。这就是碎片化判断标准造成的问题。

### 本次要达成的目标

1. **统一判断标准**：所有入口都用 `getTaskDoneDate()` + `ARCHIVE_DAYS` 来判断"是否已归档"
2. **统一渲染方式**：任务大厅和村庄往期复用工作台已有的 `renderArchiveList` 函数，不再各自手写渲染逻辑
3. **补全村庄往期**：`data.archived_periods` 数据一直在被写入（`closePeriod` 创建快照），但从没有任何 UI 去读取它。把"即将开放"的占位符替换为真实的营期浏览功能

## 一、已有资产（可直接复用，不需要改）

上一轮实施的工作台存档机制产出了以下函数，本轮直接复用：

### `getTaskDoneDate(task, claim)` — 约 line 1573

统一计算一个任务的「完成日期」。它是整个归档系统的核心判断依据。

工作方式：如果传了 `claim`（领取记录），优先取 `claim.reviewed_at`；如果没传（比如用于全村视角），取最后一个完成领取者的 `reviewed_at`；如果都没有，兜底取 `task.posted_at`。

### `renderArchiveList(containerId, archiveTasks, viewMode)` — 约 line 1627

统一的存档列表渲染引擎。支持四种视图：周（按自然周分组）、月（按年月分组）、年（按年分组）、全部（时间倒序平铺）。**注意**：当前版本内部调用 `renderQuestCard` 时 context 写死为 `'workspace'`——复用前需要先改成可配置的（见下文改 0）。

### `toggleArchive(sectionId)` / `switchArchiveView(sectionId, mode)` — 约 line 1594 / 1606

存档区域的展开/折叠和视图切换逻辑。可以直接复用。

## 二、执行方案

### 改 0（新增）：给 `renderArchiveList` 增加 context 参数

**为什么**：当前函数内部调用 `renderQuestCard` 时，context 硬编码为 `'workspace'`。任务卡片在不同 context 下会显示不同的按钮（如 workspace 显示"提交"，questboard 显示"领取"）。复用到任务大厅和村庄时，必须能传入正确的 context。

**具体操作**：

1. 找到 `renderArchiveList` 函数定义（约 line 1627），签名从：
   ```
   function renderArchiveList(containerId, archiveTasks, viewMode)
   ```
   改为：
   ```
   function renderArchiveList(containerId, archiveTasks, viewMode, context)
   ```
   `context` 默认值为 `'workspace'`。

2. 函数内部所有调用 `renderQuestCard` 的地方，context 参数从硬编码的 `'workspace'` 改为使用传入的 `context` 变量。约 5 处（每个视图模式各一处，以及"全部"模式一处）。

---

### 改 1：抽 `ARCHIVE_DAYS` 共享常量

**为什么**：天数阈值目前写死为数字 `7`，分散在工作台的两处判断中。抽成常量后，所有入口统一引用，将来调天数只改一处。

**具体操作**：

1. 在 `daysSince` / `daysUntil` 附近（约 line 1572）新增一行：
   ```
   var ARCHIVE_DAYS = 7;
   ```

2. 在 `renderWorkspace` 函数中，找到两处 `daysSince(doneDate) <= 7`，把 `7` 替换为 `ARCHIVE_DAYS`。一处在 `myPublished` 收集逻辑（约 line 4034），一处在 `doneArchive` 拆分逻辑（约 line 4274）。

---

### 改 2：任务大厅「📦 已归档」接入统一标准

**为什么**：当前用 `t.status === '已完成' || '已确认'` 判断，这是旧数据格式的判断方式，新数据的 status 用 `'completed'`，会被漏掉。需要改用 `getTaskDoneDate`。

**重要**：任务大厅是**全村视角**（展示所有任务，不限当前用户），和工作的**个人视角**不同。本次只修正判断标准，不改变数据范围。也就是说：任务大厅归档继续显示所有任务中已完成的部分，只是判断"是否已完成"的方式从查 `t.status` 改为用 `getTaskDoneDate`。

**具体操作**：

1. **收集逻辑**（约 line 5819-5820）：删除现有的 `archiveTasks` 收集行，替换为：
   ```
   // 遍历所有 tasks，找出已完成且超过 ARCHIVE_DAYS 天的任务
   // 使用 getTaskDoneDate(t, null) 来判断（不传 claim，函数会自动找完成日期）
   ```
   收集 `{ task, idx, doneDate }` 数组，只保留 `daysSince(doneDate) > ARCHIVE_DAYS` 的。

2. **渲染逻辑**（约 line 5821-5826）：删除旧的平铺列表（最多显示 20 条的简单循环），替换为存档区域 HTML 结构（标题 + 折叠区 + 筛选栏 + 列表容器），然后用 `renderArchiveList('questArchiveList', archiveTasks, 'week', 'questboard')` 填充内容。注意 context 参数传 `'questboard'`。

---

### 改 3：村庄「📦 往期」接入 `data.archived_periods`

**为什么**：`closePeriod()` 已经把每期完结时的 tasks/members/finance/tips 完整快照存入了 `data.archived_periods`，但村庄的「往期」按钮点开后只显示"即将开放"。这是纯粹的功能空缺——数据在，UI 没做。

**数据源说明**：`data.archived_periods` 的结构是一个对象，key 是营期名（如 `"第四期"`），value 是该期完结时的快照：

```
{
  "第四期": {
    tasks: [{ name: "xxx", status: "active", ... }, ...],
    members: { "砚仁": {...}, ... },
    finance: [...],
    tips: [...],
    closed_at: "2026-07-01T12:00:00"
  }
}
```

**具体操作**：

1. 找到 `renderVillageQuestHall` 中处理 `villagePeriod === 'archive'` 的代码（约 line 3070-3074），删除当前的占位符 HTML。

2. 替换为：
   - 读取 `data.archived_periods`（如果没有或为空对象 → 显示"还没有完结的营期"）
   - 遍历所有营期，按完结时间倒序排列，渲染营期卡片列表
   - 每张卡片显示：营期名、完结日期、任务数、成员数
   - 点击卡片 → 展开该期任务列表（复用 `renderArchiveList`，context 传 `'village'`，视图模式固定为 `'all'`，因为快照数据没有按时间分组的必要——它们本来就属于同一个营期）
