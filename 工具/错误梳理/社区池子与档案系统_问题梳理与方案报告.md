## 二、问题详述

### 问题 1：旧版社区池子代码残留

**代码证据**：
- Wizard 终端 HTML 第 864-865 行：`wizCheerPoolTotal` 和 `wizCheerPoolDaily` 输入框，标题为「社区鼓励池（NPC/云村民打赏用）」
- 数据初始化第 1960-1961 行：`data.budget.community_pool_total = 500`、`data.budget.community_pool_daily = 10`
- `sendTip` 函数第 8134 行：`var dailyLimit = pool.community_pool_daily || 10`——每日额度检查
- Wizard 终端读取第 9073-9074 行：读取 `wizCheerPoolTotal` 和 `wizCheerPoolDaily` 的值

**问题逻辑链**：
1. 之前的社区窗口改造中，社区池子被标记为「已删除」
2. 但删除操作不完整——Wizard 终端 HTML 输入框、数据初始化代码、终端读取代码依然存在
3. 旧代码的设计意图是「NPC/云村民从池子里花钱打赏」——这与最终决策（打赏全部个人出）矛盾
4. 残留代码造成了「池子存在但没用对」的混乱状态

### 问题 2：社区池子功能缺失

**代码证据**：
- 全局搜索 `donate`、`捐献`：**0 处**
- 全局搜索 `community_pool_log`（池子流水记录）：**0 处**
- 全局搜索 `community_archives`（社区档案库）：**0 处**
- 现有系统没有捐献功能、没有社区档案库、没有池子流水追踪

**问题逻辑链**：
1. 社区需要一个公共基金（社区池子）作为社区功能的 NT 进出锚点
2. 当前系统没有捐献入口——任何人无法向社区池子捐 NT
3. 没有档案录入和核查的奖励机制——人工整理历史资料的工作无法被激励
4. 没有池子流水记录——无法追溯资金的进出

**问题逻辑链**：
1. 村口 plaza 第 3 张卡片点击 → `showMapInvite()` → 弹出邀请码输入卡
2. `villageModalMap` 本身只显示占位文字
3. Hub 菜单的实景地图标签是灰色不可点击的
4. 社区捐献功能应该放在实景地图入口——但入口本身还没建成

### 问题 4：档案室只有成员目录，缺少标签导航

**代码证据**：
- `renderArchiveHub()`（第 6686 行）：只渲染成员目录（按身份分组平铺），没有标签切换
- 没有「📋归档任务」「📦往期营期」「📜日志」「📝社区档案」「📊结算」标签

**问题逻辑链**：

2. 当前只显示成员目录——只能看人，不能看任务归档、日志、社区档案
3. 之前实现的归档功能（任务大厅「已归档」、村庄「往期」）没有接入档案室
4. 成员目录没有搜索/折叠/排序——人多了找不到

### 问题 5：没有社区档案库

**代码证据**：
- 全局搜索 `community_archives`：**0 处**
- 全局搜索 `entered_by`（数据溯源标记）：**0 处**
- 没有档案录入表单、没有核查界面、没有录入/核查奖励逻辑

**问题逻辑链**：
1. 系统数据（任务、审核、NT 流转）由系统自动产生

### 问题 6：没有运行日志

**代码证据**：
- 全局搜索 `data.activity_log`：**0 处**
- `confirmPostTask`、`reviewTask`、`sendTip`、`changeUserRole`、`saveMember` 等函数内部没有日志记录代码
- 系统执行关键操作后不留痕迹——无法追溯谁在什么时候做了什么

**问题逻辑链**：
1. 系统每天产生大量操作（发布任务、审核、打赏、角色变更）
2. 没有任何操作日志记录
3. 出问题时无法追溯操作历史
4. 社区池子管理员的支出审批更需要日志——谁批准的、多少钱、什么用途

### 问题 7：导出功能不完善

**代码证据**：
- `exportData()` 函数（第 1949 行附近）：只能导出全量 JSON
- 全局搜索 `exportArchive`、`批量导出`：**0 处**
- 没有 Markdown 格式导出、没有选择性导出（只导出某个人/某段时间）
- 没有分散在各界面的单个文件导出按钮

**问题逻辑链**：
1. 现有的 `exportData()` 是把整个 `data` 对象导出为 JSON——粒度太粗
2. 用户只想导出某个成员的案卷、某段时间的打赏记录——做不到
3. 也没有批量导出 + 合并 Markdown 下载的功能

## 三、解决方案效果（纯自然语言描述）

### 3.1 社区池子

系统建立三个 NT 池子：个人池（自己赚自己花）、营队池（管理员设定，用于营地任务奖励和薪水）、社区池（公共基金，用于奖励社区贡献）。社区池由 2-3 人共同管理，花钱需要批准。任何用户都可以向社区池捐献 NT。捐献按钮放在实景地图访客入口——新访客看到的第一页就有社区捐献区。

### 3.2 社区档案库

在 Hub 档案室新增「📝 社区档案」标签。任何人可以录入往期参与者的故事、历史贡献、未数字化资料。录入后进入核查流程——由社区池管理员审核内容真实性。核查通过后，录入者获得 5 NT 奖励，核查者获得 3 NT 奖励，都从社区池出。每条档案标明录入者和核查者——出问题找对应的人。

### 3.3 实景地图访客入口

实景地图从占位符变为新访客的正式入口。页面内容包含三部分：邀请码输入区（输入邀请码成为在地 NPC，没有就跳过）、社区池子捐献区（显示余额、最近捐献者、捐献按钮）、以及未来实景地图交互区域。同一个页面有两个入口：村口 plaza 卡片打开是一级 Modal，Hub 菜单标签打开是三级滑出面板。两个入口展示完全相同的内容。

### 3.4 档案室升级

Hub 档案室从单一成员目录变为六个标签的导航结构：成员目录（含搜索/折叠/排序）、归档任务（已完成超过 7 天的任务）、往期营期（完结营期快照浏览）、运行日志（所有关键操作的流水记录）、社区档案（人工录入的历史资料）、结算报告（每期完结时的财务汇总）。每个标签独立渲染、互不干扰。

### 3.5 运行日志

系统在执行关键操作（任务发布/审核/打赏/角色变更/成员编辑/捐献/档案录入/档案核查/池子支出）时自动记录日志。日志在档案室「📜 日志」标签按时间倒序展示。池子管理员的支出操作同时记录操作者和批准者——出问题时追溯到人。

### 3.6 导出功能

导出分为两种。单个导出：在数据所在位置各有 [📥] 按钮（成员目录旁、个人案卷顶部、归档任务筛选栏旁、每条社区档案卡片上等），点击直接下载，免费。批量导出：档案室顶部「📦 批量导出」按钮，勾选要导出的内容类型和筛选条件，支付 5 NT 后生成合并 Markdown 文件下载。

## 四、解决方案效果（代码逻辑描述）

### 4.1 社区池子

- 新增全局变量 `data.community_pool = { balance, managers, pending_approvals, log }`
- Wizard 终端新增「社区公共基金」设定区（初始金额 + 管理员名单）
- 新增 `donateToCommunityPool(amount)` 函数：检查个人余额 → 扣款 → 入社区池 → 记录流水
- 新增 `requestPayout(amount, note)` 函数：管理员申请支出 → 生成待审批条目
- 新增 `approvePayout(requestId)` 函数：另一管理员批准 → 扣池子余额 → 记录流水
- 社区池子面板渲染函数：显示余额、专线地址、待审批列表、最近流水

### 4.2 社区档案库

- 新增全局变量 `data.community_archives = []`
- 新增 `submitArchiveEntry(title, content, tags)` 函数：创建档案条目 → 状态「待核查」
- 新增 `verifyArchiveEntry(id, approved, reason)` 函数：通过 → 发放录入/核查奖励 → 状态「已入库」
- 档案条目结构：`{ id, type, title, content, tags, entered_by, verified_by, entered_at, reward_entered, reward_verified }`
- 档案室「📝 社区档案」标签：列表渲染 + 标签筛选 + 搜索 + 新建档案表单 + 核查界面

### 4.3 实景地图入口

- 新增 `renderMapEntrance()` 纯函数，返回邀请码区 + 社区捐献区 + 底图区的 HTML
- 改造 `#villageModalMap`：从占位文字 → 调用 `renderMapEntrance()` 填入 `.vm-inner`
- Hub 实景地图标签从灰色改为可点击，`onclick` 调用 `openHubSub('map')` → 三级滑出面板渲染 `renderMapEntrance()`
- 捐献按钮 onclick → 弹出捐献弹窗 → `donateToCommunityPool()`

### 4.4 档案室升级

- `renderArchiveHub()` 重构：顶部渲染六个标签按钮 + 两个常驻操作按钮
- 标签切换变量 `window._archiveTab`，切换时重新渲染对应内容
- 「📂 成员」标签：复用现有成员目录 + 新增搜索框/折叠分组/排序下拉
- 「📋 归档任务」标签：复用 `getTaskDoneDate` + `ARCHIVE_DAYS` + `renderArchiveList`
- 「📦 往期营期」标签：复用 `data.archived_periods` 遍历 + 营期卡片渲染
- 「📜 日志」标签：渲染 `data.activity_log` 倒序列表
- 「📝 社区档案」标签：渲染 `data.community_archives` 列表
- 「📊 结算」标签：渲染 `closePeriod()` 前的财务汇总面板

### 4.5 运行日志

- 新增全局变量 `data.activity_log = []`
- 在以下函数中各加 1 行日志记录：`confirmPostTask`、`confirmCampTask`、`reviewTask`、`sendTip`、`changeUserRole`、`saveMember`、`donateToCommunityPool`、`submitArchiveEntry`、`verifyArchiveEntry`、`approvePoolPayout`
- 日志条目格式：`{ time, type, text }`
- `renderArchiveHub` 的「📜 日志」标签渲染日志列表

### 4.6 导出功能

- 新增 `exportSingle(data, format)` 函数：将传入数据序列化为 MD 或 CSV → Blob → 触发下载
- 各界面 [📥] 按钮调用 `exportSingle`
- 新增 `exportBatch(opts)` 函数：按 opts.sections 收集对应数据 → 合并 MD → Blob → 下载
- 批量导出面板：勾选 + 筛选表单 → 确认消费弹窗 → 扣 5 NT → `exportBatch`
- NT 消费确认弹窗做成通用组件，所有 NT 消费场景复用

## 六、方案自查

### 少即是多

社区池子就是一个 `data.community_pool` 对象，不需要建独立的模块或页面。社区档案就是一个数组 + 表单 + 核查按钮，不建 CMS。捐献就是一个输入框 + 确认。八个 UI 面板全部复用已有弹窗/面板模板。导出按钮分散在自然位置，不建导出管理中心。

### 复用优先

实景地图入口的渲染函数 `renderMapEntrance()` 被两个入口复用（村口 Modal + Hub 三级面板）。归档任务和往期营期的渲染复用 `renderArchiveList` 和 `getTaskDoneDate`。捐献弹窗和 NT 消费确认弹窗复用同一个通用弹窗组件。档案录入表单复用已有的弹窗模板。导出数据收集复用已有数据遍历逻辑。

### 需求对齐

| 需求 | 对应方案 |
|------|------|
| 社区池子多人管、需要批准 | `data.community_pool.managers` + `pending_approvals` + 审批面板 |
| 捐献放在实景地图入口 | `renderMapEntrance()` 内嵌社区捐献区 |
| 人工录入档案需要奖励 | 录入 5 NT、核查 3 NT，从社区池出 |
| 数据溯源 | 档案条目标 `entered_by` + `verified_by` |
| 系统数据不标来源 | 只有社区档案标来源，任务/审核/NT 流转不标 |
| 单个导出免费、批量收费 | 单个 [📥] 直接下载；批量 5 NT |
| 实景地图两个入口 | 村口 Modal + Hub 三级面板，复用 `renderMapEntrance()` |
| 邀请码 + 跳过 | 实景地图入口有邀请码输入 + 跳过按钮 |

没有超出需求范围的「顺手优化」。

### 全局最优

社区池子建好后，所有社区功能的 NT 进出都有了明确锚点。实景地图入口从两个地方进入、看到同一页面。档案室六个标签集成所有历史数据查看功能。导出按钮分散在数据所在位置，用户不需要离开当前视图去导出。运行日志覆盖所有关键操作，以后查问题不需要翻代码。

### 长期主义

社区池子审批机制可扩展为多签（2/3 同意即通过），数据结构已预留 `managers` 数组和 `pending_approvals` 队列。专线地址预留链上钱包对接。档案库可扩展类型（人物、事件、地点）。日志数组可加截断机制防止无限增长。导出格式支持 MD 和 CSV，加 PDF 只需加一个 format 分支。

**确认：方案符合五项原则，无发现需修正的问题。**
