## 一、营队数据全景

### 1.1 数据分类

一个营队由以下六类数据构成：

```
                          营队 (camp)
                             │
        ┌──────┬──────┬──────┼──────┬──────┐
        │      │      │      │      │      │
      身份    财政    时间    人事    任务    治理
    identity budget calendar team   tasks  governance
```

#### ① 身份数据 (identity) —「这个营队是谁」

| 字段 | 类型 | 说明 | 是否必须 |
|------|------|------|:---:|
| name | string | 营队名称，如「第四期南塘共创营」 | ✅ |
| period | string | 期数，如「第四期」 | ✅ |
| description | string | 主题描述，本期目标和理念 | ✅ |
| type | enum | `regular` 常规共创营 / `special` 特别活动营 | ✅ |
| status | enum | `draft` 草稿 / `active` 进行中 / `closed` 已结营 | ✅ |
| created_at | iso | 创营时间 | ✅ |
| created_by | string | 创营人 | ✅ |

#### ② 财政数据 (budget) —「钱从哪来、怎么花」

| 字段 | 类型 | 说明 |
|------|------|------|
| nt_total_pool | number | NT 总池（营队任务激励总额度） |
| nt_allocated | number | 已分配的 NT |
| nt_remaining | number | 剩余可用 NT（自动计算） |
| rmb_budget | number | 人民币预算总额 |
| rmb_items | array | 预算明细项 [{ name, amount, category, note }] |
| community_pool_total | number | 社区鼓励池总额（NPC/云村民打赏专用） |
| community_pool_daily | number | 每人每日打赏上限 |
| allocation_rules | object | 分配规则 { task_points_range, bonus_criteria } |

#### ③ 日历数据 (calendar) —「什么时间做什么」

| 字段 | 类型 | 说明 |
|------|------|------|
| start_date | date | 营期开始日 |
| end_date | date | 营期结束日（自动计算） |
| duration_days | number | 持续天数 |
| daily_schedule | array | 每日安排 [{ day, date, slots: [{ time, activity, note }] }] |
| milestones | array | 里程碑 [{ date, name, icon, type, deliverables }] |
| key_dates | array | 关键日期 [{ date, label, importance }] |

#### ④ 人事数据 (team) —「谁在营队里」

| 字段 | 类型 | 说明 |
|------|------|------|
| admin | string | 管理员 |
| staff_cards | array | 共建者卡片 [{ name, role, title, note, avatar_seed, task_names }] |
| members | object | 所有成员详情 { name: { role, title, bio, ... } } |

#### ⑤ 任务数据 (tasks) —「要做什么」

| 字段 | 类型 | 说明 |
|------|------|------|
| task_pool | array | 统一任务池 [{ name, category, points, type, scope, status }] |
| assignments | map | 分配关系 { builder_name: [task_names] } |
| templates | array | 任务模板 [{ name, tasks: [...] }] |

#### ⑥ 治理数据 (governance) —「怎么决策」

| 字段 | 类型 | 说明 |
|------|------|------|
| council_meetings | array | 议事厅会议 [{ topic, time, host, attendees, minutes }] |
| decisions | array | 集体决策记录 [{ title, date, votes, result }] |
| rules | object | 营队规则 { task_claim, review, reward, penalty } |

## 二、版本管理：原档不可篡改

### 2.1 设计原则

```
        创营（v1 原档）
            │
            │  锁定为只读，永久保存
            │
    ┌───────┴───────┐
    │               │
  [查看原档]    [当前版本 v3]
                    │
                    │  每次保存生成新快照
                    │
            ┌───────┼───────┐
            │       │       │
           v2      v3      v4 ...
         修改A   修改B   修改C
```

**核心规则：**
1. v1（创营原档）一经提交，**永久锁定不可修改**
2. 后续任何修改，生成新快照版本（v2, v3, ...）
3. 每个版本记录：**谁、什么时间、改了什么**
4. 当前工作版本 = 最新版本号
5. 可以查看任意历史版本（只读模式）

### 2.2 数据结构

```javascript
data.camp_info = {
  // ── 当前版本（可编辑）──
  current: {
    version: 3,
    updated_at: '2026-07-15T10:30:00',
    updated_by: '砚仁',
    
    // 六类数据都在这里
    identity: { name, period, description, type, ... },
    budget: { nt_total_pool, rmb_budget, ... },
    calendar: { start_date, duration_days, milestones, ... },
    team: { staff_cards: [...], members: {...} },
    tasks: { pool: [...], assignments: {...} },
    governance: { meetings: [...], decisions: [...] }
  },
  
  // ── 版本快照（只读，不可篡改）──
  snapshots: [
    {
      version: 1,
      created_at: '2026-07-10T09:00:00',
      created_by: '砚仁',
      label: '创营原档',
      frozen: true,  // ← 锁定标记，前端禁止修改此快照
      data: { identity: {...}, budget: {...}, ... }  // 完整快照
    },
    {
      version: 2,
      created_at: '2026-07-12T14:00:00',
      created_by: '砚仁',
      label: '调整预算和任务池',
      frozen: true,
      change_summary: 'NT总池从30000→50000；新增5项任务',
      data: { identity: {...}, budget: {...}, ... }
    },
    {
      version: 3,
      created_at: '2026-07-15T10:30:00',
      created_by: '砚仁',
      label: '确认共建者名单',
      frozen: true,
      change_summary: '新增3位共建者；分配任务包',
      data: { identity: {...}, budget: {...}, ... }
    }
  ],

  // ── 变更日志（每次修改的摘要，轻量索引）──
  changelog: [
    { version: 1, at: '...', by: '砚仁', action: 'create', summary: '创建营队' },
    { version: 2, at: '...', by: '砚仁', action: 'update', summary: '调整预算' },
    { version: 3, at: '...', by: '砚仁', action: 'update', summary: '确认名单' }
  ]
};
```

#### 3.1.3 火种的生命周期

火种不是一个静态图标，它贯穿整个营队从创建到启动的全过程。

```
🕯️ 未燃             🔥 初燃              🔥🔥 旺盛
  │                   │                    │
创营前              创营时               开营时
                 点燃火种            全员星光汇聚
                    │                    │
                    ▼                    ▼
            进入五步搭建           🏁 营队正式启动
```

| 阶段 | 火种状态 | 视觉表现 | 触发时机 |
|------|:---:|------|------|
| 创营前 | 🕯️ 未燃 | 一明一暗呼吸，微弱光晕 | Step ⓪ 表单 |
| 点燃瞬间 | 🔥 初燃 | 星点飞入，光芒扩散 | 管理员点击「点燃火种」 |
| 统筹完成 | 🔥 +1 | 火焰稍亮 | Step ① 保存 |
| 任务池完成 | 🔥 +2 | 火焰再亮一点 | Step ② 保存 |
| 选将完成 | 🔥 +3 | 添柴效果，噼啪火星 | Step ③ 确认认领 |
| 开营全员就绪 | 🔥🔥 旺盛 | 所有星汇入，大火熊熊 | Step ④-4c 就绪 |
| 营队运行中 | 🔥🔥 | 稳定燃烧 | 工作台顶部显示 |

### 3.6 仪式设计系统

#### 3.6.1 两个仪式 · 同一套语言

```
创营：点燃火种                      开营：星光汇聚

一人点亮                             三人汇聚
🕯️ → 🔥                            ☆ ☆ ☆ → 🔥🔥
「我要创建」                         「我们一起」
管理员独奏                           全体共建者合唱
火种诞生                             火种旺盛
```

| 维度 | 创营 · 点燃火种 | 开营 · 星光汇聚 |
|------|:--:|:--:|
| 谁触发 | 管理员 1 人 | 管理员发起，全员参与 |
| 参与者动作 | 点击「点燃火种」 | 每人点击「开启共创」 |
| 核心意象 | 火种从无到有 | 星光从散到聚 |
| 视觉高潮 | 星点飞入→火种第一次亮起 | 三颗星汇入→火种熊熊燃烧 |
| 色彩 | 暗色背景 + 暖橙金 | 暗色背景 + 暖橙金 + 白色星光 |
| 声音意象 | 火柴划燃、火焰升起 | 星光辉映、众人欢呼 |
| 结束时 | 进入五步搭建 | 回到工作台 |
| 不可逆 | 创营后不可删除 | 启动后原档锁定 |

#### 3.6.2 共享视觉元素

**火种**：
```
🕯️  未燃 ──── 一明一暗呼吸动画，微弱暖色光晕（~20px 半径）
🔥  初燃 ──── 稳定火焰，暖金色，小范围光扩散（~40px）
🔥  渐旺 ──── 同上但稍亮，增加微小火星粒子
🔥🔥 旺盛 ─── 大火燃烧，金色光波一圈圈扩散，火星飞溅
```

**星辰**：
```
☆  等待 ──── 灰白色，缓慢上下浮动，半透明
✨  就绪 ──── 亮白色金光，有光晕，微微旋转
★  汇聚 ──── 拖尾飞向中央，速度先慢后快
✦  背景星 ─── 极小，随机分布，缓慢漂移，偶尔闪烁
```

**背景**：
```
暗色底色（#0d0a06 或 var(--bg) 最深色）
散布 15-30 颗背景星 ✦
仪式高潮时背景星加速漂移
```

**光扩散**：
```
火种点燃/汇聚时：
  → 从火种中心发出暖金色光圈
  → 一圈一圈向外扩散（3-5圈）
  → 每圈半径递增，透明度递减
  → 类似水面涟漪，速度先快后慢
```

#### 3.6.4 仪式 UI 的技术要点

```
1. 全屏覆盖层：
   - position: fixed; inset: 0; z-index: 9999
   - background: 暗色渐变（中心稍亮，边缘接近全黑）
   - 禁用页面滚动

2. 动画：
   - 火种呼吸：CSS @keyframes，scale(0.9) ↔ scale(1.1)，周期 2s
   - 星光漂移：CSS translate + opacity，随机延迟，周期 8-12s
   - 光扩散：JS canvas 或 CSS radial-gradient + animation
   - 星光汇聚：CSS transition，从原位 fly to center，持续 1.5s

3. 状态同步：
   - 开营就绪状态通过数据层同步（camp_info.ceremony.ready_list）
   - 定期轮询或通过 saveData 事件触发刷新
   - 纯前端实现：所有人在同一浏览器/设备操作时
     通过 localStorage 事件或 data 对象共享状态

4. 可跳过（调试用）：
   - 测试模式下提供「跳过仪式」按钮
   - 正式模式下不可跳过
```

## 七、验收标准

### 仪式
1. ✅ 创营 → 点击「点燃火种」→ 全屏暗色 + 星点 + 火种呼吸 → 点燃动画 → 光扩散涟漪
2. ✅ 导航条 → 火种随步骤进度变化：🕯️ → 🔥 → 🔥渐亮 → 🔥添柴 → 🔥🔥
3. ✅ 开营仪式 → 管理员发起 → 所有共建者全屏进入星光汇聚
4. ✅ 每人点击「开启共创」→ 自己的星 ☆→✨ 亮起 → 按钮变灰 → 底部文案更新
5. ✅ 就绪计数 → 实时显示 0/3 → 1/3 → 2/3 → 3/3
6. ✅ 全员就绪 → 星光汇聚动画 → 火种熊熊 → 启动确认

### 流程
7. ✅ 管理员进入世界终端 → 看到 ⓪ 创营
8. ✅ 创营提交后 → 五步向导依次解锁
9. ✅ 启动前 → 数据自由修改，版本标记 draft
10. ✅ Step ④-4a → 选择审核人（不能是自己），提交审核
11. ✅ 审核人 → 收到通知 → 查看数据 → 通过/退回（附理由）
12. ✅ 审核通过 → 发包 → 共建者工作台显示营队 + 个人任务包
13. ✅ 全部就绪 → 营队启动 → 数据锁定为 v1 原档
14. ✅ 启动后 → 任务 draft → active

### 版本
15. ✅ 启动后修改 → 提交审核 → 通过后生成 v2, v3…
16. ✅ 版本历史 → 可查看任意版本（正式版本 + 筹备草稿）
17. ✅ v1 原档 → 🔒 只读，永久不可改
18. ✅ 测试营队 → 🧪 可删除；正式营队 → 不可删除

### 角色
19. ✅ Step ③ 选将 → 角色升级为共建者（所有人，不限于新 card）
20. ✅ 共建者 → 档案室从「云村民」移到「共建者」分组

## 八、UI 设计模式参考（研究补充）

> 基于 Google Docs、Figma、Notion、Asana、GitHub 等成熟产品的版本管理和向导设计模式。

### 8.1 修改方式：Task-Based UI（基于任务的界面）

**当前问题**：修改营队数据时，一个通用「编辑」表单打开所有字段，一次保存覆盖所有内容。无法追溯「谁改了哪个字段、为什么」。

**推荐模式**：用具体的**操作按钮**替代通用编辑。

```
❌ 当前方式：
  [编辑营队] → 全部字段的表单 → [保存]

✅ 推荐方式：
  每个数据区块有自己的操作按钮，每个按钮对应一个具体的「事件」：

  ┌─ 营队身份 ──────────────────────────────┐
  │  第四期南塘共创营                        │
  │  2026-07-15 ~ 2026-07-30               │
  │  [📋 2次修改]                           │  ← 点击展开修改历史
  └──────────────────────────────────────────┘

  ┌─ 财政 ──────────────────────────────────┐
  │  NT总池：50,000    已分配：35,000 (70%) │
  │  ┌─ 近期修改 ──────────────────────┐    │
  │  │ 07/12 砚仁：NT总池 30k→50k      │    │
  │  │ 07/10 砚仁：创建                │    │
  │  └──────────────────────────────────┘    │
  │  [调整预算] [添加支出项]                 │  ← 操作按钮
  └──────────────────────────────────────────┘
```

**规则**：
- 按钮用动词标签：`调整日期`、`修正预算`、`分配成员`，不用通用「编辑」
- 每次操作产生一条**不可变事件**，写入 changelog
- 旧值永久保留，新值叠加
- 修正不是改旧数据，是追加一条「修正记录」

## 九、关键设计决策总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 修改方式 | Task-Based UI（操作按钮） | 每次修改生成不可变事件，天然审计追踪 |
| 版本存储 | 完整快照（不是 diff） | 简单可靠，恢复快捷；数据量小（每条几 KB） |
| 原档保护 | v1 永久锁定 | 创营是一锤定音，后续只能追加 |
| 测试删除 | 仅 🧪 测试营队可删 | 防止误删正式数据 |
| 向导步骤 | 5 步（⓪①②③④），水平导航 | 步骤少、标签短，水平布局最清晰 |
| 版本对比 | 并排 + 统一视图切换 | 适应不同屏幕和数据量 |
| 财务展示 | 环形图 + 进度条 | 直观感知分配比例和剩余额度 |
