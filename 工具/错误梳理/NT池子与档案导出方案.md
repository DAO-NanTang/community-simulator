## 〇、核心概念

### 系统数据 ≠ 社区档案

| | 系统数据 | 社区档案 |
|------|------|------|
| 是什么 | 平台日常运转自动产生的记录 | 人工收集、整理、录入的历史人物资料 |
| 举例 | 任务发布、审核通过、NT 流转、打赏记录 | 往期参与者的故事、未数字化的人物档案、历史贡献 |
| 谁产生 | **系统自动** | **人手工录入** |
| 需要标来源吗 | 不需要 | **需要**——标明是谁录入的 |
| 录入有奖励吗 | — | **有**——费人工，社区池子出 NT 奖励 |

## 二、社区池子

### 2.1 钱的来源

| 来源 | 说明 |
|------|------|
| 🏦 Wizard 终端初始设定 | 管理员在建营时设定初始金额 |
| 🎁 **个人捐献** | 任何人可以捐献 NT 给社区池子 |
| 📦 批量导出费 | 用户批量导出时支付的 NT |
| 🔮 未来：实景地图收入 | 地图标注、地点解锁等社区功能收费 |

### 2.2 钱的用途

| 用途 | 说明 |
|------|------|
| 📝 **奖励档案录入者** | 手动录入社区档案（人物故事、历史资料）→ 社区池奖励 NT |
| ✅ **奖励档案核查者** | 录入的档案需要人核查真实性 → 核查者获得 NT 奖励 |
| 🎉 社区活动 | 未来：活动经费 |
| 🗺️ 实景地图 | 未来：地图功能维护 |

社区池子有一个**专线 NT 地址**。捐献和批量导出费打到这个地址。未来对接现实钱包时，这个地址就是链上对应的账户。

```

## 三、社区档案库

### 3.1 是什么

一个独立于系统数据的人工档案库。存放**不能由系统自动生成**的信息——往期参与者的故事、历史贡献记录、未数字化的资料。这些需要人花时间去收集、整理、录入。

### 3.2 档案条目结构

```
data.community_archives = [
  {
    id: "arc_001",
    type: "person",           // 人物档案
    title: "张老师 · 第三期在地画家",
    content: "张老师，福建南平人，第三期在地艺术家。负责工笔画教学工作坊…",
    tags: ["第三期", "在地艺术家", "工笔画"],
    entered_by: "砚仁",       // 谁录入的
    verified_by: "朝林",      // 谁核查的
    entered_at: "2026-07-10",
    reward_entered: 5,        // 录入奖励 NT
    reward_verified: 3,       // 核查奖励 NT
  },
  ...
]
```

### 3.3 录入流程

```
任何人 → 打开档案室 → 「📝 新建档案」
  → 填写标题、内容、标签
  → 提交 → 标记为「待核查」
  → 核查者（社区池管理员）审核内容真实性
  → 确认通过 → 录入者获得 5 NT ← 社区池出
  → 核查者获得 3 NT ← 社区池出
  → 档案永久入库
```

### 3.4 系统数据和档案信息的区别（再次明确）

| | 系统数据 | 档案信息 |
|------|:--:|:--:|
| 任务发布 | ✅ 系统自动 | — |
| 审核通过 | ✅ 系统自动 | — |
| NT 流转 | ✅ 系统自动 | — |
| 打赏记录 | ✅ 系统自动 | — |
| 往期参与者故事 | — | 📝 人工录入 |
| 人物历史贡献 | — | 📝 人工录入 |
| 未数字化资料 | — | 📝 人工录入 |

**系统数据不需要人来标来源。只有档案信息才标录入者和核查者。**

### 3.5 档案室新增标签

```
📚 档案室
  [📂成员] [📋归档任务] [📦往期营期] [📜日志] [📝社区档案] [📊结算]
```

「📝 社区档案」标签 = 浏览所有人工录入的档案条目，支持按标签筛选。

## 四、数据溯源（修正）

### 4.1 只溯源档案信息

系统产生的数据不需要标「谁录入的」——它就是系统产生的。

需要溯源的是**社区档案库里人工录入的条目**。每条记录：

- `entered_by`：谁录入的
- `verified_by`：谁核查的
- `entered_at`：录入时间
- `reward_entered`：录入奖励
- `reward_verified`：核查奖励

### 4.2 出问题找谁

档案信息不真实 → 找录入者和核查者两人。录入者承担主要责任，核查者承担审核失察责任。两次违规取消录入/核查资格。

### 4.3 池子管理员操作记录

池子管理员花钱（批准支出、发放奖励）的操作仍然记入 `data.community_pool.log`：

```
时间         操作者   类型      金额   批准者   说明
07-10 15:00  砚仁    批准支出   -50   朝林    购买实景地图素材
07-10 14:00  系统    奖励录入   -5     —      奖励·小明·录入档案
07-10 14:00  系统    奖励核查   -3     —      奖励·朝林·核查档案
07-09 10:00  宏心    捐献       +20    —      个人捐献
```

系统自动的标「系统」，人工的标操作者+批准者。

```

  → 全屏居中，适合访客首次进入

  → 三级滑出面板（复用 hub-sub-panel 样式）

**两个入口的容器差异**：

| | 村口入口 | Hub 入口 |
|------|------|------|
| 触发 | 村口卡片「实景地图」 | Hub 菜单标签「🗺️ 实景地图」 |
| 容器 | `#villageModalMap`（一级 Modal） | 三级滑出面板（复用 hub-sub-panel） |
| 关闭方式 | 点遮罩 / ✕ | 点遮罩 / ✕ / 点 Hub 其他标签 |
| 内容渲染 | `renderMapEntrance()` → 填入 `.vm-inner` | `renderMapEntrance()` → 填入面板 body |

**实现**：
- 改造 `#villageModalMap`：从占位符「敬请期待」→ 调用 `renderMapEntrance()`
- Hub 菜单标签「🗺️ 实景地图」从灰色改为可点击，调用 `openHubSub('map')` → 渲染 `renderMapEntrance()`
- `renderMapEntrance()` 是一个纯函数，返回 HTML 字符串，不含容器逻辑

## 十一、自查

- **少即是多**：社区档案就是一个数组 + 录入表单 + 核查按钮。不建复杂的 CMS。捐献就是一个输入框 + 确认。八个 UI 面板全部是已有弹窗模板的复用。
- **复用优先**：NT 消费确认弹窗做成通用组件（§8.8），所有场景复用。档案室标签结构复用已有档案室框架。导出面板复用批量导出已有面板样式。
- **需求对齐**：实景地图两个入口（村口+Hub）、捐献放实景地图访客页（你指定的）、邀请码+跳过流程、档案录入奖励（费人工）、专线地址（对接现实钱包）、系统数据不标来源。
- **全局最优**：社区池子 = 社区功能 NT 进出锚点。社区档案 = 人工历史资料库。六个标签统一在档案室下，不散落。
- **长期主义**：专线地址预留链上对接。捐献记录可导出为财务报告。档案库可扩展类型。UI 面板全部独立函数，后续改样式只改一处。
