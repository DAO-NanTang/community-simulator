### 问题 1：社区窗口独立弹窗与社区副本割裂

**代码证据**：
- 村口 plaza 第 4 张卡片「社区窗口」（已删除，原位于 `villageModalActivity` 附近）通过 `onclick="openCommunityWindow()"` 打开独立 Modal
- `openCommunityWindow()` 函数（已删除，原定义于 `openVillageModal`/`closeVillageModal` 之后）调用 `openVillageModal('community')`
- `openVillageModal('community')` 分支（已删除）将 `renderCommunityWindow()` 的返回值填入 `#communityWindowContent`
- 独立 Modal `#villageModalCommunity`（已删除）包含 `#communityWindowContent` 容器
- 村口第 1 张卡片「社区副本」通过 `onclick="openVillageModal('activity')"` 打开 `#villageModalActivity`，由 `renderVillageActivityList()` 渲染活动列表

**问题逻辑链**：
1. 访客进入村口，看到 4 张卡片
2. 第 1 张是「社区副本」——展示活动列表 + 邀请码 + 成为冒险者按钮
3. 第 4 张是「社区窗口」——展示营地动态 + 打赏面板
4. 两张卡片指向同一个营地的不同信息，但入口分开
5. 访客需要分别点开才能了解营地全貌
6. 社区副本里有「成为冒险者」按钮，但看不到社区动态，缺少加入动力

**修复**：
- 删除第 4 张卡片 + `#villageModalCommunity` Modal + `openCommunityWindow()` 函数
- 删除 `openVillageModal`/`closeVillageModal` 的 `'community'` 分支
- 在 `renderVillageActivityList()` 的「成为冒险者」/「进入营队」按钮旁边加「👀 浏览营地」按钮

**修复后逻辑**：
- 村口 3 张卡片（减少 1 张）
- 点「社区副本」→ 看到活动列表
- 点「浏览营地」→ 二级面板从右侧滑出，显示社区动态
- 一张 Modal 承载营地信息 + 社区动态 + 加入入口

### 问题 2：社区副本 Modal 垂直位置靠上

**代码证据**：
- CSS 第 130 行：`.village-modal { align-items: flex-start; padding-top: 40px; }`
- `align-items: flex-start` 将内容对齐到容器顶部
- `padding-top: 40px` 从顶部向下偏移 40px

**问题逻辑链**：
1. `.village-modal` 是 `position: fixed` 的 flex 容器，覆盖整个视口
2. `align-items: flex-start` 使子元素（`.vm-inner`）对齐到容器顶部
3. 社区副本 Modal 只有三张活动卡片，内容较少
4. 内容贴近屏幕顶部，视觉上偏上

**修复**：
- CSS 改为 `.village-modal { align-items: center; padding-top: 0; }`
- 子元素垂直居中

**修复后逻辑**：
- 所有 village modal（社区副本、任务大厅、地图）垂直居中
- 内容少时不偏上，内容多时可滚动

### 问题 3：二级面板打开后双面板重心偏右

**代码证据**（修复前 `_repositionActivityCommunity` 函数）：
- `inner.style.maxWidth = '440px'` — 一级收窄
- `panel.style.left = (ir.right + gap) + 'px'` — 二级贴在一级右侧
- 未做水平居中偏移

**问题逻辑链**：
1. 一级 Modal 由 flex `justify-content: center` 居中
2. 二级面板通过 `position: fixed` 贴在 Modal 右侧
3. 双面板的几何中心 = 一级中心 + 二级中心的一半，落在屏幕中心右侧
4. 视觉重心偏右

**修复**（`_repositionActivityCommunity` 函数，当前版本）：
- 先用 `transform: translateX()` 将一级 Modal 向左平移 `(二级面板宽度 + 间距) / 2`
- `transform` 不影响 flex 居中基准——flex 仍按一级原始位置居中，`translateX` 是纯视觉平移
- 二级面板的 `top` 和 `height` 直接从一级的 `getBoundingClientRect()` 读取——保证上下边缘精确对齐
- 关闭时恢复 `transform = ''`

**修复后逻辑**：
- 一级先被 flex 居中，然后视觉上左移半个二级面板宽度
- 二级贴在一级右侧
- 双面板作为一个整体，几何中心 = 屏幕中心

### 问题 4：选人下拉面板被二级窗口遮挡

**代码证据**：
- `.picker-dropdown` CSS 第 173 行：`z-index: 200`
- `#activityBackdrop` 行 1180：`z-index: 1001`
- `#activityCommunitySlide` 行 1181：`z-index: 1002`
- `.picker-dropdown` 的 z-index (200) < 二级面板 (1002) < 遮罩 (1001)

**问题逻辑链**：
1. 用户在社区窗口的打赏面板点击「选择成员」
2. `showReviewerPicker('activityTipTarget')` 打开 `.picker-dropdown`
3. `.picker-dropdown` 的 `z-index: 200`
4. 二级面板 `#activityCommunitySlide` 的 `z-index: 1002`
5. 选人面板被二级面板遮挡，用户看不到

**修复**：
- `.picker-dropdown` 的 `z-index` 从 `200` 改为 `1010`

**修复后逻辑**：
- 选人面板 z-index 1010 > 二级面板 1002 > 遮罩 1001 > 一级 Modal 120
- 层级从高到低：选人 > 二级 > 遮罩 > 一级

### 问题 5：多级窗口关闭顺序不正确

**代码证据**（修复前）：
- `activityBackdrop` 的 `onclick="closeActivityCommunity()"` — 直接关闭二级
- `villageModalActivity` 的 `onclick="closeVillageModal('activity')"` — 直接关闭一级
- 关闭函数内部未检查是否有更上层窗口打开

**问题逻辑链**：
1. 用户打开社区窗口（二级），再打开选人面板（三级）
2. 点击遮罩 → `closeActivityCommunity()` 直接关闭二级 → 三级一同消失（DOM 仍存在但不可见）
3. 或者：一级 Modal 打开 + 二级社区窗口打开 → 点击 Modal 背景 → `closeVillageModal('activity')` 直接关闭一级 → 二级一同消失

**修复**：
- `closeActivityCommunity()` 开头检查：如果 `.picker-dropdown` 可见（三级开着），则先关闭三级，`return`，不关二级
- `closeVillageModal()` 开头检查：如果 `#activityCommunitySlide` 可见（二级开着），则调用 `closeActivityCommunity()` 关闭二级，`return`，不关一级

**修复后逻辑**：
- 点遮罩/✕ → 逐级关闭：三级先关 → 二级再关 → 一级最后关
- 参考现有 `hubSubPanel` + `dropdown-menu` 的级联关闭模式

### 问题 6：打赏无确认、无反馈、无实时刷新

**代码证据**（修复前，`renderCommunityWindowEmbedded` 内联 onclick）：
- 免费鼓励：`sendTip(...)` — 直接发送，无确认，无结果提示
- 付费礼物：`var r=sendTip(...);if(!r.ok)alert(r.reason)` — 仅失败时弹 alert，成功无反馈
- 发送后未重新渲染面板——余额显示不更新

**问题逻辑链**：
1. 用户点击打赏按钮
2. 免费鼓励直接发送，用户不知道成功与否
3. 付费礼物仅在失败时有提示
4. 面板余额是发送前的快照，发送后不会刷新
5. 用户无法确认自己的打赏有没有生效

**修复**：
- 付费礼物：`confirm('确定送 xxx？')` → 确认后才发送
- 发送成功后：设置 `window._lastTipResult = { ok: true, msg: '✅ 已送…' }` → 面板重渲染时在顶部显示绿色结果条
- 发送失败后：设置 `window._lastTipResult = { ok: false, msg: '❌ …' }` → 面板重渲染时显示红色结果条
- 每次发送后调用 `renderCommunityWindowEmbedded()` 刷新整个面板

**修复后逻辑**：
1. 用户选人、点打赏按钮
2. 付费礼物弹出确认框
3. 发送 → 结果存入全局变量
4. 面板刷新 → 顶部显示结果条（绿色成功 / 红色失败）
5. 余额数字实时更新
6. 结果条在面板内持久显示（不随时间消失）

### 问题 7：打赏不进信箱，收发双方没有通知

**代码证据**：
- `sendTip` 函数（行 7722-7766）：写入 `data.tips` 和 `data.finance`（付费时），但没有任何入信箱逻辑
- `computeInboxMessages` 函数（行 3216-3376）：遍历 `data.tasks`、`data.finance` 等生成消息，不包含对 `data.tips` 的扫描
- 信箱消息类型 `TYPE_ICON` 和 `TYPE_LABEL`（行 3398-3409）：18 种已定义类型，不含 `tip_received`

**问题逻辑链**：
1. A 在社区窗口打赏 B
2. `sendTip` 写入 `data.tips` 和 `data.finance`
3. B 打开信箱 → `computeInboxMessages` 不扫描 `data.tips`
4. B 看不到任何打赏通知
5. B 的 NT 余额可能因打赏而增加，但不知道是谁送的

**修复方案**（未执行）：
- `sendTip` 不直接写信箱消息（信箱消息是动态计算的，不是持久存储的）
- 在 `computeInboxMessages` 中新增一段：扫描 `data.tips`，筛选 `tp.to === currentUser.name` 的记录，为每条生成 `tip_received` 消息
- 消息 ID 格式：`tip_` + tip 对象在数组中的索引（与现有消息 ID 格式一致，支持已读/未读追踪）

**修复后逻辑**：
1. A 打赏 B → `sendTip` 写入 `data.tips`
2. B 打开信箱 → `computeInboxMessages` 扫描 `data.tips` → 找到发给 B 的打赏
3. 生成 `tip_received` 消息 → 显示在信箱「💬 打赏」分类下
4. B 看到「A 送了 🌹小花 · 1NT」

## 三、解决方案效果（纯自然语言描述）

### 3.1 社区窗口入口

访客进入村口，看到 3 张卡片。点击「社区副本」，在一个窗口中看到营地活动列表。点击「浏览营地」按钮，右侧滑出一个面板，展示营地的实时动态——谁在创作、有什么任务在招募、最近的打赏记录。不需要跳转、不需要关掉当前窗口。

### 3.2 视觉表现

社区副本窗口在屏幕上垂直居中。打开浏览营地后，两个面板作为整体居中于屏幕，上下边缘精确对齐，宽度比例合理（活动信息较窄，社区动态较宽）。关闭二级面板后，一级窗口恢复原始宽度和位置。

### 3.3 打赏流程

用户在社区动态面板中选择要送给谁，从成员列表中点击选中。选好后，点击鼓励按钮（免费）直接发送；点击礼物按钮（付费）会弹出确认框，确认后发送。发送结果在面板顶部以绿色（成功）或红色（失败）文字显示，余额数字同步更新。自己发送的打赏记录在「最近打赏」中以高亮标记。

### 3.4 窗口关闭逻辑

如果选人面板打开着，点击背景遮罩会先关掉选人面板，再点才会关社区动态面板，再点才会关营地活动窗口。不会出现点了遮罩全部消失、或者想关一个结果关了另一个的情况。

### 3.5 信箱通知

收到打赏的人打开信箱，在「💬 打赏」分类下看到谁在什么时候送了什么。信箱顶部有分类标签（任务、审批、打赏、财务、活动），可以按类型筛选消息，不再需要在混杂的消息列表中翻找。

## 四、解决方案效果（代码逻辑描述）

### 4.1 社区窗口入口

- 删除：`openCommunityWindow()` 函数、`#villageModalCommunity` DOM 元素、`openVillageModal`/`closeVillageModal` 的 `'community'` 分支
- 新增：`renderVillageActivityList()` 中，成员和访客分支各加一个 `onclick="openActivityCommunity()"` 的按钮
- 新增：`#activityBackdrop` 遮罩层 + `#activityCommunitySlide` 滑出面板（复用 `.hub-sub-panel` CSS 类）

### 4.2 视觉居中与对齐

- CSS：`.village-modal` 改为 `align-items: center; padding-top: 0`
- `_repositionActivityCommunity()` 函数：先设一级 `maxWidth: 380px` + `minHeight: availH` + `transform: translateX(shift)`，然后读 `getBoundingClientRect()` 取一级实际位置，用 `panel.style.top = ir.top` 和 `panel.style.height = ir.bottom - ir.top` 将二级固定在同一位置
- 关闭时 `closeActivityCommunity()` 清除一级的 `maxWidth`、`transform`、`minHeight`

### 4.3 打赏确认与反馈

- 付费礼物 onclick：`confirm()` 确认 → `sendTip()` → 结果写入 `window._lastTipResult` → `renderCommunityWindowEmbedded()` 刷新
- 面板顶部：读取 `window._lastTipResult`，渲染带颜色的结果提示条
- 免费鼓励 onclick：同上但不弹 confirm

### 4.4 窗口层级

- `.picker-dropdown` z-index: 1010
- `#activityCommunitySlide` z-index: 1002
- `#activityBackdrop` z-index: 1001
- `.village-modal` z-index: 120

### 4.5 级联关闭

- `closeActivityCommunity()`：检查 `#pickerDropdown` 是否可见 → 是则调 `closeReviewerPicker()` 并 `return`
- `closeVillageModal()`：检查 `#activityCommunitySlide` 是否可见 → 是则调 `closeActivityCommunity()` 并 `return`

### 4.6 信箱打赏通知（方案，未执行）

- `sendTip` 函数不变（不直接写信箱）
- `computeInboxMessages` 新增一段：遍历 `data.tips`，筛选 `tp.to === currentUser.name`，为每条生成 `{ type: 'tip_received', category: 'tip', ... }` 消息
- `renderInbox` 新增分类变量 `inboxCategory`，消息列表按 `m.category === inboxCategory` 过滤
- 5 个分类标签按钮切换 `inboxCategory`
