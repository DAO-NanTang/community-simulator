# 《PHASE2_整体方案_v2.md》审查报告

> 审查对象：[PHASE2_整体方案_v2.md](PHASE2_整体方案_v2.md)  
> 对照文件：dashboard.html（5,677 行，已含 Phase 1 部分实现）  
> 审查日期：2026-07-06

---

## 总体判断

**这份方案做对了一件很难的事：身份模型的逻辑是自洽的。五个场景走下来没有矛盾，升级路径清楚，每种身份该看到什么、能做什么都说清楚了。这是方案最值钱的部分。**

但方案把三件不同的事捆在了一起：统一登录路由、多世界架构、南塘村口。这三件事的复杂度不在一个量级上，混在一起会导致实施范围失控。另外，方案没有面对一个现实问题——要把单 `data` 全局变量改造成多世界架构，现有代码里 100+ 处数据访问和 25+ 处 `localStorage` 写入全部要改。这不止是"加新功能"，是"动地基"。

---

## 🟢 方案做对了什么

### 身份模型（§一、§二）——这是整份方案最扎实的部分

五个身份的来路和去向都清楚：

- **管理员**——预置，不开放注册。这是对的。你不需要在登录页放一个"我是管理员"的按钮。
- **共建者**——不能直接注册，由管理员在 Step3 认领后转化。这也是对的。共建者意味着承担运营责任，不应该自封。
- **冒险者**——凭世界码注册。码是邀请，不是锁。
- **南塘NPC**——直接注册或游客升级。和游客的区别是"在地常驻" vs "临时来访"，这个区分在实际运营场景中有意义。
- **游客**——无门槛。让任何人可以先逛逛，降低了参与门槛。

**"有码就能升级，码是邀请不是锁"**——这条设计原则值得写大一点。它让整个身份系统不是冷冰冰的权限框架，而是有社交温度的。

### 场景走查（§八）——逻辑完整

五个场景覆盖了从管理员到游客的所有路径，包括小红的"游客→NPC→共建者"三连跳和朝林的"NPC→冒险者→共建者"三身份叠加。走下来没有矛盾，没有死循环，没有"这个人在这一步不知道该看到什么"的空白。

### 简化登录页（§一）——比 Phase 1 的方案更好

Phase 1 的登录页设计比较重（入口 + 注册表单 + 用户列表 + 密码弹窗四层嵌套）。Phase 2 简化为"我有账号"和"我是新来的"两个入口——更轻，直觉更强。

## 🔴 需要重新审视的问题

### 问题 1：方案的"第二阶段"边界太大了

方案 §九 列了四个子阶段：

| 子阶段 | 内容 | 实际改动量 |
|--------|------|-----------|
| 2A 统一登录+路由 | 登录页简化、注册身份选择、路由逻辑 | 中等——Phase 1 已有登录框架，主要是改造 |
| 2B 世界管理器 | `loadWorld`/`switchWorld`/`createWorld`、数据迁移 | **巨大**——触及整个应用的数据层 |
| 2C 南塘村口 | 实景地图、日常委托、P2P流水、住宿、管理员面板 | **巨大**——几乎是一个独立应用 |
| 2D 多视角 | 不同身份看不同 tab | 中等——Phase 1 已有双面板，主要是扩展 |

**2B 和 2C 不应该捆在一起。** 2B 是基础设施（改了所有函数怎么读数据），2C 是新产品（村口是一个全新的界面+数据模型+交互流程）。把它们分开，先通基础设施，再加新空间。

### 问题 2：多世界架构需要动地基，方案没算这笔账

当前 dashboard.html 里，`data` 是一个全局变量，整个应用所有函数直接读 `data.tasks`、`data.members` 等等。这包括：

- **100+ 处** `data.xxx` 的直接读写
- **25+ 处** `localStorage.setItem('camp_data', ...)` 的持久化调用
- **`updateAll()` 函数**（第 1552 行）——一口气调 8 个渲染函数，每个都依赖全局 `data`
- **`loadData()` 函数**（第 1446 行）——逐字段解析 JSON → 写入 `data` → 调 `updateAll()`
- **`exportData()` 函数**（第 1499 行）——把 `data` 对象序列化导出

改成多世界架构后，这些全都要变：
- 全局 `data` → `getCurrentWorld()` 或 `worldData`
- `localStorage.setItem('camp_data', ...)` → `saveWorld(currentWorldId)`
- `loadData()` → 需要支持"导入到哪个世界"、"导入的是世界数据还是村口数据"
- `exportData()` → 需要支持"导出当前世界"还是"导出全部世界"
- `updateAll()` → 每个渲染函数都要知道自己该渲染哪个世界的内容

**方案 §七 画了数据结构，但没有说这 100+ 处代码访问怎么改。** 这不是小修小补，是需要一条一条追踪替换的工作量。建议方案至少列一个"需要改动的关键函数清单"，让实施者心里有数。

### 问题 3：南塘村口是一个独立产品，不应塞在"第二阶段"

方案 §五 描述的南塘村口包含了：

- 实景地图（底图 + 坐标标注 + 点击弹出面板）
- 日常委托板（独立于营期任务的委托系统）
- P2P 流水（from/to/confirmed_by，完全不同于现有的 `finance_cny` 结构）
- 住宿登记（全新数据模型）
- 总排行榜
- 村口管理员面板

这些功能从数据模型到交互逻辑到渲染方式都是全新的，和现有的 dashboard 代码几乎没有复用关系。它不是"改造现有功能"，是"新建一个空间"。

**建议**：把南塘村口拆成 Phase 3。Phase 2 集中精力把登录路由和多世界架构打通——做到"管理员登录后能看到多个世界的切换栏，切到不同世界能看到对应的数据"——这就已经是很扎实的交付了。

### 问题 4：方案没有论述和 Phase 1 的关系

Phase 1 已经部分实施——代码里有 `getUsers()`、`saveUsers()`、`NT_SESSION_KEY`、登录页框架、`builderTabs`/`adventurerTabs` 双面板、`previewMode` 切换。

Phase 2 的方案：
- §一重新设计了登录页（比 Phase 1 更简洁）
- §二引入了五种身份（Phase 1 只有 builder 和 adventurer 两种）
- §四重新设计了 tab 配置（Phase 1 已有 builder/adventurer 两套 tab）

**这些是"在 Phase 1 基础上修改"还是"推倒 Phase 1 重来"？** 方案没有说明。如果是前者，需要指明"Phase 1 的哪些代码保留不动，哪些需要改造"；如果是后者，需要诚实标注"这个方案会覆盖 Phase 1 的部分实现"。

比如，Phase 1 的 `switchTab` 已经处理了 builder/adventurer 两套面板的权限门禁。Phase 2 新增了 NPC 和游客身份——这两种身份能不能看到 Phase 1 里已经建好的双面板？还是它们只进村口？方案里 NPC 和游客"无切换栏（只有村口）"，那他们根本不会进到 Phase 1 建的那些 tab 页面。这个边界需要交代清楚。

## 🟡 具体设计细节的问题

### 问题 5：P2P 流水的"确认交付"模型不完整

方案 §六和 §七定义了：
```js
finance_plaza: {
  transactions: [
    { from, to, amount, reason, status, task_id, confirmed_by, created }
  ]
}
```

管理员"确认交付" → NT 从委托方划给接单人。但这个模型缺少了几个关键状态：

- **委托方 NT 不够怎么办？** 如果委托方余额不足，接单人已经完成了委托但拿不到 NT——系统怎么处理？
- **接单人中途放弃怎么办？** 委托被接了但没完成，NT 是锁在委托方账户里还是从未划拨？
- **"已退回"状态**——方案的管理员面板有"退回"按钮，但交易结构里 `status` 字段的取值没有定义（是 `pending`/`confirmed`/`rejected` ？还是 `待确认`/`已完成`/`已退回`？）

### 问题 6：世界码的格式不一致

方案 §二用的示例是"1100"（纯数字），§七的 `nt_worlds_index` 也是 `["plaza", "1100", "5700"]`（纯数字字符串）。但 Phase 1 的邀请码格式是 `"NT-A3F8K2"`（带前缀的字母数字组合）。

**世界码和邀请码是不是同一套编码系统？** 如果是，世界码应该也是 `NT-XXXX` 格式。如果不是，那管理员需要同时管理两套码（世界码给冒险者、NPC邀请码给游客升级）。方案对此没有交代。

### 问题 7：五个身份但路由分叉只有三条路径

方案 §三 的登录后路由有五条分叉，但仔细看：

- 管理员 → 上次活跃世界（或村口）
- 共建者 → 被分配的世界（或村口）
- 冒险者 → 注册时的世界
- NPC → 村口
- 游客 → 村口

**NPC 和游客的路由是相同的（都进村口），区别只在于有没有世界切换栏。** 也就是说，核心路由逻辑实际上只有三条路径：管理员、有世界的人（共建者/冒险者）、没世界的人（NPC/游客）。方案可以把这个简化表达出来，减少实施时的困惑。

## 🔧 和现有代码的直接冲突点

### 冲突 1：`exportData()` 导出的是一整个 `data` 对象

如果要支持多世界，用户点"导出"时导什么？当前导的是完整的 `data`（包含所有任务、成员、财务、日程等）。在多世界架构下，每个世界的数据是独立的——导出当前世界还是导出全部世界？格式要不要变？方案没有讨论。

### 冲突 2：历史数据里没有"世界"概念

当前 `camp_data` 里存的是单期（第四期）的数据。方案说"首次运行自动迁移：`camp_data` → `nt_world_1100`"。但如果用户已经有第五期的备选数据（另一个 JSON 文件），导入时会怎样？是把第五期覆盖第四期，还是并存为两个世界？`loadData()` 当前是全覆盖合并，不支持"导入为另一个世界"。

### 冲突 3：Wizard 步骤生成的数据写到哪

当前的营地设置 Wizard（五步向导）所有步骤操作的都是同一个全局 `data` 对象。多世界架构下，Wizard 创建的内容应该写到当前激活的世界。这意味着 Wizard 的 `renderStep1()` ~ `renderStep5()`、`executeSync()`、`switchStep()` 等全部需要世界感知。方案没有提 Wizard 怎么适配。

## 📋 审查原则逐项评分

| 原则 | 评分 | 说明 |
|------|------|------|
| **少即是多** | 🔴 不通过 | 三件事捆在一起：登录路由 + 多世界架构 + 南塘村口。2C（村口）和 2B（世界管理器）的复杂度不在一个量级。建议拆成两个 Phase |
| **复用优先** | 🟡 勉强通过 | 数据结构上保留了现有 `data` 的内部字段（tasks, members, budget 等），但没说明 Phase 1 已经建好的登录框架、双面板、`switchTab` 怎么被复用还是覆盖 |
| **需求对齐** | 🟡 勉强通过 | 身份逻辑（§二）和登录路由（§三）的需求来源清楚——五类用户确实存在。但村口的 P2P 流水、住宿登记、总榜的需求来源没有交代——是用户提的还是方案自主设计的？ |
| **全局最优** | 🔴 不通过 | 多世界架构改动量巨大（100+ 处数据访问、25+ 处持久化、loadData/exportData/updateAll 全部重写），方案只画了数据结构蓝图，没有评估实施代价。Wizard 五步向导怎么适配多世界完全没提 |
| **长期主义** | 🟡 通过 | 身份模型可扩展（新身份可以加升级路径）；多世界架构支持无限期数；数据迁移保留旧数据。但 P2P 流水的状态机不完整，长期运营会遇到"委托方余额不足""接单人中途放弃"等未定义场景 |

## 修正建议

### 拆分 Phase

| 原方案 | 建议 |
|--------|------|
| Phase 2A（统一登录+路由） | → **Phase 2：统一身份 + 路由**。就做这个，把登录页简化、身份注册、登录后路由打通 |
| Phase 2B（世界管理器） | → **Phase 3：多世界架构**。这是基础设施，改了 100+ 处代码，需要独立时间窗口 |
| Phase 2C（南塘村口） | → **Phase 4：南塘村口**。这是全新产品，独立于前两个 Phase |
| Phase 2D（多视角） | → 合并到 Phase 3。多世界架构通了之后，多视角是自然而然的事 |

### 补充缺失的关键信息

1. **列出需要改动的关键函数清单**：`updateAll()`、`loadData()`、`exportData()`、`executeSync()`、Wizard 五个 `renderStep` 函数、`switchTab()`、所有带 `localStorage.setItem('camp_data'...)` 的函数
2. **说明和 Phase 1 已实现代码的关系**：哪些保留、哪些改造、哪些覆盖
3. **定义 P2P 流水状态机**：pending → accepted → completed → confirmed（或 rejected）
4. **统一世界码和邀请码的格式**
5. **说明 Wizard 在多世界架构下的行为**

## 最终结论

**身份模型可以直接拿去用，设计是扎实的。** 五个场景走查没有问题，升级路径自洽，每种身份的行为边界清楚。

**但整个方案需要拆。** 当前把它当"第二阶段"来规划，实际上包含了三个不同复杂度的工程：改登录路由（中）、改数据架构（大）、建南塘村口（大）。建议拆成三个独立的 Phase，先做登录路由，再做多世界架构，最后做村口。每一步都有明确的交付物，每一步都可以独立验证。

拆完之后，每份子方案的"少即是多"和"全局最优"自然就过了——因为每一步的范围都是可控的。
