## 🚨 致命问题：删除重复代码的方向反了

**位置**：§〇（第 10-30 行）

**方案说**：
> 删除第二组重复函数（行 ~2700-3510），只保留第一组

**实际情况**：第一组函数里，`recalcFinanceByPerson`（第 2004 行）缺少一个闭合括号 `}`。这导致跟在它后面的 `aggregateMaterialBudgets`、`getMilestoneTasks`、`isStaff2`、`showAddMember`、`renderMemberDetail`、`renderPlayers`、`renderPlayerDetail`、`renderTimeline`、`renderBudget`、`renderSettlementAll` 等十几个函数**全部被包在它里面，成了局部变量**，外界调用不到。

第二组（~2730-3510 行）才是括号正确、功能完整的版本。

**如果你按方案说的"删第二组，保留第一组"，你保留下来的恰恰是有 bug 的版本。** 公会资金池的统计会坏，NPC 详情页会坏，结算画面会坏，因为这些渲染函数本质上变成了私有函数。

**正确做法**：
- 方案 A：**删第一组有问题的代码块，保留第二组。**
- 方案 B：先修复第一组 `recalcFinanceByPerson` 的缺失括号，验证所有函数都能被全局访问到，再删第二组。

## 🔐 安全相关（不是 bug，但必须诚实标注）

### 问题 1：密码"哈希"其实是明文编码

**位置**：§1.1、§4.1（第 95-100 行、第 468-470 行）

```js
function hashPassword(password, username) {
  return btoa(encodeURIComponent(password + ':' + username));
}
```

`btoa` 是编码，不是哈希。编码和哈希的区别在于：编码是可逆的。任何能看到 localStorage 的人，在控制台输入：

```js
decodeURIComponent(atob(编码后的字符串))
```

就能拿到 `密码:用户名` 的原始值。

对于一个本地单文件 HTML 应用，用 localStorage 存密码本来就没有真正的安全性——这不丢人，但**不应该把这个函数叫 `hashPassword`**，这个命名会给后续维护者一个虚假的安全感。

**建议**：
1. 函数改名 `encodePassword` 或 `obfuscatePassword`
2. 函数上方加一行注释：`// 简单编码，防止路过式窥屏，不防恶意篡改`
3. 文档的风险表里加一行："密码存储为可逆编码，而非加密哈希"

### 问题 2：登录态可以手动伪造

**位置**：§1.2、§4.2（第 110-111 行、第 757-764 行）

`nt_session` 里存的是 `{ name, role }`，没有 token，没有过期时间，没有任何服务器验证。任何人打开浏览器控制台，写一行：

```js
localStorage.setItem('nt_session', '{"name":"砚仁","role":"builder"}')
```

刷新页面就直接以共建者身份进入了。

同样，对于一个本地单文件应用，**这是可接受的限制，不是设计失误**。但要诚实地写出来。建议在方案里加一段：

> 本阶段的"认证"本质是「本地分身份使用」，不是真正的安全认证。它解决的是"多人共用一台电脑时，互相不误看对方的界面"，而不是"防止恶意访问"。真正的访问控制需要后端服务器。在此之前，不要用本系统存储真正的敏感数据。

## ⚠️ 代码级问题（实施时会报错或逻辑出错）

### 问题 3：`isCurrentUserAssignee` 没有空值保护

**位置**：§4.4（第 910-915 行）

```js
function isCurrentUserAssignee(assignee) {
  if (!assignee) return false;
  return (assignee || '').split(/[,，、]/).map(...).includes(currentUser.name);
}
```

如果 `currentUser` 是 `null`（用户还没登录、会话数据损坏），`currentUser.name` 会直接报错 `Cannot read properties of null`，**整个页面崩溃白屏**。

**修正**：在函数开头加一行：
```js
if (!currentUser) return false;
```

### 问题 4：任务卡片在预览模式下权限判断错了

**位置**：§5.1（第 1091 行）

```js
onclick="dailyCheckTask(' + index + ', \'' +
  (currentUser && currentUser.role === 'builder' ? 'staff' : 'player') + '\')"
```

当共建者点击「切到冒险者视角」（此时 `previewMode = true`，但 `currentUser.role` 仍然是 `'builder'`），这行代码会以 `'staff'` 身份调用 `dailyCheckTask`。也就是说：**共建者在预览模式中仍然拥有管理所有任务的权限**——这和预览的设计目的（"看看冒险者看到什么"）矛盾。

**修正**：用方案自己定义的 `isAdventurerView()` 来判断：
```js
(isAdventurerView() ? 'player' : 'staff')
```

### 问题 5：用户迁移的幂等保护太"一刀切"

**位置**：§4.1（第 483-485 行）

```js
function migrateExistingMembers() {
  const users = getUsers();
  if (Object.keys(users).length > 0) return; // 已迁移，跳过
```

这行的意思是：**只要 `nt_users` 里有哪怕 1 个人，迁移就永久不再执行。** 但如果手动注册了 1 个人之后，`data.members` 里又新增了 5 个人——这 5 人的账号永远不会被创建。

**修正**：不要把门完全关死，改成"只迁移尚未在 `nt_users` 里出现的人"：
```js
Object.values(data.members || {}).forEach(function(m) {
  if (!users[m.name]) {
    users[m.name] = { ... }; // 创建账号
  }
});
```

### 问题 6：共建者名单匹配仍然用了子串模糊匹配

**位置**：§4.8（第 1041 行）

```js
!staffNames.some(function(s) {
  return m.name === s || m.name.indexOf(s) !== -1;
})
```

这和之前审查 `修复方案.md` 时发现的 `getPersonMaterials` 的 `indexOf` 模糊匹配是**同一个问题**。如果共建者名单里有"红"，普通冒险者"小红"会被误判为共建者，从而在冒险者计数中被排除。

方案自己在 §4.4 里专门写了 `isCurrentUserAssignee` 用精确匹配来避免这个问题，但 `updateWorldHUD` 又用了 `indexOf`。**同一份方案出现了自相矛盾的做法。**

**修正**：统一用精确匹配（逗号/顿号分隔后再逐项比对）。

## 🔧 设计层面的问题（不会报错，但可以做得更好）

### 问题 7：CSS 变量命名和现有代码不一致

**位置**：§2.1（第 131-181 行）

方案定义了 `--card-bg`，但现有 dashboard.html 里用的变量名是 `--card`（第 8 行）。如果你在替换 `#fffdf9 → var(--card-bg)` 的同时，现有代码里有 `.card { background: var(--card) }`——你需要把旧变量名全部批量替换成新的，或者反过来让新变量名对齐旧的。

**建议**：沿用现有的 `--card` 而非新建 `--card-bg`。减少改名量，降低遗漏风险。

### 问题 8：localStorage key 表和实际代码不对齐

**位置**：§八（第 1309-1320 行）

方案写的 key 是：
- `camp_templates_tasks`
- `camp_templates_schedules`

但 dashboard.html 实际的 key 是：
```js
const TEMPLATE_STORAGE_KEY = 'camp_templates';  // 单 key，靠 pool 字段区分类别
```

**这不影响功能**（新增的 key 不会和旧 key 冲突），但如果你按方案去代码里搜 `camp_templates_tasks`，搜不到，会以为还没实现。建议文档表对齐实际代码。

### 问题 9：邀请码管理每次打开都强制生成新码

**位置**：§4.1（第 731-737 行）

```js
function showInviteManager() {
  var codes = getInviteCodes();
  var codeStr = ...;
  var newCode = generateInviteCode();  // ← 每次打开都生成！
  alert('...已生成新邀请码：' + newCode);
}
```

用户只是想看一眼现在还剩下哪些邀请码，结果**被强制消耗了一个新码**。而且它即使没被复制就走了，那个码也会留在数组里永远不被使用。

**建议**：把"生成新码"做成一个独立的按钮，和"查看现有码"分开。

### 问题 10：重复代码清理的边界太粗糙

**位置**：§〇（第 27 行）

方案说删"行 ~2700-3510"，但实际情况更复杂：
- `escAttr` 在 1944 行（第一组不完整版）和 2732 行都有。如果统一删了第二组的 `escAttr`，但第一组的版本因为嵌套在 `recalcFinanceByPerson` 里而变成局部函数，全局就少了 `escAttr`。
- `parseDate` 在 2737 行，它只有一个版本。如果被一刀切删掉，整个系统解析日期的功能全坏。

**建议**：清理之前先跑一遍函数交叉对比（哪些在两处都存在、哪些只有一处），再精确决定删什么。不要靠行号范围一刀切。

### 问题 11：5 个模式中 3 个是空的

**位置**：§4.5、§6（第 936-964 行、第 1218-1242 行）

预测、整理、成就三个模式的内容是统一的占位符：
```html
<p style="color:var(--muted)">此功能将在下阶段完善。</p>
```

这三个按钮会一直显示在共建者界面上，但用户点进去只能看到一个"此功能将在下阶段完善"。**从用户体验角度看，不如先把这三个按钮隐藏**，等对应功能做好了再放出来。这也符合"少即是多"——目前真正有用的只有创世（对应 wizard）和监察（冒险者进度总览）。
