## 一、数据架构

### 1.1 localStorage 结构

```
localStorage
├── nt_users              ← 用户账号（全局共享，不改）
│     { "砚仁": { name, password, role, avatar_seed, created } }
│
├── nt_worlds_index       ← 世界索引（新增）
│     ["plaza", "1100", "5700"]
│
├── nt_world_plaza        ← 南塘村口（新增，永续）
│     { tasks, finance_plaza, stay_records, ... }
│
├── nt_world_1100         ← 第四期（从 camp_data 迁移过来）
│     { tasks, members, budget, finance_cny, finance, schedule,
│       camp_progress, camp_dates, staff_cards }
│
├── nt_world_5700         ← 第五期（新建时生成）
│     { ...同上结构... }
│
├── nt_session            ← 当前登录会话（不变）
└── nt_remembered_user    ← 记住我（不变）
```

### 1.4 JS 全局状态

```js
// 当前活跃的世界
let currentWorldId = 'plaza';          // 当前加载的世界 ID
let worldData = {};                    // 当前世界的数据（替代原来的 data）
let allWorlds = ['plaza'];             // 所有世界的 ID 列表

// 用户账号系统不变
let currentUser = null;
```

### 1.5 数据迁移

首次运行时，如果检测到旧的 `camp_data` 存在但 `nt_worlds_index` 不存在：

1. 读取 `camp_data` → 迁移到 `nt_world_1100`
2. 创建 `nt_worlds_index = ["plaza", "1100"]`
3. 创建空的 `nt_world_plaza`
4. 删除旧的 `camp_data` key（或保留做备份）

## 三、社区管理员

### 3.1 怎么产生

- 系统首次运行时，第一个注册的共建者自动成为村口管理员
- 管理员可以在村口设置里转让管理权给另一个人
- 管理员不一定是某个期的共建者——可能是个专门管日常运营的人

### 3.2 管理员能做什么

| 功能 | 操作 |
|------|------|
| 发布日常委托 | 和发布委托一样，但任务归属于 `nt_world_plaza`，不在任何期里 |
| 确认交付 | 委托人/接单人标记完成后，管理员点「✅ 确认」→ NT 正式从委托方划转 |
| 登记住宿 | 录入 `谁 + 入住日 + 离开日` → 自动算总费用 → 跟踪是否已付给管理员 |
| 修改公告 | 编辑村口顶部的公告文字 |
| 转让管理权 | 指定另一个共建者为新管理员 |

### 3.3 管理界面

管理员登录后，在村口页面看到一个额外的「⚙️ 管理」按钮：

## 四、怎么进出

### 4.1 登录后默认进入

```
登录
  │
  ├─ 共建者 → 默认进入「上一次活跃的世界」（有则进，无则进村口）
  ├─ 冒险者 → 默认进入「注册时选择的世界」（有则进，无则进村口）
  └─ 在地伙伴 → 只能进村口
```

### 4.2 切换世界

在顶部世界切换栏点击对应世界名：

1. **保存当前世界数据**到 localStorage
2. **加载目标世界数据**到 `worldData`
3. **重新渲染界面**（HUD、tab、内容全部刷新）

### 4.3 从副本去村口

在任何营期世界的界面上：

```
┌─ 🧙 世界终端 ── 砚仁 ── 🔮 第四期 ──────────────────┐
│                                            [🏕️ 去村口] │
```

点击「🏕️ 去村口」→ 自动保存当前世界 → 加载村口数据 → 切换到村口界面

### 4.4 从村口回副本

```
┌─ 🏕️ 南塘村口 ──────────────────────────────────────┐
│  [🔮 第四期·寻找]  [🔮 第五期]                       │
│     点击回到副本                                      │
```

## 五、P2P 流水记账

### 5.1 委托流转过程

```
① 委托方发布委托
    砚仁 发了一个委托：「打扫画室」NT 50

② 有人接单
    王五 点了「🙋 接单」

③ 完成后
    王五 点「✅ 完成」→ 状态变为「等待确认」

④ 管理员确认（或委托方自己确认）
    管理员点「确认交付」→ NT 从砚仁的村口余额划给王五

    如果委托方自己确认（小额的简单任务），流程 ③→④ 合并
```

### 5.2 流水数据结构

```js
{
  id: "tx_001",
  type: "delegation",        // "delegation" | "transfer" | "lodging"
  from: "砚仁",
  to: "王五",
  amount: 50,
  reason: "打扫画室",
  task_id: "task_005",       // 关联的委托 ID
  status: "confirmed",       // "pending" | "confirmed" | "cancelled"
  created: "2026-07-06",
  confirmed_by: "小红",       // 谁确认的
  confirmed_at: "2026-07-06"
}
```

### 5.3 NT 余额的计算

```
一个人的 NT 余额 = 所有世界里（收入 - 支出）的汇总

收入来源：
  · 营期任务奖励（来自各期预算池）
  · 村口委托报酬（来自委托方付款）
  · 别人直接转给你的 NT

支出去向：
  · 村口委托付款（你委托别人做的事）
  · 食宿费（交给社区管理员）
  · 转给别人的 NT
```

每个人的 NT 余额**跨世界汇总**，在村口的排行榜里显示。

## 六、与现有系统的关系

### 6.1 改动范围

| 模块 | 改动 |
|------|------|
| 数据模型 | `data` → `worldData`，所有引用全局 data 的地方改为 worldData |
| 数据存取 | `localStorage.getItem('camp_data')` → `loadWorld(id)` / `saveWorld()` |
| 导出导入 | 导出 → 导出「当前活跃世界」的 JSON；导入 → 导入到「当前活跃世界」 |
| 登录系统 | `nt_users` 不变，增加世界归属 |
| 注册流程 | 冒险者注册时增加「世界码」输入框 |
| 期数标记 | 冒险者注册时填的世界码 = 所属期数 |
| 顶部 UI | 新增世界切换栏 |
| 村口 | 全新的一套 tab 和渲染逻辑 |

### 6.2 不改的

- 5 步创世流程逻辑（①②③④⑤）
- 任务系统数据模型
- 预算渐进细化逻辑
- 日历系统
- 头像系统
- localStorage 存储方式（只改 key 名和存取方式）
