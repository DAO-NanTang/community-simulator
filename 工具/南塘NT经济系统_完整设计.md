# 南塘 NT 经济系统 · 完整设计

> 把 NT 从「固定工资」变成「贡献凭证」——互评分配 + 闭环流通
> 2026-07-07

---

## 〇、一句话说清

**NT = 被看见的贡献。** 不是货币，是社区对一个人劳动价值的集体确认。

## 一、NT 从哪里来、到哪里去、怎么消失

### 完整循环

```
                  ┌──────────┐
                  │ 营地预算  │  ← 每一期营地的总拨款（如 30,000 NT）
                  └────┬─────┘
                       │ 注入
                       ▼
              ┌─────────────────┐
              │   周期 NT 总池    │  ← 每周期（如1周）从预算中拨出一份
              │   (如 3,000)    │
              └────────┬────────┘
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ 日常劳动  │ │ 营期委托  │ │ 特殊贡献  │  ← 三类事项分别互评
    │ (社区)   │ │ (任务板) │ │ (自发)   │
    └────┬─────┘ └────┬─────┘ └────┬─────┘
         │            │            │
         └────────────┼────────────┘
                      │ 互评确定分配比例
                      ▼
              ┌─────────────────┐
              │  每个人拿到 NT   │
              └────────┬────────┘
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ 住宿费    │ │ 委托支出  │ │ 转赠/打赏 │  ← 消费 NT
    │ (回收)   │ │ (P2P)   │ │ (P2P)   │
    └────┬─────┘ └────┬─────┘ └────┬─────┘
         │            │            │
         ▼            ▼            ▼
    ┌─────────────────────────────────────┐
    │        回收的 NT 去哪里？            │
    │  · 住宿费 → 回归营地预算（下期复用） │
    │  · 委托支出 → 转给接单人            │
    │  · 打赏 → 转给被打赏人             │
    └─────────────────────────────────────┘
```

### 三条铁律

| 铁律 | 说明 |
|------|------|
| **总量封顶** | 每个周期的 NT 发行量 = 营地预算 ÷ 周期数。不发超过预算的 NT |
| **评价在后** | 先做事，后评价。不做固定定价。一件事值多少 NT，由参与者互评决定 |
| **消费回收** | 住宿费吃掉流通中的 NT，回到预算池，下一周期重新分配 |

## 二、互评分配：怎么把劳动变成 NT

### 周期设定

一个**评价周期 = 一周**。每周日晚上进行互评。

```
周一～周六：做事（打扫、帮厨、农活、委托……）
周日晚上：  互评会（30分钟）
            → 列出本周所有事项
            → 每个人对每件事打分
            → 讨论分歧
            → 确定分配比例
            → 系统划转 NT
```

### 评分阶梯

| 分数 | 含义 | 什么时候给 |
|------|------|-----------|
| **0** | 未参与 | 这件事跟你无关 |
| **5** | 轻参与 | 帮了个小忙、提供了建议 |
| **13** | 一般参与 | 参与了但非主力 |
| **43** | 主要贡献 | 是这件事的主力之一 |
| **89** | 独挑大梁 | 这件事几乎是你一个人扛的 |

离散阶梯（非0→1→2→3线性）强制拉开差距——你没法"给大家都打 10 分"蒙混过去。

### 互评流程

```
Step 1: 列出事项
  系统自动生成本周事项列表（从事件流中提取）
  + 人工补充遗漏的事项

Step 2: 逐事项打分
  每个人对每个事项的每个参与者打分
  （自己的分可以跳过，系统用他人均分填补）

Step 3: 讨论分歧
  对某个人的某件事，如果评分差异大（如有人打89、有人打5）
  → 讨论为什么差距这么大
  → 修正或保留

Step 4: 确定分数
  掐头去尾取平均（去掉一个最高和一个最低后算均值）
  或者多轮趋同后直接取众数

Step 5: 汇总分配
  个人总得分 / 全员总得分 = 个人比例
  个人比例 × 本周 NT 总池 = 个人所得
```

### 算例：一周日常

本周 NT 池 = 3,000 NT。5 个在地成员。做了 4 件事。

| | 砚仁 | 朝林 | 小红 | 王五 | fc |
|:---|:---:|:---:|:---:|:---:|:---:|
| 打扫画室 | 5 | 0 | 43 | 13 | 0 |
| 帮厨午餐 | 0 | 43 | 13 | 0 | 5 |
| 菜地浇水 | 0 | 0 | 0 | 43 | 13 |
| 接待访客 | 43 | 5 | 0 | 0 | 0 |
| **总得分** | **48** | **48** | **56** | **56** | **18** |
| **比例** | 21.2% | 21.2% | 24.8% | 24.8% | 8.0% |
| **所得** | 636 | 636 | 744 | 744 | 240 |

- 小红和王五各自挑了一项主力，得分最高
- fc 只参与了帮厨和浇水（都是轻参与），得分低
- 砚仁和朝林各有一项主力和一项轻参与，居中

**没有"固定工资"。做了事就有分，没做就没有。**

## 三、三类事项池

一个周期内的 NT 来源分三块，每块独立分配。

### 池A：日常劳动（社区维护）

**来源**：营地预算拨款（如每周 1,500 NT）

**覆盖事项**：
- 打扫（画室、厨房、卫生间、宿舍走廊）
- 帮厨（采购、做饭、洗碗）
- 农活（浇水、施肥、除草、收割）
- 倒垃圾、整理公共区域
- 突发维护（修水管、搬东西）

**评价者**：所有在地成员 + 当时在场的访客

**特点**：这些事情每天都发生，不容易被记住。系统从事件流中自动生成"本周事项清单"作为评价的起点。

### 池B：营期委托（任务系统）

**来源**：营地预算中的 NT 任务拨款（如每周 1,000 NT）

**覆盖事项**：
- 主线任务（课程、创作）
- 支线任务（写生、社区融合）
- 共创任务
- 隐藏任务

**评价者**：NT管理员 + 任务发布者 + 参与者

**特点**：这些已经有 `data.tasks` 中的 `claimants` 和 `status` 追踪。互评在这里的作用是——不是任务发布者一个人说"值 70 NT"，而是所有参与者确认"这个人的贡献确实值这个比例"。

### 池C：特殊贡献（自发行为）

**来源**：营地预算中的弹性拨款（如每周 500 NT）

**覆盖事项**：
- 提出了一个被采纳的建议
- 主动带新来的人参观
- 解决了一个没人注意的问题
- 拍了一张传播效果好的照片

**评价者**：全员提名 + 投票

**特点**：这些事情不在任务列表上，容易被忽略。池C 的存在就是为了让"多做了"的人被看见。

## 四、NT 怎么回收（防止通胀）

NT 不是无限发行。花掉的 NT 要么流转给另一个人，要么被系统回收。

| 消费类型 | 流向 | 经济效应 |
|---------|------|---------|
| **住宿费** | 交给管理员 → 回归预算池 | **回收**：减少流通 NT |
| **委托支出**（P2P） | 委托方 → 接单人 | **流转**：不增减总量 |
| **打赏/转赠** | 打赏者 → 被打赏者 | **流转**：不增减总量 |
| **物品消耗** | NT 标记消耗，不实际扣减 | **记录**：用于统计 |

**关键**：只有住宿费是真正的 NT 回收。它把流通中的 NT 抽回预算池，下一周期重新分配给劳动者。这就形成了一个**贡献→获得→消费→回收→再分配**的闭环。

### 住宿费定价

住宿费 = 日费率 × 天数。审查报告指出当前日费率 70 NT/天严重偏高（月住宿 2100 NT vs 月收入 ~130 NT）。

**修正**：日费率应根据实际经济情况浮动，而不是固定值。有两种方式：

| 方式 | 逻辑 | 优点 | 缺点 |
|------|------|------|------|
| **浮动比例制** | 每月住宿费 = 当月收入的 X%（如 30%） | 永远付得起 | 收入高的人付得多 |
| **固定低价制** | 日费率设为 5-10 NT（月 150-300） | 可预期 | 需要验证经济平衡 |

**推荐浮动比例制**。理由：住宿费是社区维系成本，理应由所有受益者按能力分担，而不是定额收费把低收入者逼走。一个人每月赚 240 NT，交 30%（72 NT）后还剩 168 NT——够用。一个人每月赚 800 NT，交 30%（240 NT）——也是公平的。

## 五、数字验证：一个月的经济循环

场景：5 个在地成员，1 期营地（4 周），总预算 30,000 NT。

### 发行侧

| 池 | 每周金额 | 4周合计 | 占比 |
|----|---------|--------|:---:|
| 池A 日常劳动 | 1,500 | 6,000 | 50% |
| 池B 营期委托 | 1,000 | 4,000 | 33% |
| 池C 特殊贡献 | 500 | 2,000 | 17% |
| **周发行** | **3,000** | **12,000** | — |

（剩余 18,000 NT 为委托预扣预留 + 下期滚动）

### 流通侧（人均）

| 项目 | 金额/月 | 说明 |
|------|--------|------|
| 互评收入 | 2,400 | 12,000 ÷ 5人，假设均分 |
| 住宿费（30%） | -720 | 浮动比例制 |
| **净留存** | **1,680** | 可自由支配 |

1,680 NT 可以用来：发布委托请人帮忙（池B流转）、打赏别人的记录、攒着等下期。

### 和固定定价的对比

| | 固定定价 | 互评分配 |
|----|---------|---------|
| 打扫一次 | 固定 10 NT | 看贡献程度，可能 0-200+ |
| 偷懒的人 | 也能领 10 NT | 得 0 分，拿 0 NT |
| 一个人扛了所有打扫 | 还是 10 NT × 次数 | 比例大幅提升 |
| 忘了做但平时很勤快 | 直接没有 | 别人会提"ta平时一直在做"，补上 |

## 六、和现有 dashboard 的衔接

### 不需要改的

- `data.tasks` 的 `points` 字段保留——它作为"预算参考值"，用于发布时预扣
- `data.finance` 的 NT 流水结构保留
- `nt_users` 的账户结构保留

### 需要加的

| 新增 | 说明 |
|------|------|
| `data.eval_cycles` | 评价周期记录：周期ID、起止日期、NT池金额、参与人、事项列表 |
| `data.eval_scores` | 评分记录：谁评谁、哪个事项、分数、日期 |
| `evalDistribute()` | 分配函数：汇总得分 → 计算比例 → 写入 NT 流水 |
| 互评 UI | 一个简单的评分界面（列表 + 打分按钮） |

### 和现有任务系统的关系

现有任务系统（NT管理面板）的 `task.points` **不会被删除**——它仍然是"预算参考值"。区别在于：

- **发布时**：`points` 用来预扣委托方的 NT（防止赖账）
- **结算时**：不再直接按 `points` 发放，而是等周期结束时通过互评确定实际分配

这意味着 `points` 从"定价"变成了"预算上限"——你可以设 70 NT，但互评后实际可能只给 50（如果贡献不够）或给 90（如果超额完成，从池C补）。

## 七、实施路径

### Phase A：先跑手工互评（不写代码）

用一个 Excel 或 Google Sheets 跑 2-4 周的互评。验证：
- 分配结果是否让人觉得公平？
- 评分阶梯 0/5/13/43/89 是否好用？
- 每周 30 分钟的互评会是否可行？

### Phase B：把互评写进 dashboard

- 新增「互评」tab（管理员可见）
- 自动从事项列表中生成评价表
- 打分 → 自动汇总 → 写入 NT 流水

### Phase C：经济参数校准

根据 Phase A 的手工数据，调整：
- 三个池的比例
- 住宿费比例
- 评分阶梯是否需要调整
