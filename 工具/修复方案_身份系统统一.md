## 一、新增 `changeUserRole()` 统一函数

### 函数签名

```javascript
function changeUserRole(name, newRole, opts) {
  // name:    用户名
  // newRole: 'admin' | 'builder' | 'adventurer' | 'npc' | 'visitor'
  // opts:    { period, inviteCode, fromRole, staffCard }
  //           period:      '第四期'（可选，写进 season_history）
  //           inviteCode:  'NT-A3F2'（可选，升级路径使用的邀请码，自动消耗）
  //           fromRole:    旧角色（可选，不传则自动从当前数据读取）
  //           staffCard:   { role: 'NT', ... }（可选，共建者卡片信息）
}
```

### 返回值

```javascript
// 返回 { ok: true } 或 { ok: false, error: '原因' }
// 调用方应检查返回值，失败时不继续后续操作
```

### 函数内部做的事（6 步）

```
changeUserRole('小明', 'adventurer', { period: '第四期', inviteCode: 'NT-A3F2' })
    │
    ├─ Step 0: 校验
    │   如果 users[name] 不存在 → return { ok: false, error: '用户不存在' }
    │   如果 name === adminNames 中的管理员且 newRole 非 admin → return { ok: false }
    │   如果 newRole === 'builder' 且用户无 adventurer 历史 → return { ok: false }
    │
    ├─ Step 1: 读旧角色，写 season_history
    │   从 users 表读当前角色 → 追加到 users[name].season_history[]
    │   如果 season_history 不存在则初始化为 []
    │   记录格式：{ role: 'visitor', period: '第四期', changed_at: '2026-07-10' }
    │
    ├─ Step 2: 更新 users 表（浏览器 localStorage）
    │   users[name].role = newRole
    │   如果有 world_id → 写入
    │   如果有 staffCard → 写入关联
    │
    ├─ Step 3: 更新 data.members（营地数据）
    │   如果 data.members 不存在则初始化
    │   if (!data.members[name]) data.members[name] = {}
    │   data.members[name].role = newRole
    │
    ├─ Step 4: 如果改的是当前登录用户
    │   currentUser.role = newRole
    │   refreshUserHUD()
    │   showMainInterface(newRole, 'workspace')  ← 界面立刻刷新
    │   （注：页面初始化时 L1767 会用 users[name].role 覆盖 currentUser.role，
    │    因此无需额外保存 session，刷新页面也不会丢失）
    │
    └─ Step 5: 消耗邀请码（如果传了 inviteCode）
        同时查 world_codes 和 invite_codes 两个池
        从匹配的池中删除该码，调用 saveWorldCodes() / saveInviteCodes()
    → return { ok: true }
```

### 关键约束

- **管理员不可被降级**：维护 `adminNames` 数组（默认 `['砚仁']`），管理员改为非 admin 时拒绝
- **共建者必须有冒险者历史**：改为 builder 前检查用户 `season_history` 中曾有 adventurer 记录
- **一次只改一个角色**：不会出现同时触发两条路径的冲突

## 二、5 条身份升级路径

### 路径 A：注册 → 云村民

```
注册页填名字+密码+头像 → doRegister()
    → registerUser() 创建用户（role='visitor'）
    → loginUser() → enterWorld()
    → 无 lastPage → showPlazaPlaceholder() → 村口
```

- 不调 `changeUserRole`（初始创建，不是变更）
- 注册时默认就是 visitor

### 路径 B：云村民 → 冒险者

```
村口 → 活动信息站 → 选营期 → 输入冒险者邀请码
    → changeUserRole(name, 'adventurer', {
          period: '第四期',
          inviteCode: 'NT-A3F2',
          fromRole: 'visitor'
      })
    → 界面刷新：标签栏出现 📋任务大厅
    → 邀请码 NT-A3F2 从池子删除
    → season_history 追加 { role: 'adventurer', period: '第四期' }
```

**调用位置**：`doUpgradeWithCode()` 中的两处（L2532, L5471）合并为一处调用

### 路径 C：云村民 → 在地 NPC

```
触发方式一：村口「实景地图」→ 输入 NPC 邀请码
触发方式二：注册满 7 天 → 系统提示 → 确认升级
    → changeUserRole(name, 'npc', {
          inviteCode: 'NPC-X1Y2',  // 方式一有码，方式二无码
          fromRole: 'visitor'
      })
    → 界面不变（NPC 和云村民标签栏相同，但可以领在地任务了）
    → world_id 写入
```

**调用位置**：`upgradeToNPC()` L2308 改为调用统一函数

### 路径 D：冒险者 → 共建者

```
创世终端 Step3 → 管理员分配角色卡片给冒险者
    → changeUserRole(name, 'builder', {
          fromRole: 'adventurer',
          staffCard: { role: 'NT', group: '课程' }
      })
    → 界面刷新：标签栏变成共建者版
    → staff_cards 写入选中的卡片
```

**调用位置**：Step3 分配逻辑 L8553、L8653 合并

### 路径 E：冒险者 → NPC（营期结束）

```
管理员点「完结本期」→ 扫描所有 period 匹配的 adventurer
    → 批量 changeUserRole(name, 'npc', {
          fromRole: 'adventurer',
          period: '第四期（完结）'
      })
    → season_history 追加，不丢勋章
    → 下次登录：营地任务显示「🔒 上期已完结」
```

**调用位置**：`closePeriod()`（暂未实现，Phase 10/11 时做）

## 三、调用点改造

> ⚠️ 关键发现：**身份角色**（admin/builder/adventurer/npc/visitor）和**岗位角色**（WIZARD_ROLES：统筹/宣传/NT/后勤/财务等）是两套不同的概念。`changeUserRole` 只处理身份角色变更。⑤⑥⑦ 操作的是岗位角色（`WIZARD_ROLES`），不纳入本次改造。

### 3.1 纳入改造的身份角色变更（②③④）

> ⚠️ 改造前调用方会**预先消耗邀请码**（调 `saveWorldCodes` / `saveInviteCodes`），改造后由 `changeUserRole` 统一消耗，调用方已有的码消费代码**必须删除**。

| # | 原来 | 改为 | 需删除的预消费代码 |
|---|------|------|------|
| ② | `upgradeToNPC()` 手动写 3 处 | `changeUserRole(name, 'npc', { inviteCode })` | `submitMapInvite()` 中的 `saveWorldCodes(wc.filter(...))` |
| ③ | `doUpgradeWithCode()` L2532 | `changeUserRole(name, 'adventurer', { period, inviteCode })` | 函数内的 `saveWorldCodes(codes.filter(...))` |
| ④ | `doUpgrade()` L5467-5475（两条路径） | 先尝试 `changeUserRole(name, 'adventurer', ...)`，失败则尝试 `changeUserRole(name, 'npc', ...)` | 两条路径各自的 `saveWorldCodes(...)` / `saveInviteCodes(...)` |

### 3.2 补丁（①）

| # | 原来 | 改为 |
|---|------|------|
| ① | `registerUser()` 缺少 `season_history` 初始化 | 加 `season_history: [{ role, period: '', changed_at }]` |
| ① | `data.members[name].role` 写 `''` | 改为写实际 `role` |

### 3.3 不纳入改造的岗位角色操作（⑤⑥⑦）

这些位置操作的是 `WIZARD_ROLES`（统筹/NT/后勤等岗位角色），不是身份角色。它们在当前架构中通过 `migrateExistingMembers()` 在页面加载时自动同步到 `users` 表：

| # | 位置 | 说明 | 动作 |
|---|------|------|------|
| ⑤ | 成员档案表格 | 管理员修改岗位角色 | 保持原样 |
| ⑥ | Step3 分配卡片 | 创世终端分配岗位 | 保持原样 |
| ⑦ | Step3 改卡片角色 | 创世终端改岗位 | 保持原样 |

## 四、season_history 格式

```javascript
// users['小明'].season_history
[
  { role: 'visitor',   period: '',        changed_at: '2026-06-01' },
  { role: 'adventurer', period: '第四期',  changed_at: '2026-07-01' },
  { role: 'npc',        period: '第四期（完结）', changed_at: '2026-07-20' },
]
```

### 用途

- **个人页**：显示"曾参与：第四期冒险者、第五期冒险者"
- **勋章**：`getUserBadges()` 直接读这个数组，升级不丢
- **活跃度**：算参与了几期
