## 总体判断

这份方案说了"要做什么"（改登录页、分五种身份、加路由），但**几乎没说"怎么改"**。一份能拿来施工的方案应该让实施者不看 dashboard.html 也知道改哪个函数、改哪几行、注册流程的每一步用户看到什么。现在这份方案做不到——下面逐条说哪里缺了。

## 🔴 码的方案：概念有了，实现细节全部空白

这是你问的核心问题——方案里码的设计到底写清楚了没。答案是：**没写清楚。**

### 码的现状（dashboard.html 已经有的）

现在系统里只有一种邀请码：

```javascript
// 格式：NT-XXXX（如 NT-A3F8K2）
// 存储：localStorage key 'nt_invite_codes' → 一个数组
// 用法：注册共建者时校验，校验通过后从数组中移除（消耗型）
// 例外：'1100' 是硬编码的永久码，始终不被消耗
```

### 方案说要什么

方案 §1.1 和 §2.2 提到了三种不同的码：

| 码类型 | 谁用 | 方案描述 |
|--------|------|---------|
| 世界码 | 冒险者注册时填 | "冒险者（需要世界码）" |
| NPC邀请码 | 游客升级为NPC | "NPC邀请码 → 升级为南塘NPC" |
| 角色认领 | 管理员在Step3操作 | "不可用码自升级" |

### 缺口清单

**缺口 1：世界码和现有邀请码是什么关系？**

现在的 `nt_invite_codes` 里存的都是 `NT-XXXX` 格式的码，目前只用于共建者注册。"1100" 是永久码。方案引入"世界码"后：

- 世界码是 `NT-XXXX` 格式还是纯数字（如 "1100"）？
- "1100" 从"共建者永久码"变成"第四期世界码"了？还是两套系统并存？
- 冒险者注册时填的"世界码"，校验逻辑是查 `nt_invite_codes` 还是查一个新的 `nt_world_codes`？
- 世界码是消耗型（一人一码）还是共享型（一个码多个人用）？方案 §10 待确认第 4 条问了这个——说明方案作者自己也没定。

**实施者如果现在就动手，不知道该写什么校验逻辑。**

**缺口 2：NPC邀请码从哪来、怎么生成、怎么发？**

方案只说"管理员/共建者在村口生成"。但"村口"现在根本不存在——它是要到下阶段才建的。今天就要实施的登录页改造里，注册 NPC 是"直接注册"，不需要码。那么 NPC邀请码是谁、在什么时候、用什么界面生成的？方案 §2.2 提了一句话但完全没展开。

**缺口 3：码的格式有没有区分？**

如果世界码是 "1100"，NPC邀请码是 "NT-XXXX"，现有的 `getInviteCodes()` 返回的是一个不分类型的数组 `["1100", "NT-A3F8K2", ...]`。怎么区分哪个是世界码、哪个是NPC邀请码？方案没提。要么：
- 格式区分（世界码纯数字，NPC码带NT前缀）
- 存储分离（`nt_world_codes` vs `nt_invite_codes`）
- 加 type 字段（`{ code: "1100", type: "world" }`）

**以上三种方案方案都没选，实施者猜着写。**

## 🔴 登录后路由：五个分支，两个没地方去

### 方案说的（§1.3）

```
登录
  ├─ admin    → 世界终端（创世模式）
  ├─ builder  → 被分配的世界
  ├─ adventurer → 码对应的世界
  ├─ npc      → 村口占位页
  └─ visitor  → 村口占位页
```

### 实际代码能做什么

`enterWorld()` 函数（第 1318 行）当前只有两个分支：

if (user.role === 'builder') {
  // 显示 builderTabs + builderModeBar → 创世模式
} else {
  // 显示 adventurerTabs → 任务大厅

### 要改什么

|------|------|-----------|-----------|
| admin | ❌ 不存在 | 新增。管理员进创世终端，但权限比 builder 大（如可创建世界、可看全部数据） | ❌ 没展开 |
| builder | ✅ 已有 | 把 role === 'builder' 改成 role === 'admin' \|\| role === 'builder' | ❌ 没说 |
| adventurer | ✅ 已有 | 不变 | — |
| npc | ❌ 不存在 | 新增。显示村口占位页，隐藏所有 tab 面板 | ❌ 没说 |
| visitor | ❌ 不存在 | 新增。同上 | ❌ 没说 |

**最要命的是 NPC 和 visitor 要去的"村口占位页"——方案 §1.4 只写了一句话：**

## 🟡 其他没写清楚的地方

### "今天"阶段和"本周"阶段的边界模糊

方案分了三个时间阶段（今天 / 本周 / 下阶段），但：

- §1.5 说"双面板架构不变"（builderTabs + adventurerTabs 两套）
- §2.3 却说共建者看到的 tab 和管理员、冒险者都不同——有"我的委托""我的材料"这两个新 tab

**如果共建者的 tab 和管理员不同，那它需要第三套 tab 面板。** HTML 里要加 `id="builderTabs"`（管理员）、`id="adventurerTabs"`（冒险者）、`id="contributorTabs"`（共建者）——三套。但方案 §1.5 说双面板不变。这俩自相矛盾。

### 管理员的"admin"角色没落地到代码

方案 §1.2 说"初始化时自动创建管理员账号：砚仁 / 1500 / admin"。

但当前代码里 `migrateExistingMembers()` 已经创建了砚仁，角色是 `'builder'`。初始化时再创一个 `role: 'admin'` 的砚仁，会跟已有数据冲突吗？`getUsers()` 会返回一个 key 冲突的对象。到底是改已有砚仁的 role 从 builder 变 admin，还是删了重建？初始化代码写在哪一行、在 `migrateExistingMembers()` 之前还是之后？方案没写。

### `switchTab()` 的权限门禁需要大改

当前权限门禁（第 1519 行）：

```javascript
if (currentUser && currentUser.role === 'adventurer' && !previewMode) {
  if (name === 'settings' || name === 'members' || name === 'players') return;
}
```

加了五种身份后，不同身份能看什么 tab（§2.3 那张矩阵），这段逻辑要变成什么？方案 §2.3 列了一张表但是没有任何对应的代码逻辑。至少应该给一个 `isTabAllowed(role, tabName)` 的判断函数或映射表。

### "村口占位页"要在"今天"做完，但它依赖的数据要"下阶段"才有

方案 §1.4 是"今天"的任务——建一个简单的占位页。但占位页上要显示"自己的注册信息和身份"——这个数据在 `nt_users` 里已经有了，能显示出来。但方案没说它到底显示什么、不显示什么、长什么样。哪怕给三行伪代码或者一个 ASCII 线框图，实施者也能马上开工。现在只能靠猜。

## 📋 审查原则逐项评分

| 原则 | 评分 | 说明 |
|------|------|------|
| **少即是多** | ✅ | 分三步走（今天/本周/下阶段），每步不改的不碰——这个思路是对的。"今天"只做登录页简化+路由+占位页，不动数据层 |
| **复用优先** | 🟡 | §1.5 列了不改清单，是对的。但登录页改造方案完全没提复用现有的 `showRegister()`、`registerUser()`、`renderUserGrids()` 等函数——到底是改还是新建？ |
| **需求对齐** | 🟡 | 登录页从三个按钮简化为两个——有需求（Phase 1 审查报告指出的"登录页太花"）。去掉共建者和管理员的公开注册——有需求（Phase 2 审查确认的身份模型）。但 NPC 和游客注册加进来的具体用户场景是什么？"南塘在地伙伴"和"临时来访"在实际运营中怎么区分？ |
| **全局最优** | 🟡 | "今天"不动数据层、不动 Wizard、不动导入导出——边界划得对。但 §2.3 的 tab 矩阵暗示需要三套面板，和 §1.5 的"双面板不变"矛盾 |
| **长期主义** | 🟡 | `enterWorld()` 按 role 路由的设计预留了扩展点，加多世界时只需要在分支里加加载世界的逻辑——这个衔接关系写得清楚。但码的类型系统没定义（世界码 vs NPC码 怎么区分），往后扩会越来越乱 |

## 修正建议

1. **给码定型号**：明确世界码用 `NT-W-XXXX` 前缀、NPC邀请码用 `NT-N-XXXX` 前缀，或者存成 `{ code, type: 'world'|'npc' }` 对象。一个自然段就能定下来。
2. **把注册流程画一遍**：从用户点"我是新来的"到看到注册表单，每一步调哪个函数、传什么参数、函数里加哪几行。不需要很长，一张小流程图 + 每个函数的三行伪代码就够了。
3. **给村口占位页写一段 HTML 草稿**：哪怕只有 5 行 `div`，告诉实施者它放在 DOM 的哪里、id 叫什么、渲染函数叫什么。
4. **把 §2.3 的 tab 矩阵落到代码**：给一个 `getUserTabs(role)` 的函数草稿，或者一个映射表 `{ admin: [...], builder: [...], adventurer: [...] }`。矩阵画在表里没法写代码。
5. **补上"本周"阶段 `switchTab()` 的改造伪代码**。
6. **把 §1.5 的"双面板架构不变"和 §2.3 的共建者专属 tab 之间的冲突解决掉**——要么承认需要第三套面板，要么说明共建者复用管理员面板的哪些 tab。
