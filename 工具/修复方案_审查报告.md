## 三、重要问题（影响正确性或可维护性）

### 问题 A：`removeScheduleDay()` 存在重复定义 bug，方案未提及

**位置**：dashboard.html:3912-3934

```javascript
function removeScheduleDay() {     // ← 第一个定义头 (L3912)
function removeScheduleDay() {     // ← 第二个定义头 (L3913)，覆盖了第一个
  // ... 完整逻辑 (L3914-3925) ...
}
  // ... 游离的死代码 (L3926-3934)，永远不执行 ...
```

这是与 `修复方案.md` 开头提到的"已完成的紧急修复"（`recalcFinanceByPerson()` 缺失闭合括号）**同类的函数结构错误**。方案注意到了 3 个紧急 bug 却漏掉了这一个。

此外，锚点检测逻辑在 `renderDailySchedule()`（L3761-3763）和 `colDrop()`（L3991-3994）中**重复实现**，违反 DRY 原则，也应一并修复。

**裁决**：新增为第 0 项——先修这个真 bug，再改功能。

### 问题 B：「已完成的紧急修复」可能未完全生效

**位置**：dashboard.html:1968-2017

第一个 `recalcFinanceByPerson`（第 1968 行）仍然缺少闭合括号——第 1986 行之后直接开始 `function aggregateMaterialBudgets()`，没有 `}`。这导致后续 4 个函数仍嵌套在其内部，成为局部函数。而第二个 `recalcFinanceByPerson`（第 2755 行）是正确的。

**文件中存在两套重复的函数定义**——一套有 bug（局部函数），一套正确（全局函数）。第 1968-2017 行的错误代码块应当清理。

**裁决**：实施任何修改前，先确认代码基线是干净的。

---

### 问题 C：`findAnchorDates()` 字符串匹配在边界情况下不可靠

**位置**：修复方案 第 344-353 行

```javascript
if (cell.indexOf('开营') >= 0 && !anchors.openDay)
  anchors.openDay = { index: i, date: dates[i], label: cell };
```

以下真实场景会导致误匹配：

| 场景 | 单元格内容 | 匹配结果 |
|------|-----------|---------|
| 开营前动员会（中间天） | `📋 开营前动员会` | ❌ 误匹配为开营日锚点 |
| 用户写了两列含"开营"的内容 | `开营日准备` + `开营第一天` | ❌ 只取第一列，可能选错 |

模板填充的锚点内容格式为 `🔒 开营第一天`——应利用 `🔒` 前缀做精确锚定，而非依赖子串匹配。

**裁决**：加强匹配逻辑，匹配 `🔒 开营` 格式或改用更严格的规则。

---

### 问题 D：第 5 项的任务活跃度统计漏掉了「每日」任务

**位置**：修复方案 第 455-460 行

方案的过滤逻辑（`md >= st && md <= dl`）未处理 `deadline === '每日'` 的任务。已有代码（dashboard.html:1332-1333）正确处理了：

```javascript
const isDaily = /每日|每天/.test(t.deadline||'');
return inRange || isDaily;
```

但修复方案未包含此逻辑。

**裁决**：在过滤条件中追加 `|| /每日|每天/.test(t.deadline || '')`。

---

### 问题 E：第 1 项的流水录入未复用已有 UI

**位置**：修复方案 §第1项-C（第 123-135 行）

方案选择用 `prompt()` 弹窗从零写录入表单。但公会资金池页面已有交易录入表单，应复用其逻辑而非重写。同时，方案说"不影响公会资金池"，但未验证新增流水后 `recalcFinanceByPerson()` 是否能正确更新 `by_person` 汇总。

**裁决**：复用资金池已有录入 UI；明确新增流水后的刷新链（`recalcFinanceByPerson()` → `renderMemberDetail()` → 必要时 `renderBudget()`）。
