> 分析日期：2026-07-08
> 分析对象：dashboard.html（6448 行）+ data.json（1713 行，含第三期真实数据）
> 分析目的：从零开始，逐按钮、逐身份盘清整个系统的数据流动

---

## 一、系统总览：这个系统是什么？

一句话：**一个单页 Web 应用，用于管理艺术共创营的全流程**——从筹办、招生、教学到结项，通过 NT（代币）激励机制驱动任务分配与完成。

**技术实现**：单个 HTML 文件（dashboard.html），无后端，所有数据存储在浏览器的 localStorage 中。JavaScript 约 5300 行，内嵌 CSS 约 1100 行。

## 二、数据全景图：数据存在哪里？

### 2.1 localStorage 九把钥匙

系统中存在 **9 个 localStorage key**，分为三层：

**核心业务数据层：**

| Key | 存储内容 | 读写位置 |
|-----|---------|---------|
| `camp_data` | `data` 对象的完整 JSON。包含 tasks/members/budget/finance/schedule/camp_progress 等全部顶层字段 | 页面加载时读（行 1169），40+ 处写入 |
| `camp_wizard_tasks` | 创世终端 Step2 编辑中的任务数组 | Step2 保存时写入，页面加载时恢复（行 1249） |
| `camp_templates` | 预设模板 + 用户自定义模板 | 页面加载时从 `PRESET_TEMPLATES` 重建（行 6150-6157） |

**用户与权限层：**

| Key | 存储内容 | 读写位置 |
|-----|---------|---------|
| `nt_users` | 所有用户账户对象 `{ name, role, password(编码), avatar_seed, season, created, world_id }` | 注册/登录时读写（行 1283） |
| `nt_session` | 当前登录会话 `{ name, role }` | 登录时写入，页面刷新时自动恢复（行 1478） |
| `nt_remembered_user` | "记住我"缓存 `{ name, role }` | 勾选"记住我"时写入（行 1479） |
| `nt_invite_codes` | NPC 一次性邀请码数组（格式 `NT-XXXXXXXX`） | admin 管理邀请码时写入（行 1287） |
| `nt_world_codes` | 世界码数组（如 `['1100']`） | 注册冒险者时校验用（行 1285） |

**UI 状态层：**

| Key | 存储内容 | 读写位置 |
|-----|---------|---------|
| `inbox_lastOpened` | 信箱上次打开的时间戳（ISO 字符串） | 打开信箱时更新（行 1548） |

### 2.2 data 对象：核心业务数据的结构

```
data = {
  tasks:         [],      // 所有任务（46个第三期任务 + NT系统字段）
  decisions:     [],      // 提案/修正案/决策记录
  members:       {},      // 成员档案，key=姓名，value={name, role, gender, avatar_seed, bio, wallet, nt_actual, ...}
  budget:        {},      // 预算汇总（tuition, sponsor, salary, material, reserve, income, expense, diff, nt_raised, nt_auction, nt_total_out, nt_remaining）
  budget_items:  [],      // 预算明细行（17项默认：3收入 + 14支出）
  finance_cny:   {},      // 人民币财务（summary + transactions数组 + by_person + by_type）
  finance:       [],      // NT交易流水（{type, from, to, amount, task_name, date, scope, ...}）
  schedule:      {},      // 日程（dates数组 + slots数组 + weekdays）
  camp_progress: {},      // 创世终端进度（{step, steps: {'1':'done','2':'active',...}}）
  camp_dates:    {},      // 营期日期（start, end, duration_days, milestones）
  staff_cards:   [],      // 工作人员卡片（Step3打包命名产生，旧名bundles会自动迁移）
  mapMilestones: [],      // 策略地图里程碑（getMilestones自动初始化）
  summary:       {},      // 元数据（total_meetings, total_tasks, total_decisions, meetings数组）
  stats:         {},      // 统计数据（total_tasks_all, records, nt_distributed, people_covered, completion_rate）
  _wizardTasks:  省略，  // 导出时才注入，不属于data本身
}
```

**现状中 data.tasks 的每个任务包含 22 个字段**：name, type, category, assignee, start, deadline, confirmer, points, status, source, note, scope, visibility, reviewer, min_season, claimants, active, poster, posted_at, max_slots, materials, subtasks, outputs, steps。

## 三、入口流向：从打开页面到看到界面

### 3.1 首次访问（无任何 localStorage 数据）

```
打开页面 → DOMContentLoaded 触发（行 1167）
  ├─ 读 camp_data → 空 → data 保持默认值（空数组/空对象）
  ├─ updateAll() → 渲染空的各页面
  ├─ loadTemplates() → 从 PRESET_TEMPLATES 重建模板写入 camp_templates
  ├─ migrateExistingMembers() → 无数据，跳过
  ├─ seedInitialInviteCode() → 创建世界码 '1100' 写入 nt_world_codes
  ├─ seedAdminAccount() → 创建 admin 账户 { 砚仁, role:'admin', 密码:'1500' } 写入 nt_users
  ├─ 检查 nt_session → 空 → 没有会话
  └─ 显示登录页面
```

### 3.2 已注册用户访问（有 localStorage 数据）

```
打开页面 → DOMContentLoaded 触发
  ├─ 读 camp_data → 恢复 data 对象（包含所有历史任务/成员/财务）
  ├─ updateAll() → 渲染所有页面
  ├─ seedAdminAccount() → 用户'砚仁'已存在 → ⚠️ 每次都会把密码重置为 '1500'（Bug #2）
  ├─ 检查 nt_session → 有值 → 自动登录，恢复 currentUser
  └─ 进入对应角色的世界界面
```

### 4.2 管理员（admin = '砚仁'）的完整操作流程

#### 🅰 登录入口

**触发**：打开页面 → 看到身份选择页（登录/注册/预览三个入口）

**数据读**：`nt_users`（用户列表）
**数据写**：登录成功 → `nt_session`（当前会话）、`nt_remembered_user`（如果勾选）

**管理员登入后进入 workspace Tab**。

---

#### 🅱 Tab 1：工作台（workspace）

**渲染函数**：`renderWorkspace()`

**显示四块面板**：
1. **概览面板**：显示当前用户 NT 余额（分活动/私人两维计算）
2. **活动任务面板**：从 data.tasks 中筛选 claimants 中有当前用户且 scope=camp 的任务
3. **私人委托面板**：同上，scope=personal
4. **我发布的待审核**：从 data.tasks 中筛选 poster/reviewer 为当前用户且有 submitted 状态 claimants 的任务
5. **NT 流水**：data.finance 中与本用户相关的最近 20 条

**数据读**：data.tasks, data.finance
**数据写**：无（纯展示）

**按钮操作**：
- **任务状态下拉**：直接修改 task 的 claimants[x].status → 写入 camp_data
- **材料实际金额输入**：修改 task 的 materials[x].actual → 写入 camp_data

---

#### 🅲 Tab 2：委托大厅（questboard）

**渲染函数**：`renderQuestBoard()`

**筛选逻辑**：从 data.tasks 中筛选 active='yes'、visibility 匹配当前用户角色的任务

**数据读**：data.tasks
**数据写**：
- **认领按钮**（claimTask）：推入 claimants 数组 → 写入 camp_data
- **提交按钮**（submitTask）：改 claimants[x].status 为 'submitted' → 写入 camp_data
- **放弃按钮**（dropTask）：改 claimants[x].status 为 'dropped' → 写入 camp_data

---

#### 🅳 Tab 3：创世终端（settings）
> *仅 admin 可见*

**五步向导**：
1. **Step1 统筹定框架**：设置营期日期、预算参数、团队成员
   - 数据读：data.camp_dates, data.budget, data.members, data.schedule
   - 数据写：修改上述字段 → camp_data
2. **Step2 任务池梳理**：管理任务列表（wizardTasks 临时数组）
   - **saveStep2()**：⚠️ `data.tasks = wizardTasks.map(...)` 直接替换，丢失 NT 字段（Bug #1）
3. **Step3 打包命名**：将任务打包给工作人员卡片
4. **Step4 角色填卡**：填写各角色卡片信息
5. **Step5 同步展示**：**executeSync()** 将 staff_cards 展开为 tasks → ⚠️ 同样直接替换 data.tasks（Bug #1）

**关键缺陷**：Step2 和 Step5 的保存都是**替换而非合并**，导致 NT 系统字段（scope, visibility, claimants, max_slots 等）在 wizard 操作后丢失。

---

#### 🅴 Tab 4：NT管理（ntadmin）

**渲染函数**：`renderNTAdmin()`

**数据读**：data.tasks, data.finance, data.budget
**数据写**：
- **审核通过**（reviewTask approve）：改 claimants[x].status='completed'，向 data.finance 插入 task_reward 记录 → camp_data
  - ⚠️ 池余额检查使用 `data.budget.nt_total_pool || data.budget.nt_pool`，但某些流程会删除 nt_pool（Bug #25 审计发现）
  - ⚠️ 审核后只刷新 NTAdmin，不刷新工作台/排行榜/结算页（Bug #11）
- **审核退回**（reviewTask reject）：改 status='rejected'，prompt 获取退回原因
- **发布任务**（showPostTask）：通过连续 prompt 收集 11 个字段，推入 data.tasks → camp_data
- **触发隐藏任务**（triggerHiddenTask）：推入 claimants，status='triggered'

---

#### 🅵 Tab 5：成员档案（members）

**渲染函数**：`renderMembers()`

**数据读**：nt_users + data.members + data.tasks（三个来源合并构建 staffData）
**筛选**：按 memberFilter（builder/adventurer/npc/visitor）切换显示

**操作**：
- **添加成员**（showAddMember）：推入 data.members → camp_data
- **编辑成员**（saveMember）：改写 data.members[name] → ⚠️ 会覆盖丢失 wallet 和 actual_materials（Bug #7）
- **删除成员**（deleteMember）：删除 data.members[name]，⚠️ 不清理 data.finance 中的关联记录（Bug #9）
- **材料表视图**：聚合任务的 materials 数组，对比 budget vs actual
- **任务状态修改**：直接改 claimants[x].status → camp_data

---

#### 🅵-2 Tab：玩家总览（players）

**渲染函数**：`renderPlayers()`（冒险者视角的成员列表）

**筛选逻辑**：只显示 role 为 adventurer 的成员（通过 nt_users 判断）

---

#### 🅶 Tab 6：任务看板（taskhall）

**渲染函数**：`renderDailyView('staff')`

**功能**：按日期的任务视图，分工作人员/玩家筛选，显示每日任务卡片

**数据读**：data.tasks, data.members, data.finance_cny
**操作**：可修改任务状态、分配任务

---

#### 🅷 Tab 7：时间线（timeline）

**渲染函数**：`renderTimeline()`

**功能**：按里程碑阶段（筹备期/营期/营后/改进项）分类展示任务
- classifyPhase() 通过正则匹配 `task.name + task.note + task.type` 自动分类

**数据读**：data.tasks

---

#### 🅸 Tab 8：资源金库（budget）

**渲染函数**：`renderBudget()`

**数据读**：data.budget, data.finance_cny, data.financeTypes
**展示内容**：RMB KPI、NT KPI、交易列表、人员汇总

**操作**：
- **添加交易**：推入 data.finance_cny.transactions → camp_data
- **编辑交易**：双击行编辑

#### 🅹 Tab 9：通关结算（settlement）

### 4.3 冒险者（adventurer）的操作流程

**可见 Tab**：workspace, questboard, timeline, leaderboard

**核心操作链**：
```
登录 → workspace（查看自己的 NT 余额和任务）
  → questboard（浏览可领取的任务）
    → 点击"认领" → 任务进入 workspace 的"活动任务"
      → 完成任务后点"提交" → 状态变为 submitted
        → 等待 admin/builder 在 ntadmin 中审核
          → 审核通过 → NT 到账（finance 中增加记录）
```

**排行榜**：从 data.tasks/finance 聚合所有冒险者的 NT 数据排序展示

**⚠️ 当前缺陷**：任务认领后如果 admin 在 wizard 中操作，可能丢失 scope/visibility 等字段，导致任务在 workspace 中不显示。

## 五、模板系统全景

### 5.1 模板定义位置

模板硬编码在 `PRESET_TEMPLATES` 常量中（行 5878-6146），约 **310 行代码**，嵌入在 dashboard.html 内部。

### 5.2 模板种类（共 3 个预设模板，均硬编码在 PRESET_TEMPLATES 常量中）

| 模板名称 | 类型池 | 任务数 | 行号 |
|---------|--------|:--:|------|
| 📋 流程手册·完整任务包（综合最优） | staff | 54 | 5879-5934 |
| 共创人·隐藏任务包 | creator | ~20 | 5936-5959 |
| 共创人·课程框架包 | creator | ~20 | 5961-5983 |

> ⚠️ 注意：NT_DESIGN_CATEGORIES（4 类 NT 设计任务模板）不在 PRESET_TEMPLATES 中，而是独立常量（行 5523），用于创世 Step4 的 NT 任务设计表。

### 5.3 模板加载机制

```
页面加载 → loadTemplates()
  → 遍历 PRESET_TEMPLATES 的所有 key
  → 深拷贝每个模板数据
  → 写入 localStorage 的 camp_templates
  → ⚠️ 每次页面加载都会覆盖用户的 camp_templates，丢失自定义模板
```

### 5.4 模板的使用

- **Step2 "任务池梳理"** ：通过 `getTemplatesByPool('staff')` 或 `getTemplatesByPool('creator')` 获取模板列表
- 用户选择一个模板 → 其 tasks 数组被合并到 `wizardTasks` 中
- **模板编辑**：可修改、删除模板（修改后存回 camp_templates）

## 六、关键数据流缺陷总结

以"一个冒险者认领任务→完成任务→被审核→看到 NT 到账"为黄金路径，追踪数据流：

### 缺陷 A：wizard 覆盖（最严重）
```
冒险者认领任务 → claimants 推入 → 任务多出 scope/visibility/claimants 字段
→ admin 在创世终端操作 Step2 → saveStep2() 用 wizardTasks 替换 data.tasks
→ ⚠️ scope/visibility/claimants 丢失 → 冒险者的 workspace 中任务消失
```

### 缺陷 B：审核后不刷新
```
admin 审核通过 → finance 中插入记录 → renderNTAdmin() 刷新
→ ⚠️ workspace/leaderboard/settlement 不刷新 → 用户看不到 NT 变化
```

### 缺陷 C：结算页 NT 不一致
```
finance 中 NT 记录正确 → 结算页读的是 members[name].nt_actual
→ ⚠️ 如果 nt_actual 不手动更新 → 显示旧数据
```

### 缺陷 D：导入 JSON 丢字段
```
导出 data.json → 重新导入 → tasks 被 normalize 为 21 个白名单字段
→ ⚠️ materials/subtasks/outputs/steps 丢失（Bug #4）
```

### 缺陷 E：密码每次重置
```
页面加载 → seedAdminAccount() → 无条件覆盖 users['砚仁'].password = encodePassword('1500')
→ 改过密码的 admin 刷新后密码变回 '1500'（Bug #2）
```

## 七、关于"清理数据、提取模板、搭建干净框架"的建议

### 现状问题
当前 dashboard.html 耦合了三层内容：
1. **代码**（JS 逻辑 + HTML 结构 + CSS 样式）
2. **模板数据**（PRESET_TEMPLATES 常量，约 310 行，9 个模板，共约 150 个任务）
3. **运行时数据**（localStorage 中的 camp_data，包含第三期完整的 46 个任务、17 个成员、70+ 条财务记录等）

### 建议的分层方案

```
工具/
├─ dashboard.html          # 干净框架：只含代码，不含模板数据，不含运行数据
├─ 模板/
│   ├─ 流程手册_完整任务包.json
│   ├─ 共创人_隐藏任务包.json
│   ├─ 共创人_课程框架包.json
│   └─ nt设计任务模板.json           ← NT_DESIGN_CATEGORIES 常量
├─ 数据备份/
│   └─ 第三期_20260702.json    # 当前 data.json 归档
├─ data.json               # 空白初始数据（仅含 admin 账号种子逻辑需要的结构）
└─ 修复方案_按阶段执行.md
```

### 干净框架应保留的内容
1. **admin 账号**：`seedAdminAccount()` 保留，但需修复每次都重置密码的 Bug
2. **世界码**：保留 '1100'
3. **模板加载机制**：改为从外部 JSON 文件加载，而非硬编码
4. **所有 UI 和业务逻辑**：完整保留

### 干净框架应清除的内容
1. **PRESET_TEMPLATES 常量**（约 310 行）→ 改为从 模板/ 文件夹加载
2. **localStorage 中所有运行时数据**→ 开发/测试时手动清空
3. **data.json 中的第三期真实数据**→ 归档到 数据备份/
