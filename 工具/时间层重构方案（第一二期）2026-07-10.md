- **日期**：2026-07-10
- **对象**：`工具/dashboard.html`
- **范围**：第一期（建 Clock 时间源 + 修 bug）、第二期（80 处调用收敛）。第三期（内测工具本体）待前两期落地后单独成文。
- **状态**：待验收（本文为方案，尚未改动代码）
- **执行方式**：第一期 → 第二期分 4 批，**每批做完停下来，由砚仁验收后再继续**。

## 一、背景与目标

系统内"当前时间"的读取散落在约 80 处，全部是业务代码**直接** `new Date()` / `Date.now()`，没有任何统一封装。这带来三个问题，也让后续"内测世界"工具无法干净地接管时间。本次重构把所有时间读取收敛到一个入口 `Clock`，既修正现有隐患，又为第三期内测预留唯一注入点。

### 现状诊断（三个真问题）
1. **无统一入口**：80 处裸调用，散落各处。
2. **时区 bug（真实存在）**：日期串用 `new Date().toISOString().slice(0,10)`（UTC），而餐段判断用 `getHours()`（本地）。对 UTC+8 用户，**每晚 0:00–8:00 之间系统认定的"今天"会慢一天**，直接影响逾期判断、今日筛选、提交日期。
3. **无权威时间**：纯前端 + localStorage，读的是各自设备时钟，无服务器校时。本次不解决"设备时钟错误"，但把未来接服务器/NTP 的口子收敛到一处。

> 补充：时间读取本身是纳秒级操作，**不存在性能问题**。本次重构的价值在**一致性、正确性、可测试性**，不是省资源。

### 已确认决策
- **时区基准**：锁定 **UTC+8（Asia/Shanghai）**，与设备时区解耦。
- **执行节奏**：分批实现，每批停下来验收。

## 二、审查修订记录（相对初版方案）

初版方案按"少即是多 / 需求对齐 / 全局最优 / 长期主义"自审后，**削掉 3 处过度设计**，记录如下：

| # | 初版问题 | 违反原则 | 修订 |
|---|---|---|---|
| 1 | 第一期 Clock 里预置了第三期的"内测注入机器"（偏移量、倍速、加速锚点、`nowMs` 分支） | 少即是多 / 需求对齐 | **移除**。第一期只保留"唯一入口"这个**接缝**；偏移、倍速等第三期再加。 |
| 2 | `stamp()` 加了"基准+自增序号"防撞逻辑 | 少即是多 | **简化**为真实时间转 36 进制。理由：ID 读真实时间不受内测影响，且现有 ID 都还带 `Math.random()` 后缀，唯一性早已保证。 |
| 3 | 第二期替换时"顺带做渲染快照优化" | 需求对齐 | **删除**。复查发现"循环内反复取 now"处其实全是带参解析（本就不替换），该优化无实际收益，只会把"纯机械替换"变成"动逻辑"。 |

**保留且确认正确的设计**：
- 所有时间收敛到一个入口——全份方案的核心价值，将来接服务器时间、做内测都只改这一处。
- `iso()` 保持 UTC、`today()` 用 UTC+8：完整时间戳与历史数据一致，日期串按中国日历给人看。**有意为之，将写入注释防误改。**
- 系统内确无现成时间函数可复用（已搜，零匹配），新建 `Clock` 是必要而非重复造轮子。

### 2.1 第二轮审查修订（2026-07-10 · 砚仁审查后）

对照实际代码逐条核验后，追加以下 3 项修订：

| # | 发现 | 影响原则 | 修订 |
|---|---|---|---|
| 4 | **「带参一律不动」与硬门槛冲突**：代码存在 `new Date(Date.now() - 7*86400000)`（[dashboard.html:3713](dashboard.html#L3713) 茶馆归档）。外壳是带参 `new Date(...)`，但括号内嵌着裸 `Date.now()`。若照「带参绝不替换」办则内层 `Date.now()` 残留，会被 4.3 硬门槛判为未完成。 | 需求对齐 / 长期主义 | 规则改为：**带参 `new Date(...)` 的外壳不动，但其括号内部的裸 `Date.now()` / `new Date()` 仍要替换**（本例 → `new Date(Clock.ms() - 7*86400000)`）。见 4.0 补充。 |
| 5 | **既有债登记不全**：除 `parseDL`（硬编码 2026，两处）外，还有 `parseDate`（[6608](dashboard.html#L6608)）、`parseDateStr`（[6621](dashboard.html#L6621)）把 "M/D" 硬编码为 **2025**（共 4 处）。同一个 "7/10" 走不同解析器会得到不同年份。且三个函数都在解析 "M/D"，属重复实现。 | 全局最优 / 复用优先 | 补入第五节既有债清单（本期仍不动，仅登记全）；标注「三个 M/D 解析器重复、年份不一致」为后续统一候选。 |

## 三、第一期（Phase 0）：建 Clock 层，不改行为

### 3.1 目标
- 引入统一时间源，**行为等价现状**——仅修正时区 bug。
- 只**新增**代码、不动旧逻辑，可随时回滚。
- 为第二期迁移、第三期内测预留唯一接缝。

### 3.2 新增模块（放主 `<script>` 顶部，早于 `let data` 与所有业务函数）

```js
// ═══════════ Clock：全系统统一时间源（唯一时间入口）═══════════
// 业务代码禁止裸调用 new Date()/Date.now()；带参解析 new Date('...') 不受影响。
// iso() 存 UTC 完整时间戳（与历史数据一致）；today() 按 UTC+8 取日历日——请勿改动此约定。
var Clock = (function () {
  var TZ_MIN = 8 * 60;                 // 锁定 UTC+8（Asia/Shanghai）
  function _wall() { return Date.now(); }   // 真实设备时钟：ID 用，永远真实
  function _now()  { return _wall(); }      // 应用时钟：业务逻辑用；★第三期内测仅在此一处注入虚拟时间
  function _shift(ms) { return new Date(ms + TZ_MIN * 60000); }  // 移到锁定时区再取字段
  function _p2(n) { return (n < 10 ? '0' : '') + n; }

  return {
    date:  function () { return new Date(_now()); },
    ms:    function () { return _now(); },
    iso:   function () { return new Date(_now()).toISOString(); },     // 完整时间戳（UTC ISO）
    today: function () { var d = _shift(_now());                       // 'YYYY-MM-DD'（UTC+8 日历日）
             return d.getUTCFullYear() + '-' + _p2(d.getUTCMonth() + 1) + '-' + _p2(d.getUTCDate()); },
    hour:  function () { return _shift(_now()).getUTCHours(); },
    min:   function () { return _shift(_now()).getUTCMinutes(); },
    stamp: function () { return _wall().toString(36); }                // ID 前缀；后缀 Math.random() 由调用处保留
  };
})();
```

**关于接缝**：全模块只有 `_now()` 与 `_wall()` 两个一行内部函数。`_now()` 是"应用时钟"，第三期做内测时**只改它一处**即可让全系统进入虚拟时间；`_wall()` 是"真实设备时钟"，ID 生成走它，**内测冻结/回拨时间也不会撞号**。这是本期唯一预留的接缝，不含任何偏移/倍速机器。

### 3.3 本期修掉的真问题
1. **时区一致**：`today()/hour()/min()` 统一按 UTC+8，消灭"凌晨今天慢一天"。全系统只有一个"今天"。
2. **完整时间戳一致**：`iso()` 仍存 UTC，与历史 `created_at` 数据格式一致，不产生断层。

### 3.4 第一期验收点
- 引入 Clock 后**不替换任何旧调用**，先确认页面照常运行（新模块零副作用）。
- 单测四法：`today()` 与日历一致（尤其晚间 0–8 点窗口）、`iso()` 与现状 `toISOString()` 一致、`stamp()` 连续调用不重复、`hour()` 与本地小时一致。

## 四、第二期（Phase 1）：80 处调用收敛到 Clock

**原则：严格 1:1 机械替换，一处不多做。** 只替换**无参** `new Date()` 与 ID/时间差处的 `Date.now()`；**带参** `new Date(...)` 的**外壳**一律不动。

> **补充（第二轮修订 #4）**：「带参外壳不动」指 `new Date(变量/字符串)` 这层解析不改；但若括号**内部**嵌了裸 `Date.now()` 或 `new Date()`，内层仍要替换。例：`new Date(Date.now() - 7*86400000)` → `new Date(Clock.ms() - 7*86400000)`。这样才能过 4.3 的硬门槛。

### 4.1 精确映射表（before → after）

| 类 | 现状写法 | 替换为 | 约处数 |
|---|---|---|---|
| A 写日期 | `new Date().toISOString().slice(0,10)` | `Clock.today()` | ~22 |
| A 写时间戳 | `new Date().toISOString()` | `Clock.iso()` | ~8 |
| B 读今天 | `... = new Date().toISOString().slice(0,10)` | `Clock.today()` | ~15 |
| B 读时刻 | `new Date().getHours()` / `getMinutes()` | `Clock.hour()` / `Clock.min()` | ~8 |
| B 读 now 对象 | 无参 `new Date()`（如 `var now = new Date()`） | `Clock.date()` | ~10 |
| C 生成 ID | `Date.now().toString(36)`（仅此段，后缀 `Math.random()` 保留） | `Clock.stamp()` | ~12 |
| C 时间差 | `Date.now() - t` / `Date.now() - N*86400000` | `Clock.ms() - ...` | ~3 |
| D 页脚年份 | `new Date().getFullYear()` | `Clock.date().getFullYear()` | 1 |
| D "今天"按钮 | `valueAsDate = new Date()` | `valueAsDate = Clock.date()` | 2 |
| **E 不动** | `new Date('...')`、`new Date(y,m,d)`、`new Date(iso)`（带参解析外壳） | **外壳保持原样**（内层裸调用照替换，见 4.0 补充） | ~50 |

> 唯一需人工判断的是"无参 vs 带参"，边界已由上表明确：带参解析绝不替换。

### 4.2 分批顺序（每批独立可验收、可回滚，做完即停）

1. **批1 · ID 类（C）** —— 风险最低，无逻辑判断。
   验收：新建任务/物品/订单/帖子，ID 正常且唯一。
2. **批2 · 写入类（A）** —— 提交、创建、成交时间戳。
   验收：提交任务后 `submitted_at`、上架 `listed_at`、成交 `sold_at` 正确。
3. **批3 · 读判断类（B）** —— 逾期、"即将截止"、餐段、会议/拍卖临近、daysSince。
   验收：逾期统计、今日筛选、10:30/16:30 餐段卡点、催单通知全部正确。
4. **批4 · 展示类（D）** —— 页脚年份、"今天"按钮、"x 分钟前"。
   验收：页脚年份、日期选择器"今天"、相对时间显示正确。

### 4.3 完成判定（硬门槛）
全库 grep：无参 `new Date()` 与 `Date.now()` 除 `Clock` 内部外**零残留**（带参 `new Date(...)` 允许保留）。这是第二期完成的客观标准，也保证第三期内测能 100% 接管时间。

### 4.4 风险与回滚
- 纯机械替换、语义不变；单批出错只回退该批。
- 边界判断仅"无参 vs 带参"，已由映射表框定。
