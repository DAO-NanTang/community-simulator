## 一、目标

### 1.1 这个功能要解决什么问题

当前 dashboard 的营地设置标签页是一个纯表格编辑器，用户可以逐条添加任务，但存在三个根本问题：

1. **没有引导**：用户面对空白表格，不知道应该填什么、为什么要填、填了会影响什么
2. **没有时间骨架**：任务散乱排列，看不出哪些是筹备期做、哪些是营期做、哪些互相依赖
3. **开会时不好用**：需要大量打字（输入姓名、日期、说明），在会议场景下操作迟缓

### 1.2 目标效果

一个**不依赖记忆、不依赖打字、开会时能流畅使用的营地信息填写工具**。具体表现为：

- 一个人选好开营日期后，系统自动生成所有筹备、营期、结项的里程碑节点
- 每个节点下挂对应的任务卡片，卡片预填了已知信息（做什么、为什么重要、默认子项），空白部分只需点头像、勾选、下拉选择
- 整个过程像"跟着引导做题"，而不是"面对空白表格自己想"
- 填完后一键同步到所有展示页面（任务大厅、攻略地图、结算界面、预算表）

### 2.2 组织理念

用户描述的营地组织方式是：

- **角色分工**：统筹、财务、宣传、课程、后勤、共创人，各有职责
- **两种参与模式**：合伙人（承担全程+兜底+分红）和任务认领者（领零散任务+即时结算）
- **任务可以合并**：同类任务打包认领，减少碎片化
- **任务从流程手册来**：基于前三期经验提炼的标准化流程（共创营_流程手册_v5）

## 三、设计理念

### 3.1 为什么是"日历驱动"

营地是一个有明确起止日期的事件。开营日是一切时间的锚点。从开营日出发：

- **往前倒推**：筹备会（D-30）、招募启动（D-21）、开营前准备（D-14/D-7/D-2）
- **往后正推**：营期各节点（第N天）、展览日、结营日
- **结营后再推**：复盘会（+3天）、结项报告（+7天）

一旦选定开营日，整个时间骨架自动成立。用户只需要检查日期是否合理，不需要手动计算。

### 3.2 为什么是"卷子式"

用户明确说"像一个卷子一样的"。卷子的核心特征：

- **题目已出好**（来自流程手册的 54 条任务）
- **空白处填写**（负责人、协作者、截止日期、产出物）
- **有标准答案参考**（第三期数据作为默认值）
- **做完一题再看下一题**（按节点分组，不一次铺开）

这与"空白表格"的本质区别：表格没有上下文，卷子有。

### 3.3 设计原则（来自 UX 研究）

| 原则 | 来源 | 在本方案中的应用 |
|------|------|-----------------|
| 识别优于回忆 | UX 启发式原则 | 可选项（人员、时间节点、产出物）全部列出，不让用户凭记忆输入 |
| 错误预防 | UX 启发式原则 | 必填项标*，保存时检查；遗漏栏实时提示 |
| 渐进披露 | 交互设计原则 | 先选日期→再看节点→再填卡片，复杂信息分层展示 |
| 菲茨定律 | 交互设计原则 | 头像选择器放在卡片内，点击距离最短 |

### 4.2 卡片的字段设计

| 字段 | 控件类型 | 是否必填 | 默认值来源 | 设计理由 |
|------|---------|---------|-----------|---------|
| 📖 做什么 | 只读文本 | - | 流程手册 | 降低认知负担，不需要自己定义任务 |
| ⚡ 为什么重要 | 只读文本 + 阻塞提示 | - | 流程手册 | 让人理解后果，主动填而非被催 |
| 负责人 | 头像选择器 | * | 无（用户选） | 点击即选，零打字 |
| 时间节点 | 下拉选择 | * | 当前节点日期 | 预设常用节点，支持自定义 |
| 协作者 | 头像选择器（多选） | | 无 | 同一交互模式 |
| 子任务清单 | 勾选框 + 可添加 | | 流程手册 | 大部分情况不改，特殊情况可增减 |
| 产出物 | 勾选框 + 可添加 | * | 流程手册 | 有标准选项，也允许自定义 |
| 激励点 | 数字输入 | | 流程手册 + 锚定基准 | 锚定提示辅助决策 |
| 截止日期 | 日期选择器 | | 当前节点日期 | 标准日历控件 |
| 🔗 阻塞关系 | 自动计算 | - | 依赖分析 | 让人知道这个任务影响多少后续 |

## 五、交互流程

### 5.1 初次使用流程

```
用户打开营地设置
    │
    ▼
① 看到日历设定栏
    │
    ├── 输入开营日：2026/07/15
    ├── 选择营期天数：15
    │
    ▼
② 时间线自动生成（13个里程碑节点）
    │
    ├── 筹备期：D-30, D-21, D-14, D-7, D-2, D-1
    ├── 营期：D1(开营), D2(拿笔), D-7(展览前7天), D-3, D-1, 展览日, 结营日
    └── 结项期：结营+3(复盘), 结营+7(结项)
    │
    ▼
③ 点击第一个节点「D-30 筹备会」
    │
    ├── 下方展开该节点的5张任务卡片
    ├── 每张卡片预填了已知信息
    │
    ▼
④ 逐张填写卡片（开会时可以投屏逐张讨论）
    │
    ├── 点头像 → 选负责人
    ├── 勾选 → 确认子项
    ├── 下拉 → 选产出物类型
    ├── 点 [保存] → 卡片变绿色
    │
    ▼
⑤ 重复③④，直到所有节点处理完毕
    │
    ▼
⑥ 点击 [＝ 同步到展示]
    │
    └── 任务大厅、攻略地图、结算界面、预算表自动更新
```

### 5.2 开会场景流程

```
会议开始
    │
    ▼
投屏营地设置 → 点节点「D-30 筹备会」
    │
    ▼
讨论第1张卡片「召集筹备会」：
    ├── 砚仁：这个我做 → 点自己头像
    ├── 朝林：我协办 → 点朝林头像
    ├── 小红：还需要订会议室 → 主持人在 ☐预订会议室 打勾
    ├── 富章：产出加个参与确认名单 → 点产出 [+添加] → 输入
    └── [保存]
    │
    ▼
讨论第2张卡片「确定预算框架」：
    └── ...（同上节奏）
    │
    ▼
5张卡片填完 → 节点进度 5/5 → 绿色完成标记
    │
    ▼
进入下一个节点「D-21 招募启动」
```

**关键设计点：全程无人打字。** 操作只有三种：点击头像、勾选复选框、下拉选择。打字仅用于极少数自定义场景（添加新子项、添加新产出物）。

### 6.3 同步机制

```
[＝ 同步到展示] 被点击
        │
        ├── 遍历所有已保存的任务卡片
        ├── 按 name 匹配 data.tasks 中已有任务 → 更新
        ├── 不匹配 → 新增
        ├── 更新 data.budget（如果预算页有改动）
        ├── 更新 data.schedule（如果日程框架有改动）
        ├── 更新 data.members（如果团队页有改动）
        └── 调用 updateAll() + localStorage 持久化
```

**反向同步**：营期中在任务大厅新增/修改的任务 → 点 [← 回读] 按钮 → 读取 data.tasks 中带有 `phase: "营中执行"` 的条目 → 追加到对应节点下，标记 `source: "营中新增"`。
