# 方向 A：功能逻辑与效果描述

## A1. 整体逻辑链

```
用户打开营地设置 → 看到 6 个子标签
                  │
                  ├── ① 工作人员任务池
                  │     ├── 上方：模板下拉框 + [使用模板] [保存为模板]
                  │     ├── 中间：批量编辑表格（名称/类别/激励点/类型/来源/状态/删除）
                  │     └── 下方：任务计数 + 激励点合计 + [提交到任务大厅]
                  │
                  ├── ② 共创人任务池
                  │     └── 结构和操作逻辑同①，但模板列表不同

                  ├── ③ ~ ⑥ 团队/预算/日程/AI导入（已有，逻辑不变）
```

## A2. 模板系统的运作方式

### 什么是模板
模板是一组预设好的任务清单。每个模板属于「工作人员」或「共创人」其中一个类型。点击「使用模板」后，模板里的任务批量填入当前表格，用户可以逐条修改（改名称、改激励点、删掉不需要的行、加新的行）。

### 预设模板（10个，首次自动创建）
系统内置 10 个模板，数据来自前三期的实际经验和四份参考文档：

**工作人员模板（5个，按阶段拆分）：**
1. 筹备会完整包——涵盖筹备会前到筹备会后的全部工作
2. 招募与招生——文案发布、面试流程、确认与保险
3. 开营前准备——网络见面会、动员会、基础设施检查
4. 营期执行——每日晨会、进度可视化、展览追问、NT审核
5. 结项与复盘——复盘会议、结项报告、财务结算、资料归档

**共创人模板（5个，按任务类型拆分）：**
6. 主线任务包——12项必做任务，含3个质量闸口
7. 支线任务包——9项选做任务，晨修/写生/农务/入户闲谈
8. 共创任务包——7项协作任务，小组为单位验收
9. 隐藏任务包——8项惊喜触发任务
10. 课程框架包——茶话会→公共课→小组画→自学小组→两人会议

### 用户使用流程
1. 打开营地设置 → 点「① 工作人员任务池」
2. 下拉框选「筹备会完整包」→ 点「使用模板」→ 表格瞬间填满 ~30 条任务
3. 浏览每条任务，删除不合适的，修改激励点
4. 点「提交到任务大厅」→ 任务进入主界面的任务大厅标签页
5. 如果这套任务以后还要用，点「保存为模板」→ 命名 → 存入浏览器本地存储

### 模板的生命周期
- **创建**：用户编辑好表格 → 点「保存为模板」→ 输入名称 → 存入浏览器本地存储
- **使用**：下拉框选择 → 点「使用模板」→ 覆盖当前表格（有未保存内容会弹窗确认）
- **删除**：下拉框右侧有一个删除按钮，删除选中的模板
- **持久化**：模板存在浏览器 localStorage 中，关闭页面不丢失。导出 data.json 时模板一并导出
- **重置**：如果用户删除了预设模板，可以通过「恢复默认模板」按钮一键恢复 10 个内置模板

### 预算重置
预算子页在「保存预算」按钮旁有一个「重置默认值」按钮。点击后所有数值恢复为第三期的实际数据。这个操作不会自动保存——用户需要再点「保存预算」才会生效。

## A3. 两个任务池的独立运作

工作人员任务池和共创人任务池是两个完全独立的表格。它们：
- 有各自独立的模板列表（工作人员模板只出现在工作人员的模板下拉框里）
- 提交到任务大厅时，工作人员任务的默认状态是「待认领」，共创人任务会携带各自的任务类型标记（主线/支线/共创/隐藏）
- 两个池可以分别提交，不需要同步

## A4. 实施后的效果

新一期营地启动时，用户的典型操作流程：
1. 打开营地设置 → ① 工作人员任务池 → 选模板「筹备会完整包」→ 使用 → 根据本期实际情况删改 → 提交
2. 点 ② 共创人任务池 → 选模板「主线任务包」→ 使用 → 调整 NT 值 → 提交
3. 同样操作支线、共创、隐藏任务包
4. 点 ④ 预算 → 查看第三期默认值 → 根据本期调整 → 保存
5. 点 ⑤ 日程框架 → 选择本期起止日期 → 生成日期列表 → 保存
6. 导出 data.json → 任务大厅、攻略地图、结算界面全部自动填充

整个过程从「逐条手打」变为「选模板 → 微调 → 提交」。

# 方向 B：代码逻辑与实施细节

## B1. 解决的问题及对应改动

审查报告指出 5 处方案缺失。以下逐条补充。

### 审查问题 1：两个任务池需要两套独立的 JS 状态变量

**当前**：`settingTasks` 一个数组存储所有任务池数据。

**改为**：`settingStaffTasks` 和 `settingCreatorTasks` 两个独立数组。切换子标签时不清空对方的数据。

### 审查问题 2：模板 `type` 字段与任务条目 `type` 字段命名冲突

**解决方案**：模板的分类字段使用 `pool`。

模板存储结构更新为：
- `pool: "staff"` 表示工作人员模板
- `pool: "creator"` 表示共创人模板
- 任务条目本身的 `type` 字段（初始/新增/变更）不受影响

### 审查问题 3：子标签标识符定义

新增标识符映射：

| 子标签编码 | 显示名称 | 对应 DOM ID | switchSettingsTab 参数 |
|-----------|---------|------------|----------------------|
| ① | 工作人员任务池 | `set-staffTasks` | `'staffTasks'` |
| ② | 共创人任务池 | `set-creatorTasks` | `'creatorTasks'` |
| ③ | 团队 | `set-team` | `'team'`（不变） |
| ④ | 预算 | `set-budget` | `'budget'`（不变） |
| ⑤ | 日程框架 | `set-schedule` | `'schedule'`（不变） |
| ⑥ | AI导入 | `set-ai` | `'ai'`（不变） |

### 审查问题 4：边缘情况处理

| 场景 | 处理方式 |
|------|---------|
| 有未保存内容时点「使用模板」 | 弹窗确认「当前表格内容将被替换，是否继续？」 |
| 保存模板时名称与已有模板冲突 | 弹窗确认「模板已存在，是否覆盖？」 |
| 删除模板 | 模板下拉框旁边有删除按钮（×），仅对用户自建模板生效。预设模板不可删除，显示为灰色 |
| 预设模板被删除后恢复 | 预算页和模板区各有一个「恢复默认模板」入口 |

### 审查问题 5：模板选择器独立

工作人员任务池和共创人任务池各自拥有独立的模板下拉框。下拉框只显示对应 `pool` 类型的模板。过滤逻辑：工作人员池的下拉框只显示 `pool === 'staff'` 的模板，共创人池的下拉框只显示 `pool === 'creator'` 的模板。

## B2. 完整函数清单

### 模板管理层（新增）

| 函数名 | 功能 |
|--------|------|
| `loadTemplates()` | 从 localStorage 读取 `camp_templates`，无则初始化默认 10 个 |
| `getTemplatesByPool(pool)` | 返回指定 pool 的模板名列表 |
| `saveTemplate(name, pool)` | 将当前 pool 对应的 settingTasks 数组保存为模板 |
| `deleteTemplate(name)` | 删除指定模板（仅用户自建） |
| `applyTemplate(name, pool)` | 将模板填充到对应 pool 的 settingTasks，覆盖前确认 |
| `restoreDefaultTemplates()` | 恢复 10 个内置模板到 localStorage |

### 任务池渲染层（修改现有函数）

| 函数名 | 功能 |
|--------|------|
| `renderSettingStaffTasks()` | 渲染工作人员任务池表格 |
| `renderSettingCreatorTasks()` | 渲染共创人任务池表格 |
| `addSettingTask(pool)` | 向指定 pool 添加空行 |
| `removeSettingTask(pool, idx)` | 删除指定 pool 的第 idx 行 |
| `updateSettingStats(pool)` | 更新指定 pool 的统计计数 |
| `commitSettingTasks(pool)` | 提交指定 pool 的任务到 data.tasks |

### 预算层（新增 1 个函数）

| 函数名 | 功能 |
|--------|------|
| `resetBudgetDefaults()` | 将预算输入框恢复为第三期默认值 |

### 子标签切换层（修改现有函数）

`switchSettingsTab(sub)` 的合法参数从 5 个扩为 6 个。增加 `'staffTasks'` 和 `'creatorTasks'`，移除原来的 `'tasks'`。

## B4. 函数调用关系图

```
页面加载
  │
  └── loadTemplates()  ← 初始化 camp_templates（首次使用 = 写入 10 个预设）
  
用户点「使用模板」
  │
  └── applyTemplate(name, pool)
        ├── 检查当前表格是否有未保存内容 → 弹窗确认
        ├── 从 camp_templates 读取模板数据
        ├── 写入 settingStaffTasks 或 settingCreatorTasks
        └── 调用 renderSettingXxxTasks() 刷新表格

用户点「保存为模板」
  │
  └── saveTemplate(name, pool)
        ├── 检查名称是否已存在 → 弹窗确认覆盖
        ├── 读取当前 settingXxxTasks 数组
        ├── 写入 camp_templates
        └── 刷新模板下拉框

用户点「提交到任务大厅」
  │
  └── commitSettingTasks(pool)
        ├── 遍历 settingXxxTasks
        ├── 匹配 data.tasks 中同名任务 → 更新 / 不存在 → 新增
        └── 调用 updateAll() 刷新全局
```

## B6. 模板名称冲突的处理

用户自建模板与预设模板可以同名——保存时弹窗确认覆盖。覆盖后原预设模板的数据被替换，但「恢复默认模板」可以恢复原始版本。预设模板本身在模板列表中显示为锁定图标（🔒），用户不能直接删除，但可以通过覆盖来修改内容。
