## 核心概念修正

上一版把 NT 上限理解为「用户随便设的一个数字，预算不能超过它」——这是错的。

**正确的逻辑：**

```
预算表计算 NT 总需求
    │
    └── NT 总需求 = 工作人员NT工资 + 赏金池NT + 共创人任务NT池
                    │
                    ▼
              这个总需求 = 向 DAO 申请的 NT 总额
                          = 向社区募捐的 NT 目标
                    │
                    ▼
        共创人任务NT池 = NT总需求 - 工作人员NT - 赏金池NT
                    │
                    ▼
              ┌─────────────────────────┐
              │ 这个池子是 NT 角色的约束  │
              │                         │
              │ 在步骤④设计任务激励时：   │
              │ · 主线任务 NT 合计       │
              │ · 支线任务 NT 合计       │
              │ · 共创任务 NT 合计       │
              │ · 隐藏任务 NT 合计       │
              │ +────────────────────    │
              │ 总计 ≤ 共创人任务NT池    │
              │                         │
              │ 超过 → 🔴 标红 → 调整    │
              └─────────────────────────┘
```

上限约束的是**发任务的 NT 角色**，不是预算表本身。

## 修改 1：重写 NT 发行逻辑

### 1.1 原方案 3.1 数据结构修改

**删除：**
```javascript
nt_issue_cap: 30000,  // ← 删掉，不需要用户设上限
```

**新增：**
```javascript
// === 步骤①-④ 逐步填入，步骤⑤锁定 ===

// NT 总需求 = 下面三项之和（自动计算）
nt_total_required: 0,

// 三项构成：
staff_nt_total: 0,         // 工作人员 NT 工资合计（步骤③④填入）
prize_pool_nt: 4500,       // 赏金池 NT（步骤①填入）
creator_task_nt_pool: 0,   // 共创人任务 NT 池 = nt_total_required - staff_nt_total - prize_pool_nt

// 共创人任务 NT 池的消耗（步骤④ 实时计算，不存储）
// creator_task_nt_used = 主线NT合计 + 支线NT合计 + 共创NT合计 + 隐藏NT合计
// creator_task_nt_remaining = creator_task_nt_pool - creator_task_nt_used
// 如果 remaining < 0 → 🔴 超支警告
```

### 1.2 NT 总需求的计算时机

```
步骤①：基于粗估参数计算预估 NT 总需求（用于向 DAO 提交初步申请）
步骤②：任务 NT 精确化后更新
步骤③④：工作人员 NT 工资填入后更新，同时算出共创人任务 NT 池
步骤⑤：最终锁定
```

### 1.3 预算表底部显示

每个阶段的预算表末尾（支出合计之后）加：

```
步骤①-③：
| NT 总需求（预估）     | — | NT 30,000 | = 工作人员NT + 赏金池NT + 共创人任务NT池 |
| 　├ 工作人员 NT       | — |     —     | ⏳ 待步骤③④填入 |
| 　├ 赏金池 NT         | — | NT 4,500  | 步骤①设定 |
| 　└ 共创人任务 NT 池   | — | NT 25,500 | = 总需求 - 工作人员 - 赏金池 |
| 📌 需向 DAO 申请      | — | NT 30,000 | 等于 NT 总需求 |

步骤④（NT 角色填入后，未超支）：
| 共创人任务 NT 池       | — | NT 25,500 | = 总需求 - 工作人员 - 赏金池 |
| 　主线任务 NT (12项)   | — | NT 8,800  | |
| 　支线任务 NT (9项)    | — | NT 2,850  | |
| 　共创任务 NT (7项)    | — | NT 2,800  | |
| 　隐藏任务 NT (8项)    | — | NT 1,120  | |
| 　已分配 NT 合计       | — | NT 15,570 | |
| ✅ NT 剩余可分配       | — | NT 9,930  | 绿色 |

步骤④（NT 角色填入后，超支）：
| 共创人任务 NT 池       | — | NT 15,000 | = 总需求 - 工作人员 - 赏金池 |
| 　已分配 NT 合计       | — | NT 17,200 | |
| ⚠ NT 超出池上限       | — | NT 2,200  | 🔴 请削减任务激励点或增加 NT 总需求 |
```

### 2.3 表格模板参考

表格结构与 `学员任务系统清单.md` 一致：

| 列 | 说明 | 来源 |
|------|------|------|
| 编号 | M1-M12 / S1-S9 / C1-C7 / H1-H8 | 任务系统清单 |
| 任务名 | 任务简称 | 同上 |
| NT/人 | 每人获得的 NT 值 | NT 角色填写 |
| 验收标准 | 完成任务的条件 | 同上（可修改） |
| 对应日程 | 任务在营期中的位置 | 同上 |

**关键交互**：
- NT/人 列是数字输入框，修改后自动重算：`小计 = NT/人 × 共创人数`，`已分配合计 = 所有小计之和`
- `剩余可分配` 实时更新，变负数时整行标红
- 每行可 ✎ 编辑、🗑 删除
- 四个子表各有一个 `[+添加]` 按钮
- `[从模板导入]`：下拉选择预设模板（如「第三期任务系统清单」），填充表格
- `[导出为模板]`：当前表格存为模板，下期复用

### 2.4 与任务池的同步

这个表格不是独立数据——它**就是**共创人任务包的激励点来源：

```
NT 任务设计表中：
  M1 入营注册  30 NT/人

↓ 自动同步到 ↓

data.tasks 中：
  { name: "M1 入营注册", points: 30, category: "课程", ... }

↓ 自动汇总到 ↓

data.budget 中：
  creator_task_nt_used += 30 × 10人 = 300
```

修改表格 → 任务卡片 `points` 同步更新 → 预算表 `creator_task_nt_used` 同步更新。

## 修改 8：任务系统清单标准化为结构化模板

### 8.1 问题

当前 `学员任务系统清单.md` 是一份 Markdown 参考文档。NT 角色在步骤④设计任务时，只能手动对着文档逐条录入——慢、容易错、无法复用。

### 8.2 方案

将任务清单转化为结构化 JSON 模板，存入 `camp_templates`（与已有的 10 个预设模板同库），通过步骤④ NT 设计表的「从模板导入」按钮一键加载。

### 8.5 模板的创建与复用

| 操作 | 说明 |
|------|------|
| `[从模板导入]` | 从 `camp_templates`（pool=nt_design）中选择，覆盖当前表格 |
| `[导出为模板]` | 当前 NT 设计表 → 命名 → 存入 `camp_templates`（pool=nt_design） |
| 预设模板 | 首次使用时，系统自动将「第三期·任务系统清单」写入 `camp_templates` |
| 下期复用 | 新营期打开 → 导入上期模板 → 修改 NT 值 → 保存为新模板 |

### 8.6 与已有模板系统的关系

`camp_templates` 目前有三种 pool 类型：

| pool | 用途 | 来源 |
|------|------|------|
| `staff` | 工作人员任务模板 | 模板方案 v2（5个预设） |
| `creator` | 共创人任务模板 | 模板方案 v2（5个预设） |
| `nt_design` | **NT 任务设计模板（新增）** | 本次修改 |

三者共享同一套模板增删改查函数（`loadTemplates` / `saveTemplate` / `deleteTemplate` / `applyTemplate`），只是 `pool` 值不同。NT 设计表的 `[从模板导入]` 只显示 `pool === 'nt_design'` 的模板。
