# 营地设置 · 流程方案 v3

> 基于审查报告的补充修订版。与 v2 的核心差异：
> 1. 新增数据模型章节（bundles / camp_progress / locked / active）
> 2. ③④合并为「打包并命名角色」，总步骤 6→5
> 3. 系统合并建议标注为第二阶段
> 4. ②任务池增加三态标记（🟢要做 🟡备选 🔴不做）

---

## 一、5 步流程

```
① 统筹定框架 → ② 任务池梳理 → ③ 打包并命名角色 → ④ 角色填卡片 → ⑤ 同步展示
```

## 二、输入 → 展示：数据分配规则

### 2.1 三个展示标签

| 标签 | 名称 | 受众 | 显示内容 |
|------|------|------|---------|
| 📋 | 任务大厅 | 策划组 | 策划组头像(左) + 当日任务卡片(中) + 日期选择器 + 里程碑 |
| 🗺️ | 攻略地图 | 共创人 | 玩家头像(左) + 可接/进行中任务(中) + 日期选择器 + 里程碑 |
| 📊 | 任务总览 | 管理视角 | 全部任务表格 + 阶段筛选 + 完整性检查 |

### 2.2 角色→标签分配规则

| 角色 | → 展示标签 | 原因 |
|------|-----------|------|
| 统筹/宣传/后勤/财务/课程/社区活动/议事规则 | 📋 任务大厅 + 📊 任务总览 | 策划组工作 |
| NT | 🗺️ 攻略地图 + 📊 任务总览 | 共创人任务 |
| （未命名/留空） | 📊 任务总览 | 只出现在管理表格，不出现在任务大厅和攻略地图 |

### 2.3 其他数据→标签

| 数据类型 | 来源步骤 | 分配到 |
|---------|---------|--------|
| 日期/时间线 | ① | 📅 日历系统 + 攻略地图里程碑 + 任务大厅里程碑 |
| 预算 RMB/NT | ① | 💰 公会资金池 |
| 团队名单 | ① | 👥 游戏策划组 |
| 共创人名单 | ④ | ⚔️ 游戏玩家 |
| 财务流水 | 营中新增 | 💰 公会资金池 |
| 结算数据 | 营后 | 🧾 结算界面 |

## 三、数据模型

所有结构存储在 `data` 对象的顶层字段，与 `data.tasks` 同级，随 `updateAll()` 写入 localStorage。

### 3.1 `data.bundles` — 任务包

③ 打包产出的包，④ 填卡片按包展示，⑤ 同步按包的角色分流。

```
data.bundles = [
  {
    name: "宣传工作包",
    role: "宣传",
    task_names: ["设计招募文案", "每日推文", "拍摄记录", "视频剪辑", "结项宣传"]
  },
  {
    name: "共创人NT任务包",
    role: "NT",
    task_names: ["M1 入营注册", "M2 南塘认知", "M3 议事入门", ...]
  }
];
```

独立任务（未打包的）同样存在 bundles 里，`task_names` 只有一项：

```
{ name: "召集筹备会", role: "统筹", task_names: ["召集筹备会"] }
```

**为什么不在 `data.tasks` 里嵌套？** 任务被合并后，角色和包归属从包继承。拆包时角色跟着包走，不残留。

### 3.2 `data.camp_progress` — 步骤进度

导航栏显示状态，刷新不丢失。

```
data.camp_progress = {
  "step1_framework": "done",     // ✅ 已完成
  "step2_taskpool":  "done",     // ✅ 已完成
  "step3_bundle":    "active",   // ⚠ 当前步骤
  "step4_fill":      "locked",   // 🔒 未解锁
  "step5_sync":      "locked"    // 🔒 未解锁
};
```

| 状态 | 显示 | 含义 |
|------|------|------|
| `done` | ✅ | 已完成 |
| `active` | ⚠ | 进行中 |
| `pending` | ⬜ | 可进入但未开始 |
| `locked` | 🔒 | 前置未完成，点击提示"请先完成步骤X" |

### 3.3 `data.camp_dates.milestones[].locked` — 日历锁定

```
milestones: [
  { id: "D-30", label: "第一次筹备会", date: "2026-06-15", rule: "开营-30", locked: false },
  { id: "EX",   label: "展览日",       date: "2026-07-28", rule: "结营-1",  locked: true  }
]
```

- `locked: true` → 开营日变化时不重算
- `rule` 保留计算规则，解锁后可重新计算
- 用户在节点详情中勾选「锁定此日期」

### 3.4 `data.tasks[].active` — 三态标记（②任务池）

```
{ name: "社区破冰活动", active: "maybe", active_reason: "视招募人数决定" }
```

| 标记 | active 值 | 行背景色 | 进入③时 |
|------|----------|---------|---------|
| 🟢 要做 | `yes` | 正常 | 默认显示，可勾选合并 |
| 🟡 备选 | `maybe` | 灰黄 | 默认显示（提醒讨论），可勾选 |
| 🔴 不做 | `no` | 灰红 | 默认隐藏，可切换显示 |

**原则**：不删除、可恢复、可追溯。`active_reason` 可选填。

## 六、关键设计决策

| 决策 | 理由 |
|------|------|
| 5步默认引导但不强制 | 默认按顺序，完成自动解锁下一步。但可直接点击任意步骤跳转。老用户跳过，新用户跟引导 |
| 每步可暂存退出 | 开会时间有限，不需要一次完成 |
| ②可从模板导入 | 流程手册提供预设任务池 |
| ③合并可拆分 | 合并错了拆开，不丢数据 |
| ③④合并为一步 | 打包和命名角色是连续操作，合并减少打断 |
| 系统合并建议 → 第二阶段 | MVP 只做手动合并+类别筛选辅助。同类别+同节点的精确匹配约40行JS，无歧义 |
| 三态标记不删除 | 🔴不做的任务保留数据，可恢复可追溯 |
| ④按角色筛选 | 每人只看自己的任务 |
| NT任务链接字段 | 链接指向NT平台具体任务页 |
| 食宿NT覆盖检查 | 统计共创人任务总NT ≥ 食宿所需NT |

## 七、两条时间线的关系

| | 营地设置中的时间线 | 日历系统中的时间线 |
|------|-----------------|------------------|
| 定位 | 初始默认值 + 存储地 | 展示 + 日常修改 |
| 数据来源 | `data.camp_dates`（开营日自动计算） | `data.schedule`（设置同步过来） |
| 编辑 | 设置初始框架（节点级别的默认日期） | 日常增删改（每天每时段的具体活动） |
| 版本历史 | 无 | 有，每次修改留痕，保留最近50条 |
| 同步 | → 初始同步到日历 | ← 修改后可回写设置 |

**冲突处理**：日历系统上方提示「📅 展览日与设置不一致。[以日历为准] [以设置为准]」。日历修改历史存储在 `data.schedule_history`。

## 附录 B：开会场景操作流程

**场景：第一次筹备会，6人参与，投屏操作。**

1. 打开营地设置 → ①统筹定框架 → 已填好开营日和预算 → ✅
2. 点 ②任务池梳理 → 从模板导入「工作人员·筹备会完整包」→ 47条任务出现
3. 大家逐条过：
   - 「社区破冰活动」有人说不确定 → 标 🟡备选，原因「视招募人数」
   - 「NFT铸造」有人说本期不做 → 标 🔴不做，原因「技术不成熟」
   - 其他全标 🟢要做
4. 点 ③打包并命名角色：
   - 勾选宣传类5条 → 包名「宣传工作包」→ 角色选「宣传」→ 合并
   - 勾选后勤类5条 → 包名「后勤工作包」→ 角色选「后勤」→ 合并
   - 逐条给未打包任务选角色
5. 切换到④角色填写 → 选NT角色 → 检查食宿NT是否够 → 填链接
6. 点⑤同步 → 检查遗漏 → [＝ 同步到全部展示页面]
7. 导出 data.json 备份

**全程耗时约 40-60 分钟（首次，含讨论）。第二次筹备会只需修改几张卡片，约 10 分钟。**

## 附录 D：卡片字段设计表

每条任务卡片包含以下字段，控制类型决定填写方式：

| 字段 | 控件类型 | 是否必填 | 默认值来源 | 设计理由 |
|------|---------|---------|-----------|---------|
| 📖 做什么 | 只读文本 | - | 流程手册 | 降低认知负担 |
| ⚡ 为什么重要 | 只读文本 + 阻塞提示 | - | 流程手册 | 理解后果，主动填写 |
| 负责人 | 头像选择器（点击即选） | * | 无 | 零打字 |
| 时间节点 | 下拉选择（预设+自定义） | * | 节点日期 | 预设常用节点 |
| 协作者 | 头像选择器（多选） | | 无 | 同一交互模式 |
| 子任务清单 | 勾选框 + 可添加 | | 流程手册 | 大部分不改，可增减 |
| 产出物 | 勾选框 + 可添加 | * | 流程手册 | 标准选项+自定义 |
| 激励点 | 数字输入 + 锚定提示 | | 流程手册 | "基准：打扫一天=3点" |
| 截止日期 | 日期选择器 | | 节点日期 | 标准日历控件 |
| 🔗 链接 | 文本输入 | 仅NT任务 | 无 | 指向NT平台任务页 |
| 阻塞关系 | 自动计算 | - | 依赖分析 | 知道影响多少后续 |

## 附录 E：设计原则

| 原则 | 来源 | 在本方案中的应用 |
|------|------|-----------------|
| 识别优于回忆 | UX 启发式 | 可选项全部列出（人员、时间节点、产出物），不让用户凭记忆输入 |
| 错误预防 | UX 启发式 | 必填项标*，保存检查；遗漏栏实时提示；三态标记防止误删 |
| 渐进披露 | 交互设计 | 5步分层：日期→任务池→打包命名→填卡片→同步 |
| 菲茨定律 | 交互设计 | 头像选择器放在卡片内，点击距离最短 |
| 少即是多 | 设计哲学 | 仅加1个日历栏+1个时间线+1个卡片区，复用现有组件 |
| 复用优先 | 工程原则 | 卡片字段兼容 tasks 格式；头像选择器复用 member 数据 |
| 长期主义 | 工程原则 | localStorage + JSON 导出，无后端依赖；字段以扩展方式加入 |
